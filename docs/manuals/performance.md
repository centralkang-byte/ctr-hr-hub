# 성과평가 매뉴얼 (Performance)

> **최종 갱신**: 2026-05-12
> **담당**: 인사관리팀
> **상태**: Draft v1 (CEO 리뷰 전)

---

## 1. 개요

### 1.1 무엇을 다루는가

CTR HR Hub의 성과평가 모듈은 다음을 통합 관리합니다.

- **평가 사이클 (Performance Cycle)** — 반기·연간 평가 회차 등록 + 9단계 진행 상태 관리
- **목표 관리 (MBO Goal)** — 직원별 목표 설정·승인·진척 관리·잠금
- **분기 체크인 (Quarterly Review)** — 분기별 직원·팀장 듀얼 제출 형식 회고
- **1:1 미팅 (One-on-One)** — 정기·임시·목표 리뷰·개발 1:1, AI 가이드·요약
- **자기평가·상사평가·동료평가 (Self/Manager/Peer Eval)** — 3종 평가 통합 입력·집계
- **EMS 9-Block 매트릭스** — 성과 × 역량 2축 9개 블록 분류
- **4등급 체계** — 탁월(O) / 우수(E) / 보통(M) / 미흡(S) — CEO 확정안 (2026-04-06)
- **캘리브레이션 (Calibration)** — 부서·전사 등급 분포 조정 + 가이드라인 적용
- **MBO/BEI 가중치** — 60/40 (회사·사이클별 조정 가능)
- **펄스 설문 (Pulse Survey)** — 정기 직원 만족도·몰입도 측정
- **승계 계획 (Succession Plan)** — 핵심 직위 후계자 풀 관리
- **팀 건강도 점수 (Team Health Score)** — 팀별 건강도 추이 추적

### 1.2 도입 의도

- **성과 가시화** — 직원·팀장·인사담당자가 성과·역량을 한 화면에서 시점별로 확인
- **공정성 확보** — 캘리브레이션을 통해 부서·관리자 간 평가 편차 보정 + 9-Block 매트릭스로 객관적 분포 관리
- **개발 중심 문화** — 단순 등급 통보가 아닌 1:1·체크인·분기 회고로 지속적 개발 대화 촉진
- **데이터 기반 의사결정** — 평가 결과 → 보상(승급·인상률) 자동 연계, 승계 후보 추천
- **AI 활용** — 자기평가 초안·1:1 가이드·평가 편향 검출에 AI 보조
- **사일런트 데이터 마스킹** — 평가 결과를 정해진 시점까지 직원에게 비공개 (캘리브레이션 완료 전 결과 노출 방지)

### 1.3 보상·인사관리·결재 매뉴얼과의 관계

| 항목 | 본 매뉴얼 (성과평가) | 관련 매뉴얼 |
|------|---------------------|-----------|
| 평가 등급 결정 | 본 매뉴얼이 SSOT | 보상 매뉴얼·급여 매뉴얼이 참조 |
| 인상률 매트릭스 | 본 매뉴얼이 정의 | 보상 매뉴얼이 적용 |
| 9-Block 결과 | 본 매뉴얼이 산출 | 인사관리(승진 결정·승계 계획) + 보상(인상률) |
| 목표 결재 | 본 매뉴얼이 신청 | 결재 매뉴얼 §5.2 직속 팀장 검토 |
| 평가 사이클 + 보상 사이클 | 본 매뉴얼이 시작 → 보상으로 이관 | [compensation.md](compensation.md) |

## 2. 성과평가 구성 요소

### 2.1 평가 사이클 (PerformanceCycle)

회사별 반기·연간 평가 회차.

| 속성 | 설명 |
|------|------|
| 이름·연도·반기 | 예: "2026년 H1 평가" (H1/H2/ANNUAL) |
| 목표 기간 | 목표 설정 시작·종료일 |
| 평가 기간 | 평가 실시 시작·종료일 |
| 체크인 마감일 | 중간 체크인 마감 |
| 체크인 모드 | 의무 (게이트) / 권장 (경고만) |
| 동료 평가 활성화 | 동료 평가 사용 여부 |
| 동료 평가 인원 | 최소·최대 동료 평가자 수 (기본 3~5명) |
| MBO/BEI 가중치 | 목표 vs 역량 가중치 (기본 60/40) |
| 수습 제외 여부 | 수습 직원 평가 대상 제외 여부 |
| 대상 필터 | 회사·부서·직급 필터 JSON |
| 상태 | 9단계 상태 (§2.2 참조) |

### 2.2 평가 사이클 9단계 상태

| # | 상태 | 의미 |
|---|------|------|
| 1 | 개시 (DRAFT) | 사이클 생성, 설정 중 |
| 2 | 목표 설정 (ACTIVE) | 직원이 목표 입력·승인 받는 단계 |
| 3 | 중간 체크인 (CHECK_IN) | 분기·중간 점검 |
| 4 | 평가 실시 (EVAL_OPEN) | 자기평가·상사평가·동료평가 입력 |
| 5 | 캘리브레이션 (CALIBRATION) | 부서·전사 등급 분포 조정 |
| 6 | 최종 확정 (FINALIZED) | 캘리브레이션 승인 완료, 결과 통보 대기 |
| 7 | 보상 기획 (COMP_REVIEW) | 평가 결과 기반 인상률·승급 결정 |
| 8 | 보상 완료 (COMP_COMPLETED) | 보상 승인 → 급여에 반영 완료 |
| 9 | 종료 (CLOSED) | 사이클 종료 |

상태는 순차 전이만 허용 (역행 불가, 사이클 취소만 가능).

### 2.3 4등급 체계 (CEO 확정 2026-04-06)

| 등급 | 코드 | 한글 | 가이드라인 | 설명 |
|------|------|------|----------|------|
| Outstanding | O | 탁월 | ~10% | 최상위 성과·역량 |
| Exceeds | E | 우수 | ~30% | 기대 초과 |
| Meets | M | 보통 | ~50% | 기대 충족 |
| Sufficient | S | 미흡 | ~10% | 기대 미달, 개선 필요 |

성과 등급(performance) + 역량 등급(competency) 각각 4등급 부여 → 2축 EMS 9-Block 매트릭스에 위치 결정.

### 2.4 EMS 9-Block 매트릭스 (성과 × 역량)

```
역량   ↑
       │  HIGH    [9: Top Talent]      [6: Future Leader]     [3: Solid Performer]
       │  MID     [8: Star Performer]  [5: Core Contributor]  [2: Inconsistent]
       │  LOW     [7: Risk]            [4: Underperformer]    [1: Detractor]
       └────────────────────────────────────────────────────────→ 성과
                  HIGH                   MID                    LOW
```

- 9-Block 라벨·임계값은 회사별 설정 (`EmsBlockConfig`)
- 보상·승계·승진 의사결정의 기준

### 2.5 목표 (MBO Goal)

직원이 평가 사이클에 등록하는 목표.

| 속성 | 설명 |
|------|------|
| 제목·설명 | 목표 명칭 + 상세 |
| 가중치 | 전체 목표 중 비중 (총합 100%) |
| 측정 지표 / 목표값 | 정량 지표 (있으면) |
| 상태 | 작성 중 / 결재 대기 / 승인 / 반려 |
| 달성 점수 | 사이클 종료 시 입력 |
| AI 생성 여부 | AI 추천 목표인지 |
| 잠금 여부 | 사이클 진행 후 변경 제한 |
| 마감 지연 | 마감 후 미설정 시 자동 마킹 |
| 개정 (Revision) | 사이클 중 목표 변경 시 결재 흐름 |

### 2.6 평가 종류 (EvalType)

| 종류 | 의미 | 입력자 |
|------|------|--------|
| 자기평가 (SELF) | 본인이 본인 평가 | 직원 |
| 상사평가 (MANAGER) | 직속 팀장 평가 | 팀장 |
| 동료평가 (PEER) | 동료가 평가 | 지명된 동료 (3~5명) |

상태: 작성 중 / 제출 완료 / 확정.

### 2.7 1:1 미팅 종류 (OneOnOneType)

| 종류 | 의미 |
|------|------|
| 정기 (REGULAR) | 정기 1:1 (예: 격주) |
| 임시 (AD_HOC) | 필요 시 |
| 목표 리뷰 (GOAL_REVIEW) | 목표 진척 점검 |
| 개발 (DEVELOPMENT) | 커리어·역량 개발 대화 |

- AI 가이드: 1:1 시작 시 AI가 가이드 질문 추천
- AI 요약: 1:1 종료 후 노트 자동 요약
- 감정 태그: 긍정 / 중립 / 우려 자동 분류

### 2.8 분기 체크인 (QuarterlyReview)

분기별로 직원·팀장이 각각 작성하는 듀얼 제출 형식 회고.

**직원 작성 영역**:
- 목표 성과 하이라이트
- 도전 과제
- 개발 필요 사항
- 코멘트

**팀장 작성 영역** (직원 제출 후):
- 인식한 강점
- 개선 영역
- 다음 분기 우선순위

**상태 5단계**: 작성 중 → 진행 중 → 직원 완료 → 팀장 완료 → 완료

## 3. 핵심 정책

### 3.1 평가 사이클 라이프사이클 (9단계)

```
[1: DRAFT] 사이클 생성·설정
   │
   ↓
[2: ACTIVE] 목표 설정 (직원 입력 + 팀장 승인)
   │
   ↓
[3: CHECK_IN] 중간 체크인 (1:1·분기 회고)
   │
   ↓
[4: EVAL_OPEN] 평가 실시 (자기·상사·동료)
   │
   ↓
[5: CALIBRATION] 캘리브레이션 (등급 분포 조정)
   │
   ↓
[6: FINALIZED] 결과 확정 → 직원 통보 대기
   │
   ↓
[7: COMP_REVIEW] 보상 기획 (인상률·승급 결정)
   │
   ↓
[8: COMP_COMPLETED] 보상 승인 → 급여 반영
   │
   ↓
[9: CLOSED] 사이클 종료
```

### 3.2 목표 설정 (MBO)

- 직원이 평가 사이클의 목표 설정 단계에서 본인 목표 등록 (보통 3~5개)
- 각 목표 가중치 합계 = 100%
- 정량 지표가 있으면 측정 지표·목표값 입력
- 팀장 결재 → 승인되면 잠금 (이후 변경 시 개정 결재 흐름)
- AI 추천: 직책·역할 기반 목표 후보 자동 추천

**목표 변경 (Revision)**:
- 사이클 진행 중 환경 변화로 목표 수정 필요한 경우
- 직원이 변경 사유와 함께 신청 → 팀장 결재 → 승인 시 새 버전 활성

### 3.3 중간 체크인 (체크인 모드)

평가 사이클 중간에 분기별 또는 정해진 시점에 진척 확인.

- **의무 (MANDATORY)**: 직원이 체크인 미완료 시 사이클 진행 차단 (Overdue 처리)
- **권장 (RECOMMENDED)**: 알림만, 사이클 정상 진행

체크인은 1:1 미팅 또는 분기 체크인 형식으로 진행. 회사별 선택.

### 3.4 평가 실시 — 자기·상사·동료 3종

**자기평가**:
- 직원이 본인 목표 달성도 + 역량 자가 진단 입력
- AI가 초안 자동 생성 (선택 — 본인이 수정 후 제출)
- 제출 후 수정 불가 (재제출 시 인사담당자 권한)

**상사평가**:
- 직속 팀장이 직원 평가 (목표 달성도 + 역량)
- 점수 + 4등급 (성과·역량) 부여
- AI 편향 검출: 평가 텍스트에 성별·연령 등 편향 표현 자동 감지 → 경고

**동료평가**:
- 사이클 시작 시 직원이 동료 평가자 3~5명 지명 → 팀장 승인
- 익명 점수·코멘트 제출
- 동료별 평가는 익명 (피평가자에게 누가 평가했는지 비공개)

### 3.5 가중치 (MBO/BEI)

평가 점수는 두 가지 영역으로 구성:

- **MBO (Management by Objectives)**: 목표 달성도 (정량·정성)
- **BEI (Behavioral Event Interview)**: 역량 평가 (회사 정의 역량 라이브러리 기반)

**기본 가중치**: MBO 60% / BEI 40% (회사·사이클별 조정 가능).

### 3.6 등급 결정 → 9-Block 매트릭스

각 직원은 다음을 받음:
- **성과 점수** + **성과 등급** (O/E/M/S)
- **역량 점수** + **역량 등급** (O/E/M/S)
- → 2축 9-Block 매트릭스에 위치 결정 (블록 1~9)

블록별 라벨은 회사 설정 (예: Top Talent, Future Leader, Risk 등).

### 3.7 캘리브레이션 (Calibration)

부서·전사 단위로 등급 분포를 조정하는 단계.

**원칙**:
- 등급별 가이드라인 (O: 10% / E: 30% / M: 50% / S: 10%) 권장
- 회사 정책으로 강제 분포(force-rank) 또는 권장 분포 선택
- 부서별 모임에서 매니저들이 검토 → 분포가 가이드라인과 큰 차이 나면 조정
- 조정 시 **사유 필수 입력** (감사 대상)

**투트랙 등급 보존**:
- 원래 등급 (originalGradeEnum): 매니저가 처음 부여한 등급
- 최종 등급 (finalGradeEnum): 캘리브레이션 후 등급
- 두 값을 모두 보존하여 사후 분석 가능

**캘리브레이션 세션**:
- 부서별 또는 전사 단위 세션 등록
- 세션 내 조정 이력 모두 기록 (CalibrationAdjustment)

### 3.8 결과 통보 (FINALIZED → 직원 공개)

캘리브레이션 완료 + 인사담당자 승인 후에만 직원에게 결과 공개:

- 그 이전: **데이터 마스킹** (직원이 본인 평가 결과 조회 불가)
- 그 이후: 직원이 결과 조회 + 매니저 코멘트 열람
- 분기 체크인 결과도 동일 마스킹 (직원·팀장 모두 제출 완료 + 사이클 단계 도달 후 공개)

### 3.9 보상 연계

평가 완료 후 보상 기획 단계 (COMP_REVIEW)로 자동 진입:

- **인상률 매트릭스 (SalaryAdjustmentMatrix)**: 등급 × 급여 밴드 위치 (LOW/MID/HIGH) 조합으로 인상률 추천 (예: O × LOW = +12%, M × HIGH = +3%). 인상률 매트릭스는 [compensation.md §2.3](compensation.md) 참조
- 인사담당자가 추천 인상률 검토·조정 → 결재 → 다음 급여 회차에 자동 반영
- 보상 정책 상세는 [compensation.md](compensation.md) 참조

### 3.10 자동 알림 + 리마인더

- **목표 설정 마감 임박**: 직원·팀장 알림
- **체크인 마감 임박**: 의무 체크인 미완료 시 경고
- **자기평가 마감 임박**: 직원 알림
- **상사평가 마감 임감**: 팀장 알림
- **동료평가 지명 요청**: 직원이 동료 지명 알림
- **동료평가 제출 요청**: 지명된 동료 알림
- **결과 통보**: 결과 공개 시점에 직원 알림
- **1:1 일정 임박**: 양쪽 알림 (1일 전)

### 3.11 펄스 설문 (Pulse Survey)

직원 만족도·몰입도 정기 측정.

- **빈도**: 회사 설정 (월간·분기 등)
- **질문**: 표준 라이브러리 + 회사 사용자 정의 추가
- **익명**: 응답자 신원 비공개, 부서·직급 등 메타데이터만 집계
- **결과**: 팀 건강도 점수에 반영

### 3.12 승계 계획 (Succession Plan)

핵심 직위의 후계자 풀 관리.

- 직위별 후계자 후보 1~3명 지정
- 준비도(Ready Now / Ready in 1-2yr / Ready in 3+yr) 표시
- 평가 결과(9-Block) + 잠재력 평가 기반

- **전사관리자**: 13개 법인 전체

탭 구조:

5. **분기 체크인** — 분기별 체크인 작성·제출
6. **내 결과** — 공개된 사이클 결과 조회 (등급·9-Block·매니저 코멘트)

### 6.2 팀장 — 팀 성과 화면

### 6.3 인사담당자 — 평가 관리 화면

## 7. 특수 상황 처리

### 7.1 사이클 진행 중 직원 입사·퇴사

- **신규 입사**: 사이클 진행 중 입사한 직원은 회사 정책에 따라 수습 제외 옵션 적용 (excludeProbation)
- **퇴사 예정자**: 퇴사 일자가 사이클 종료 전인 경우 인사담당자가 평가 대상 제외 또는 부분 평가 결정

### 7.2 사이클 진행 중 휴직

- **휴직 활성화**: 사이클 중 휴직 들어가면 평가 진행 일시 중지
- **복직 후**: 해당 사이클 완료 처리 또는 다음 사이클부터 참여 (회사 정책)
- 현재는 인사담당자 수동 처리 ([loa.md §7.3](loa.md) 참조)

### 7.3 사이클 진행 중 법인전환

- 법인 이동 시점에 평가 진행 중인 경우
- 출발 법인에서 평가 완료 후 이동이 표준 운영
- 불가피한 경우 인사담당자가 평가 대상에서 제외 또는 부분 평가

### 7.4 매니저 변경 (전보·퇴사)

- 평가 사이클 중 직원의 직속 팀장이 변경된 경우
- 신규 팀장이 상사평가 입력 (이전 팀장의 입력 데이터는 참고용 보존)
- 동료 평가는 직원 본인이 재지명 가능

### 7.5 캘리브레이션 결과 강제 변경

- 인사담당자 승인 후 결과 통보 직전에 등급 변경 필요한 경우
- 전사관리자가 결재 없이 강제 변경 가능 (감사 대상)
- 변경 사유 필수 입력 + 변경 이력 영구 보존

### 7.6 결과 통보 후 정정

- 직원 통보 후 매니저·인사담당자가 등급 변경 필요한 경우
- 인사담당자 수동 정정 + 사유 + 직원 재통지
- 운영상 권고: 통보 전 검증 철저히 (정정 시 직원 신뢰도에 영향)

### 7.7 동료 평가자 지명 실패·반려

- 직원이 지명한 동료를 팀장이 반려하는 경우 (이해관계 등)
- 직원이 재지명
- 회사별 정책으로 팀장 직접 지명 권한 부여 가능

### 7.8 평가 미제출자 (Overdue)

- 마감일 도래 시 자기평가·상사평가 미제출 직원은 자동 Overdue 마킹
- 인사담당자가 후속 처리 (마감 연장 / 대리 입력 / 평가 대상 제외)

### 7.9 사이클 취소

- 사이클 중 회사 정책 변경·합병 등으로 취소
- 인사담당자가 수동 취소 처리 + 사유 + 진행 데이터 처리 결정 (보존 / 폐기)
- 직원에게 자동 안내

### 7.10 펄스 설문 응답률 저조

- 응답률 50% 미만이면 인사담당자에게 알림
- 응답률을 높이기 위한 리마인더 자동 발송 (회사 설정)
- 응답률 저조 사유 분석 (귀찮음·불신·시간 부족 등)은 별도 진단 필요

### 7.11 1:1 미팅 무단 불참

- 예정 시각에 양쪽 미참여 → 상태 "무단 불참(NO_SHOW)"
- 빈번한 불참은 1:1 데이터 분석에서 패턴 감지 (회사 건강도 지표로 활용)

### 7.12 동료 평가 익명성 침해 우려

- 동료 평가자가 3명 미만인 경우 익명성 보장 어려움
- 시스템이 평가자 수 부족 시 경고 → 추가 지명 권고
- 정 부족한 경우 회사 정책으로 동료 평가 생략 결정 가능

## 8. 자주 묻는 질문

**Q. 내 평가 결과는 언제 볼 수 있나요?**
A. 캘리브레이션이 완료되고 인사담당자가 결과 통보 처리(FINALIZED → 공개)한 후 조회 가능합니다. 그 이전에는 시스템이 자동 마스킹하여 결과가 보이지 않습니다.

**Q. 동료가 나를 평가한 내용을 볼 수 있나요?**
A. 평가 내용은 결과 공개 시점에 익명으로 표시됩니다. 누가 작성했는지는 알 수 없습니다 (동료 평가 익명성 정책).

**Q. 자기평가를 잘못 제출했어요. 수정할 수 있나요?**
A. 제출 후 직원 본인은 수정 불가입니다. 마감 전이라면 인사담당자에게 재제출 요청을 하세요. 마감 후 정정은 어렵습니다.

**Q. 목표를 사이클 도중에 바꿔야 해요. 어떻게 하나요?**
A. `성과 → 내 목표`에서 목표 개정(Revision) 신청. 변경 사유와 함께 팀장 결재 받으면 새 버전이 활성화됩니다. 기존 버전은 이력으로 보존됩니다.

**Q. 동료 평가자를 누가 정하나요?**
A. 직원 본인이 3~5명 지명 후 팀장이 승인합니다. 팀장이 반려하면 재지명합니다. 회사 정책에 따라 팀장 직접 지명도 가능합니다.

**Q. 4등급 체계(O/E/M/S)는 어떻게 적용되나요?**
A. 성과·역량 각각 4등급 부여 → 2축 9-Block 매트릭스에 위치 결정. 가이드라인은 O 10% / E 30% / M 50% / S 10%이며 캘리브레이션에서 부서·전사 분포 조정합니다.

**Q. 캘리브레이션은 강제 분포(force-rank)인가요?**
A. 회사 정책에 따라 다릅니다. 가이드라인 권장 모드(분포 차이 발견 시 매니저에게 조정 권고)와 강제 분포 모드(가이드라인대로 강제) 중 선택 가능합니다. 현재는 권장 모드가 기본입니다.

**Q. 분기 체크인은 평가에 영향을 주나요?**
A. 분기 체크인은 평가 점수에 직접 반영되지 않습니다. 매니저가 평가 시 참고하는 자료로 활용됩니다. 단, 의무 체크인 모드에서는 미완료 시 사이클 진행이 차단될 수 있습니다.

**Q. 휴직 중인데 평가는 어떻게 되나요?**
A. 휴직 활성화 시 평가 진행이 일시 중지됩니다. 복직 후 회사 정책에 따라 해당 사이클 완료 또는 다음 사이클부터 참여합니다. [loa.md §7.3](loa.md) 참조.

**Q. 평가 결과가 보상에 어떻게 반영되나요?**
A. 평가 사이클이 결과 확정(FINALIZED) → 보상 기획(COMP_REVIEW) 단계로 이관됩니다. 등급 × 급여 밴드 위치 조합으로 인상률이 추천되고 인사담당자가 검토·확정 후 다음 급여 회차에 반영됩니다. 상세는 [compensation.md](compensation.md) 참조.

**Q. 팀장이 바뀌면 진행 중인 평가는 어떻게 되나요?**
A. 신규 팀장이 상사평가를 입력합니다. 이전 팀장이 입력한 데이터는 참고용으로 보존되지만 최종 평가는 신규 팀장 권한입니다.

**Q. 1:1 미팅에서 AI가 자동 요약한 내용이 부정확해요. 수정할 수 있나요?**
A. AI 요약은 참고용이며 직원·팀장이 수동으로 노트를 수정할 수 있습니다. AI 요약과 별도로 사람이 작성한 노트가 우선합니다.

**Q. 펄스 설문 응답은 정말 익명인가요?**
A. 네. 응답자 신원은 시스템에 저장되지 않으며 부서·직급 등 메타데이터만 집계용으로 보관됩니다. 응답 내용으로 응답자를 추정할 위험이 있으면 부서별 응답이 일정 수 이상일 때만 결과 공개합니다.

## 9. 알려진 제약

| # | 항목 | 상태 |
|---|------|------|
| 1 | 캘리브레이션 결과 강제 변경 워크플로 단순화 | 전사관리자가 결재 없이 변경 + 사유만 기록. 별도 결재 흐름 부재. **추후 개선** — 등급 변경 결재 흐름 (이중 검증) |
| 2 | 매니저 변경 시 이전 평가 데이터 자동 이관 부재 | 이전 팀장 데이터는 보존되나 신규 팀장이 자동으로 모든 컨텍스트 받지 않음. 수동 인수 필요. **추후 개선** — 매니저 변경 시 평가 컨텍스트 자동 인수 |
| 3 | 휴직 중 평가 자동 처리 부재 | 휴직 활성화 시 사이클 진행은 멈추지 않고 인사담당자 수동 제외 필요. **추후 개선** — 휴직 활성화 시 자동 평가 대상 제외 옵션 ([loa.md §9 #3](loa.md) 과 통합) |
| 4 | 사이클 중 직원 입사 처리 단순화 | 신규 입사자는 수습 제외 옵션 단순 적용. 입사일 기반 비례 평가 (Pro-rated)는 미지원. **추후 개선** — 입사일 기반 가중 평가 |
| 5 | 동료 평가 자동 추천 부재 | 직원이 동료 지명을 직접 결정. 협업 데이터 기반 추천 부재. **추후 개선** — 1:1·프로젝트 이력 기반 동료 자동 추천 |
| 6 | AI 편향 검출 결과 자동 차단 부재 | 평가 텍스트의 편향 표현을 감지해도 제출 자체는 막지 않음 (경고만). **추후 개선** — 회사 정책으로 차단 옵션 |
| 7 | 펄스 설문 결과 부서별 최소 응답 임계값 단순화 | 익명성 보장을 위한 최소 응답 수 임계값이 회사 정책으로 단순 설정. 직급·연령 등 다차원 익명성 검증은 부분. **추후 개선** — 다차원 익명성 검증 |
| 8 | 결과 통보 후 정정 워크플로 부재 | 통보 후 정정은 인사담당자 수동. 별도 정정 결재·이력 추적 단순. **추후 개선** — 정정 결재 + 직원 재통지 자동화 |
| 9 | 9-Block 라벨 회사별 차등 부분 지원 | EmsBlockConfig로 블록 라벨·임계값 설정 가능하나 회사별 다른 블록 수 (예: 4-Block, 16-Block)는 미지원. **추후 개선** — 블록 그리드 크기 회사별 설정 |
| 10 | 인상률 매트릭스 자동 적용 부분 구현 | 평가 결과 → 인상률 추천은 가능하나 인사담당자 확정·결재 흐름은 수동 처리 부분. **추후 개선** — 보상 모듈 통합 자동 흐름 ([compensation.md](compensation.md) 와 통합) |
| 11 | 다국어 평가 양식 부분 지원 | 평가 양식·1:1 가이드·분기 체크인 질문 등이 한국어 중심. 영문·중국어 등 다국어 부분 지원. **추후 개선** — 회사 locale 기반 다국어 양식 |
| 12 | 승계 계획 자동 후보 추천 부재 | 9-Block 결과로 후보 자동 추천은 부분. 잠재력 평가는 인사담당자 수동 입력. **추후 개선** — 9-Block + 1:1·평가 이력 기반 자동 추천 |
| 13 | 평가 사이클의 모듈 간 강결합 | 사이클 단계 진행이 보상 모듈 단계와 1:1 강결합. 평가만 진행하고 보상은 별도 사이클 운영하는 시나리오 미지원. **추후 개선** — 평가/보상 사이클 분리 옵션 |
| 14 | PerformanceGrade enum 마이그레이션 ADR PENDING | ADR `2026-04-06-performance-grade-enum-fix.md`가 PENDING 상태. 코드 schema는 이미 CEO 확정안(O/E/M/S) 반영. 운영 DB 기존 데이터(E/M_PLUS/M/B → O/E/M/S) 마이그레이션 작업 별도 필요. **추후 개선** — ADR 종결 + 마이그레이션 실행 |

- `pipeline.ts` 사이클 상태 전이 — 9단계 순차 강제. 역행·점프 시 데이터 무결성 깨짐. 트랜잭션 내 게이트 검증 필수
- `data-masking.ts` — FINALIZED 전 결과 마스킹 핵심 로직. 누락 시 직원이 미공개 결과 조회 가능 (보안 사고). 모든 결과 API는 본 마스킹 적용 필수
- `quarterly-review-masking.ts` — 분기 체크인 양쪽 제출 + 시점 도달 전 마스킹. 매니저 시뮬레이션·CEO 권한 우회 검증
- `PerformanceGrade` 마이그레이션 (ADR `2026-04-06`) — 운영 DB의 기존 row `E/M_PLUS/M/B` → `O/E/M/S` 변환 필요. Migration SQL 작성 시 PostgreSQL ALTER TYPE 제약 주의 (enum 값은 트랜잭션 내 변경 불가)
- `originalGradeEnum` vs `finalGradeEnum` — 캘리브레이션 후 두 값을 모두 보존. UI에서 잘못 노출 시 매니저 원래 평가가 직원에게 노출되어 갈등 가능. 항상 finalGradeEnum 노출이 원칙
- `merit-matrix.ts` — 등급 × 밴드 위치 매트릭스 계산. 매트릭스 누락 셀(예: O × HIGH)이 있으면 fallback 정책 명확화 필요
- 동료 평가 익명성 — `peerReviewMinCount`가 작은 부서에서 응답자 식별 가능 위험. 최소 3명 정책 강제 + 부서·직급 매트릭스 분석 시 추가 검증
- AI 평가 초안 (`AiEvaluationDraft`) — AI 생성 결과가 직원의 자기평가에 그대로 반영되면 평가 진정성 훼손. 직원 본인 수정 필수 정책 필요
