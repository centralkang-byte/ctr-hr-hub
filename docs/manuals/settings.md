## 1. 개요

### 1.1 무엇을 다루는가

CTR HR Hub의 마스터데이터·설정 모듈은 다음을 통합 관리합니다.

- **조직 마스터** — 회사·부서·직위·직급·호칭·직무·근무지 (인사관리 §2 참조)
- **권한 마스터 (Role + Permission)** — 시스템 역할 5종 + 권한 매트릭스
- **결재 흐름 설정** — 모듈별 결재 단계·결재자 역할 등록 (결재 매뉴얼 §6.3 참조)
- **근태 설정** — 표준 근무시간·52시간 임계값·교대·유연근무·타임존
- **휴가 설정** — 휴가 유형·연차 부여 규칙·마이너스 연차 정책
- **온보딩·오프보딩 설정** — 체크리스트 템플릿
- **급여 설정** — 4대보험 요율·비과세 한도·세금 구간·환율·수당·공제 마스터
- **보상 설정** — 급여 밴드·인상률 매트릭스·보너스 규칙
- **평가 설정** — 평가 사이클·9-Block 매트릭스·역량 라이브러리·캘리브레이션 가이드라인
- **채용 설정** — 충원 요청 결재 흐름·평가 양식
- **시스템 설정** — 회사 프로세스 설정·테넌트 설정 (전사 공통)
- **알림·통합 설정** — Teams 웹훅·이메일·Azure AD·외부 API

### 1.2 도입 의도

- **회사별 차등 정책** — 13개 법인 각자 다른 정책(노동법·통화·근무시간·휴가 등)을 회사별 설정으로 분리
- **본사 표준 + 법인 오버라이드** — 전사 표준 정책을 본사가 정의하고 각 법인이 필요 시 오버라이드
- **마스터데이터 SSOT** — 부서·직급·역할 등 핵심 마스터는 한 곳에서 관리, 모든 모듈이 참조
- **회사 정책 유연성** — 회사별 워크플로(예: 결재 단계 수, 자동 승인 일수)를 코드 변경 없이 설정에서 조정
- **감사 가능성** — 모든 설정 변경에 변경자·시점 기록

### 1.3 모든 모듈과의 관계

본 매뉴얼은 **모든 다른 매뉴얼의 메타 매뉴얼**입니다. 각 모듈의 정책 정의 화면을 모은 통합 진입점.

| 설정 영역 | 관련 매뉴얼 |
|----------|------------|
| 조직 마스터 (회사·부서·직위 등) | 인사관리 §2 |

| 휴가 유형·정책 | 휴가 §2 |
| 휴직 유형 | 휴직 §2 |
| 4대보험·세금·환율 | 급여 §2.7 |
| 급여 밴드·인상률 매트릭스 | 보상 §2.2·§2.3 |
| 평가 사이클·9-Block | 성과 §2 |
| 채용 결재 흐름 | 채용 §3.2 / 결재 §2.1 |

## 2. 설정 카테고리

`설정` 메뉴는 다음 6개 대분류로 구성됩니다 (코드: `settings-config.ts` — organization, attendance, payroll, performance, recruitment, system). 보상은 별도 카테고리가 아니라 **급여(Payroll) 하위 영역**입니다.

### 2.1 시스템 (System)

- **결재 플로우** — 모듈별 결재 흐름 등록 (결재 매뉴얼 §6.3)
- **Teams 웹훅** — 알림 발송 채널 설정
- **테넌트 설정** — 전사 공통 설정 (로고·도메인·기본 통화 등)
- **회사 프로세스 설정** (CompanyProcessSetting) — 회사별 미세 조정 가능한 정책 키-값 저장소

### 2.2 조직 (Organization)

- **회사** — 13개 법인 등록·수정 (인사관리 §2.1)
- **부서** — 회사별 부서 트리 (인사관리 §2.2)
- **직위·직급·호칭** — 직위/직급/호칭 매스터 + 매핑 (인사관리 §2.3·§2.4)
- **직군** — JobCategory
- **근무지** — WorkLocation
- **역할·권한** — Role/Permission/RolePermission

### 2.3 근태 (Attendance)

- **근태 정책** — 표준 시간·52시간 임계값·차단 활성화 (출퇴근 §2.6)
- **유연근무** — 코어타임·일일 최소 시간
- **교대 패턴** — 회사별 교대 패턴 정의
- **공휴일** — 회사·연도별 공휴일 등록
- **휴직 유형** — 회사별 휴직 유형 마스터 (휴직 §2)
- **휴가 유형** — 회사별 휴가 유형 마스터 (휴가 §2)

### 2.4 채용 (Recruitment)

- **공통 평가 양식** — 면접 평가 템플릿
- **역량 라이브러리** — 면접·성과 평가용 역량 정의
- **오프보딩 체크리스트** — 퇴직 유형별 체크리스트 (온보딩 §3.5)
- **온보딩 템플릿** — 시나리오별 체크리스트 (온보딩 §3.1)

### 2.5 급여 (Payroll)

급여 카테고리는 일반 급여 항목과 보상(Compensation) 하위 영역을 포함합니다.

**급여 일반**
- **4대보험 요율** — 연도별 갱신 (급여 §3.3)
- **비과세 한도** — 식대·차량유지비 등 (급여 §3.5)
- **세금 구간** — 회사·국가·세금유형·연도별 (급여 §3.3)
- **수당 마스터** — 회사별 수당 항목 (급여 §2.2)
- **공제 마스터** — 회사별 공제 항목 (급여 §2.3)
- **환율** — 월별 통화쌍 환율 (급여 §3.11)
- **업로드 매핑** — 해외 급여 결과 업로드 매핑 (급여 §3.1)

**보상 (Compensation) — 급여 하위 영역**
- **급여 밴드** — 회사·직급·직군별 (보상 §2.2)
- **인상률 매트릭스** — 등급 × 밴드 위치 (보상 §2.3)
- **보상 설정 (CompensationSetting)** — 급여 항목·인상률·보너스 규칙
- **승급 설정 (PromotionSetting)** — 승급 정책
- **연말정산 공제 항목** — 한국 연말정산 설정 (급여 §3.12)
- **법정 의무 교육 설정** — 산업안전·성희롱 예방 등 의무 교육

### 2.6 성과 (Performance)

- **평가 사이클** — 사이클 생성·진행 (성과 §2.1)
- **9-Block 매트릭스 설정** — EmsBlockConfig (성과 §2.4)
- **역량 라이브러리** — BEI 평가용
- **MBO/BEI 가중치** — 회사·사이클별 (성과 §3.5)
- **체크인 모드** — 의무/권장 (성과 §3.3)
- **캘리브레이션 가이드라인** — 블록별 권장 % (성과 §3.7)
- **성과 통지 정책** — 마스킹 해제 시점

## 3. 핵심 정책

### 3.1 본사 표준 + 법인 오버라이드 패턴

대부분의 설정은 본사 표준값을 정의하고 각 법인이 필요 시 오버라이드합니다.

```
[회사 미지정 (companyId = null)] = 본사 표준
        │
        ↓
[회사 X 오버라이드 (companyId = X)]
        │
        └─ 회사 X 직원에게는 오버라이드 적용, 미정의 시 본사 표준 fallback
```

**적용 모듈**: 결재 흐름·근태 설정·휴가 설정·보상 설정·평가 설정 등.

### 3.2 마스터데이터 검증 게이트

설정 변경 시 다음을 자동 검증:

| 검증 | 의미 |
|------|------|
| 참조 무결성 | 부서 삭제 시 소속 직원 존재 검사 (있으면 차단) |
| 역할 매핑 | 결재자 역할(부서장 등)이 실제 보유자 있는지 검증 |
| 시점 충돌 | 시점 기반 설정(급여 밴드·매트릭스 등)이 같은 시점 중복 없는지 |
| 권한 일관성 | Permission·Role·RolePermission 매트릭스 정합성 |
| 형식 검증 | 코드·이름 unique, 통화 enum, 날짜 형식 등 |

### 3.3 시점 기반 설정 (Time-based)

다음 설정은 시점별 시행:

- 급여 밴드 (effectiveFrom/To)
- 인상률 매트릭스 (사이클별)
- 4대보험 요율 (연도별)
- 비과세 한도 (연도별)
- 세금 구간 (effectiveFrom/To)
- 환율 (연·월·통화쌍)
- 휴가 정책 (시점)
- 휴직 유형 (deletedAt soft delete)

설정 변경 시 새 row 생성 + 이전 row의 종료일 설정 (Append-only 원칙, 인사관리 §3.1과 유사).

### 3.4 권한 매트릭스 (Role × Permission)

**시스템 역할 5종** (Role):

| 코드 | 한글 | 의미 |
|------|------|------|
| EMPLOYEE | 일반 직원 | 본인 데이터 + 자기 서비스 |
| MANAGER | 팀장 | 직속 부하 데이터 + 결재 |
| HR_ADMIN | 인사담당자 | 본인 법인 전체 |
| EXECUTIVE | 임원 | 본인 법인 전체 + 일부 전사 |
| SUPER_ADMIN | 전사관리자 | 13개 법인 전체 + 설정 |

**Permission** = 모듈 × 액션 (예: `employees:view`, `payroll:manage`, `leave:approve`)

**RolePermission** = Role과 Permission의 매핑

회사 정책으로 역할 코드별 권한 추가·제거 가능 (단, SUPER_ADMIN은 bypass).

### 3.5 결재 흐름 설정 (모듈별)

회사별·모듈별 결재 흐름을 등록합니다. 모듈 SSOT는 [approval.md §5.2](approval.md) 참조 (모듈별 실제 연동 상태 + 휴가는 [leave.md](leave.md) §5 참조).

### 3.6 회사 프로세스 설정 (Key-Value Store)

`CompanyProcessSetting`은 회사별 미세 조정 가능한 정책의 키-값 저장소입니다.

- **settingType** (카테고리) + **settingKey** (키) + **settingValue** (JSON 값)
- 코드 변경 없이 회사별 다른 동작 활성화
- 사용 예:
  - `payroll.overtimeRates`: 한국 1.5/2배 가산율 → 베트남 1.5/1.5/3배
  - `payroll.minWage`: 한국 10,030원 → 회사별 최저시급
  - `leave.useStatutoryAccrual`: 한국 법정 연차 부여 사용 여부
- 모듈 코드가 이 설정을 조회하여 동작 분기

### 3.7 시드 vs 운영 DB 정합

13개 법인 모든 마스터데이터는 시드(`prisma/seeds/*.ts`)로 초기 생성됩니다.

- **시드 = 초기값**: 처음 배포·테스트 환경 구성 시 시드 적용
- **운영 DB = 실제값**: 인사담당자가 UI에서 변경한 결과가 실제 동작

> **데이터 흐름 주의**: 시드 정의와 운영 DB의 실제값은 시간이 지나면서 차이(drift)가 발생할 수 있습니다. 특히 다음 영역:
> - 4대보험 요율: 매년 1월 갱신 (시드 + 운영 DB 동시 갱신 필요)
> - 비과세 한도: 매년 갱신
> - 휴가·휴직 유형: 시드는 정확하나 운영 DB에 적재 안 된 경우 (휴가 §9 #1)
> - 결재 흐름: 시드에 등록되어도 모듈 코드 미연동 (결재 §9 #3·#4)
>
> 정합성 검증 도구: `scripts/seed-validate-phase6b.ts` (휴가 모듈 32 checks). 다른 모듈은 별도 검증 도구 미구현.

### 3.8 자동 알림 + 운영 트리거

- **마스터 변경 알림**: 결재 흐름·급여 밴드·휴가 정책 등 핵심 설정 변경 시 인사담당자에게 변경 이력 알림
- **연도별 갱신 임박**: 4대보험 요율·비과세 한도 등 매년 갱신 필요 항목 D-7 알림 (현재 부분 구현)

## 4. 워크플로

### 4.1 설정 변경 라이프사이클

```
[인사담당자 또는 전사관리자 변경]
       │
       ↓
[변경 시도] — UI 입력
       │
       ↓
[검증 게이트]
       │     · 참조 무결성
       │     · 시점 충돌
       │     · 형식 검증
       ├── 통과 ──→ [변경 적용]
       │              │
       │              ├── 시점 기반 항목: 이전 row endDate + 새 row 생성
       │              ├── 단순 항목: 기존 row UPDATE
       │              └── 변경 이력 기록 (감사)
       │
       └── 실패 ──→ [오류 메시지 + 변경 취소]
```

### 4.2 본사 표준 설정 → 법인 오버라이드 라이프사이클

```
[전사관리자: 본사 표준 등록]
       │     · companyId = null
       │     · 모든 법인에 기본 적용
       ↓
[법인별 인사담당자: 오버라이드 등록]
       │     · companyId = 본인 법인
       │     · 본사 표준 대신 자기 법인에 적용
       ↓
[모듈 코드: 조회 시 우선순위]
       │     1. 회사 X 오버라이드 (있으면)
       │     2. 본사 표준 (없으면)
```

### 4.3 4대보험·세금·비과세 연도별 갱신

매년 1월 인사담당자(전사관리자) 수동 갱신:

```
[전년 12월: 정부 발표 확인] — 노동부·국세청
       │
       ↓
[시드 파일 갱신] — `prisma/seeds/*` 코드 변경 + PR
       │
       ↓
[운영 DB 갱신]
       │     · 새 연도 row 추가 (year = 2026, isActive = true)
       │     · 이전 연도 row 보존 (isActive 유지, 이력 추적)
       ↓
[1월 1일 부터 새 연도 row 자동 적용]
```

> **데이터 흐름 주의**: 시드 갱신과 운영 DB 갱신은 별개 작업입니다. 둘 다 안 하면 잘못된 요율 사용. **추후 개선** — 자동 정부 API 연동 (§9 #2).

## 5. 역할별 권한

### 5.1 행위 주체 매트릭스

| 작업 | 직원 | 팀장 | 인사담당자 | 전사관리자 |
|------|:----:|:----:|:---------:|:---------:|

## 7. 특수 상황 처리

### 7.1 본사 표준 변경 시 법인 오버라이드 영향

- 본사 표준 변경 시 오버라이드 없는 법인은 자동 새 값 적용
- 오버라이드 있는 법인은 변경 영향 없음 (기존 오버라이드 유지)
- 본사 표준과 오버라이드 차이가 큰 경우 인사담당자에게 알림 (현재 미구현)

### 7.2 회사 마스터 삭제 (법인 폐쇄)

- 회사 마스터 삭제 시 소속 직원·발령·급여 등 모든 데이터 영향
- 실제 운영에서는 hard delete 차단, soft delete만 허용 (deletedAt)
- 폐쇄 법인 데이터는 영구 보존 (감사·법규 요구)

### 7.3 직위·직급 폐지

- 직위 폐지 시 보유 직원의 발령 자동 처리 안 함 (인사담당자 수동)
- 직급 폐지 시 같은 직급 직원 일괄 매핑 변경 필요
- 일괄 인사이동 화면 (인사관리 §4.3) 활용 권장

### 7.4 권한 변경 시 즉시 반영

- Role × Permission 매트릭스 변경 시 영향받는 직원에게 즉시 반영
- 활성 세션 사용자는 다음 요청 시 새 권한 적용
- 권한 회수의 경우 즉시 효력 (보안)

### 7.5 시점 기반 설정의 미래 시행

- 급여 밴드·세금 구간 등을 미래 시행일로 등록
- 시행일 도래 시 자동 활성, 이전 row의 종료일 자동 설정
- 시뮬레이션 시 미래 row까지 시각화

### 7.6 회사 프로세스 설정 키 오타·삭제

- `CompanyProcessSetting`은 자유 키-값이라 오타 시 모듈 코드가 fallback 값 사용
- 키 삭제 시도는 차단 (settingKey unique 제약)
- 코드 변경 시 사용하는 키도 함께 변경 — drift 위험

### 7.7 설정 일괄 백업·복원 부재

- 현재 설정 일괄 export·import 기능 없음
- 운영 DB 백업·복원만 가능 (인프라 수준)
- 단일 설정 변경 실수 시 변경 이력 기반 수동 복원
- **추후 개선** 항목 (§9 #5)

### 7.8 다국어 마스터 부분 지원

- 회사·부서·직위 이름은 한국어·영문 두 필드 지원
- 휴가·휴직 유형도 한·영 + 일부 현지어 (스페인어·중국어 등) 시드
- 통지서·체크리스트·면담 양식은 한국어 중심 — 다국어 부분 (각 모듈 §9 참조)

## 8. 자주 묻는 질문

**Q. 설정 변경하려면 어디로 가야 하나요?**
A. 사이드바 하단 `설정` 메뉴에서 6개 대분류(조직·근태·급여·성과·채용·시스템 — 코드 `organization, attendance, payroll, performance, recruitment, system`)로 분류된 탭으로 이동합니다. 보상은 별도 탭이 아니라 **급여 탭 하위 영역**입니다. 본인 권한에 따라 보이는 탭이 다릅니다.

**Q. 본사 표준과 법인 오버라이드의 차이는?**
A. 본사 표준은 모든 법인에 기본 적용되며 회사 미지정(companyId=null) 설정입니다. 법인 오버라이드는 특정 법인이 자기 정책을 등록한 것으로, 해당 법인 직원에게는 오버라이드가 우선 적용됩니다.

**Q. 4대보험 요율은 누가 갱신하나요?**
A. 매년 1월 정부 발표 후 전사관리자가 수동 갱신합니다. 시드 파일과 운영 DB 둘 다 갱신 필요. 자동 정부 API 연동은 **추후 개선** 항목입니다.

**Q. 직급을 새로 만들 수 있나요?**
A. 인사담당자가 `설정 → 조직 → 직급`에서 회사별로 새 직급을 등록할 수 있습니다. 직급 변경은 시점 기반이며 기존 직원에게 매핑되어야 합니다 (일괄 매핑 시 인사이동 화면 활용).

**Q. 결재 흐름을 변경해도 진행 중 결재에 영향이 가나요?**
A. 진행 중인 결재는 시작 시점 흐름을 그대로 따릅니다. 새 흐름은 변경 후 신청 건부터 적용됩니다 (결재 §9 #9).

**Q. 휴가 유형을 삭제할 수 있나요?**
A. 사용 이력이 있는 유형은 hard delete 차단, soft delete(deletedAt)만 가능합니다. 비활성화하면 새 신청에는 안 보이지만 과거 이력은 보존됩니다.

**Q. 회사 프로세스 설정 (CompanyProcessSetting)은 누가 만지나요?**
A. 코드 변경 없이 회사별 동작을 조정하는 키-값 저장소입니다. 인사담당자·전사관리자가 일부 키를 UI로 조정할 수 있으나, 새 키 추가는 보통 개발자가 코드 변경과 함께 수행합니다.

**Q. 권한 매트릭스를 바꾸면 영향이 즉시 가나요?**
A. 네. 다음 요청부터 새 권한 적용됩니다. 보안 위험 시 (예: 권한 회수) 즉시 효력. 권한 부여는 활성 세션 사용자가 페이지 새로고침해야 반영됩니다.

**Q. 설정을 잘못 변경했어요. 이전 상태로 복원할 수 있나요?**
A. 시점 기반 설정(급여 밴드 등)은 새 row를 추가하여 정정합니다. 단순 키-값은 변경 이력을 보고 수동 복원해야 합니다. 일괄 복원 기능은 없으며 인프라 백업·복원이 필요할 수 있습니다 (§9 #5).

**Q. 다국어 설정은 어떻게 하나요?**
A. 회사·부서·직위 이름 등은 한·영 두 필드를 지원합니다. 직원의 locale에 따라 화면에 다른 언어 표시. 그러나 통지서·체크리스트는 한국어 중심이며 다국어 확대는 각 모듈의 추후 개선 항목입니다.

**Q. 시드와 운영 DB가 다르면 어떻게 되나요?**
A. 시드는 초기값, 운영 DB가 실제 동작값입니다. 운영 중에는 운영 DB가 우선됩니다. 시드는 새 환경 구성 시에만 사용되므로 두 값이 다른 것은 정상이나, 특정 시드(예: 휴가 유형)가 운영 DB에 적재 안 된 경우는 문제(휴가 §9 #1 참조).

> 본문이 정책 설명, 이 섹션은 개발/QA 참고용. HR은 건너뛰어도 무방합니다.

### 11.1 핵심 설정 모델

마스터·설정 모델 (모두 `prisma/schema.prisma`):

**조직 (인사관리 §11.1 참조)**:
- `Company`, `Department`, `Position`, `JobGrade`, `EmployeeTitle`, `GradeTitleMapping`, `JobCategory`, `Job`, `WorkLocation`

**권한**:
- `Role` (시스템 역할 5종 — EMPLOYEE/MANAGER/HR_ADMIN/EXECUTIVE/SUPER_ADMIN)
- `Permission` (모듈 × 액션 × 리소스)
- `RolePermission` (Role × Permission 매핑)
- `EmployeeRole` (직원별 역할 부여, 회사 scope)

**결재**:
- `ApprovalFlow` + `ApprovalFlowStep` (결재 §11.1)
- `ApprovalDelegation` (결재 §11.1)

**근태**:
- `AttendanceSetting`, `WorkSchedule`, `EmployeeSchedule`, `ShiftPattern`, `ShiftGroup`, `Holiday` (출퇴근 §11.1)

**휴가·휴직**:
- `LeaveSetting`, `LeaveTypeDef`, `LeaveOfAbsenceType` (휴가 §11.1, 휴직 §11.1)

**급여·보상**:
- `InsuranceRate`, `NontaxableLimit`, `TaxBracket`, `PayAllowanceType`, `PayDeductionType`, `ExchangeRate`, `PayrollImportMapping`, `SalaryBand`, `SalaryAdjustmentMatrix`, `CompensationSetting`, `PromotionSetting`, `YearEndDeductionConfig` (급여 §11.1, 보상 §11.1)

**성과**:
- `PerformanceCycle`, `EmsBlockConfig`, `CompetencyLibrary`, `CalibrationRule`, `EvaluationSetting` (성과 §11.1)

**채용·온보딩**:
- `OnboardingTemplate`, `OnboardingTask`, `OffboardingChecklist`, `OffboardingTask`, `OnboardingSetting` (온보딩 §11.1)

**시스템·통합**:
- `CompanyProcessSetting` — 회사별 미세 조정 키-값 저장소
- `TenantSetting` — 전사 공통 설정
- `TeamsWebhookConfig` — Teams 알림 채널
- `SocialInsuranceConfig` — 사회보험 설정 (해외 법인)
- `MandatoryTrainingConfig` — 법정 의무 교육
- `AnalyticsConfig`, `KpiDashboardConfig` — 분석·KPI 설정 ([insights.md](insights.md) §11.1 참조)

### 11.5 알려진 위험 코드 경로

- `CompanyProcessSetting` 키 오타 — 모듈 코드의 `getPayrollSetting('key')` 호출과 정확히 일치해야 동작. fallback 값 사용 시 silent miss
- 본사 표준 vs 법인 오버라이드 우선순위 — 조회 함수가 항상 `companyId asc`로 정렬하여 법인 오버라이드 우선 (PostgreSQL NULL LAST). 우선순위 변경 시 모든 호출자 영향
- Role × Permission 캐싱 — 변경 후 캐시 무효화 누락 시 stale 권한 적용. `invalidateCache(CACHE_STRATEGY.CODE_TABLES)` 필수
- 시점 기반 설정의 미래 row 동시 시행 — 같은 시점에 여러 row 활성 시 어느 게 적용될지 명확화 필요 (보통 created_at 최신)
- 마스터 soft delete (deletedAt) 일관성 — 모든 조회 함수가 `deletedAt: null` 필터 강제. 누락 시 폐기 데이터 노출
- 시드 적용 순서 — 의존 관계 있는 시드는 정해진 순서대로 실행 (회사 → 부서 → 직위 → 직원). 순서 오류 시 FK 위반
- 휴가 시드 미적재 (휴가 §9 #1) — `seedStatutoryLeaveTypes(35)` import 누락 패턴이 다른 모듈에도 잠재 가능
