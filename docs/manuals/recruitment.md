# 채용 매뉴얼 (Recruitment)

> **최종 갱신**: 2026-05-12
> **담당**: 인사관리팀 (채용)
> **상태**: Draft v1 (CEO 리뷰 전)

---

## 1. 개요

### 1.1 무엇을 다루는가

CTR HR Hub의 채용 모듈은 다음을 통합 관리합니다.

- **충원 요청 (Requisition)** — 부서가 신규 인력을 요청하는 사전 절차 (사번 발급 전 결재 단계)
- **채용 공고 (Job Posting)** — 외부·내부 공고 발행, 마감 관리, 사내 공석 공개
- **지원자 (Applicant)** — 지원자 마스터, 이력서·포트폴리오, 중복 지원자 검출
- **지원 (Application) 파이프라인** — 10단계 상태 머신 (지원 → 서류 → 1차/2차/최종 면접 → 오퍼 → 입사)
- **면접 (Interview) 일정·평가** — 일정 등록·캘린더 연동, 면접관 평가 (5단계 추천)
- **오퍼 (Offer)** — 연봉·시작일 협의, 응답 추적
- **입사 전환 (Convert to Employee)** — 합격자를 직원 마스터로 변환 (입사 발령 자동 생성)
- **인재 풀 (Talent Pool)** — 합격 보류·자진 사퇴 등 우수 지원자의 향후 채용 시 재활용
- **사내 공모 (Internal Jobs)** — 사내 직원의 다른 직무로의 이동 신청
- **채용 비용 추적 (Recruitment Cost)** — 광고비·에이전시 수수료·추천보너스·이전비 등
- **AI 스크리닝** — 지원자 자동 평가 (점수·요약)

### 1.2 도입 의도

- **채용 절차 표준화** — 충원 요청 → 결재 → 공고 → 지원 → 면접 → 오퍼 → 입사의 일관 흐름
- **인사관리규정·위임전결규정 준수** — 충원 요청·합격 결재가 정해진 결재 라우팅을 따름 (결재 매뉴얼 §2.1)
- **지원자 경험 향상** — 단계별 자동 알림, 빠른 응답
- **채용 데이터 분석** — 단계별 전환율·소요일·비용 KPI로 채용 효율성 개선
- **인재 풀 활용** — 한 번 불합격한 우수 지원자를 향후 채용에 재활용 (동의 기반)
- **인사관리 모듈과 연계** — 충원 요청은 직위(Position) 마스터와 연결, 입사 전환 시 발령 자동 생성

### 1.3 인사관리·결재 매뉴얼과의 관계

| 항목 | 본 매뉴얼 (채용) | 관련 매뉴얼 |
|------|----------------|-----------|
| 직위(Position) 마스터 | 본 매뉴얼이 참조 | 인사관리 매뉴얼 §2.4 |
| 충원 요청 결재 흐름 | 본 매뉴얼이 신청 | 결재 매뉴얼 §2.1 채용 |
| 결재자 자동 지정 | 본 매뉴얼이 RequisitionApproval 생성 | 결재 매뉴얼 §3.1·§11.2 |
| 입사 시 발령 생성 | 본 매뉴얼이 합격 → 직원 변환 시 트리거 | 인사관리 매뉴얼 §4.1 (HIRE 발령) |
| 신규 직원 온보딩 | 본 매뉴얼이 합격 후 트리거 | [onboarding.md](onboarding.md) |
| 평가 역량 라이브러리 | 본 매뉴얼이 참조 (면접 평가 시) | [performance.md](performance.md) |

## 2. 채용 구성 요소

### 2.1 충원 요청 (Requisition)

부서가 신규 인력을 요청하는 사전 절차. 결재 통과 후에야 공고를 낼 수 있습니다.

| 속성 | 설명 |
|------|------|
| 요청 번호 | REQ-001 형식 자동 발급 |
| 요청자 | 신청 직원 (일반적으로 본부장·팀장) |
| 부서 | 충원 부서 |
| 직위 | 기존 직위 연결 (또는 신규 직위) |
| 제목 | 충원 직무 제목 |
| 인원 | 충원 인원 수 |
| 직급·고용형태 | 채용할 직급(L1/L2 등) + 고용형태(정규/계약/인턴) |
| 사유 | 충원 필요성 (자유 텍스트) |
| 자격 요건 | 우대사항·필수역량 JSON |
| 긴급도 | 긴급 / 보통 / 낮음 |
| 목표일 | 충원 희망 시점 |
| 상태 | 작성 중 / 결재 중 / 승인 / 반려 / 취소 / 충원 완료 |

### 2.2 채용 공고 (Job Posting)

승인된 충원 요청을 외부·내부에 공개하는 공고.

| 속성 | 설명 |
|------|------|
| 제목·설명·자격요건 | 공고 내용 |
| 회사·부서·직위 | 채용 위치 |
| 직급·직군·고용형태 | 채용 조건 |
| 근무 형태 | 사무실 / 재택 / 하이브리드 |
| 급여 범위 | 최소·최대 (숨김 옵션) |
| 인원 | 충원 인원 |
| 마감일 | 지원 마감 |
| 모집 담당자 | 채용 담당 직원 |
| 사내 공개 여부 | 사내 공모 대상 |
| 상태 | 작성 중 (DRAFT) / 공개 (OPEN) / 마감 (CLOSED) / 취소 (CANCELLED) |

### 2.3 지원자 (Applicant) + 지원 (Application)

**지원자(Applicant)**: 사람 마스터. 이메일 기준 유일.

**지원(Application)**: 특정 공고에 지원한 기록. 같은 사람이 여러 공고에 지원하면 여러 Application 생성. 한 공고에 같은 지원자는 1회만(unique).

| 항목 | 설명 |
|------|------|
| 지원자 이름·이메일·전화 | 기본 정보 |
| 이력서·포트폴리오 | 첨부 파일 |
| 출처 | 직접지원 / 추천 / 에이전시 / 채용사이트 / 사내공모 |
| AI 스크리닝 점수 | 자동 평가 결과 (0~100) |
| AI 스크리닝 요약 | AI 생성 요약 |
| 인재 풀 동의 | 합격 보류 시 인재 풀 등록 동의 여부 |

### 2.4 지원 단계 (ApplicationStage) — 10단계

```
지원 → 서류 검토 → 1차 면접 → 2차 면접 → 최종 면접 → 오퍼 → 오퍼 수락 → 입사
                                                            ↘ 오퍼 거절
                                                            
모든 단계에서 → 불합격 (REJECTED)
```

각 단계 의미:

| 단계 | 의미 |
|------|------|
| 지원 (APPLIED) | 지원서 제출 완료 |
| 서류 검토 (SCREENING) | AI 또는 채용담당자 서류 평가 |
| 1차 면접 (INTERVIEW_1) | 실무진 면접 |
| 2차 면접 (INTERVIEW_2) | 팀장·본부장 면접 (선택 단계) |
| 최종 면접 (FINAL) | 임원·대표 면접 |
| 오퍼 (OFFER) | 연봉·시작일 제안 |
| 오퍼 수락 (OFFER_ACCEPTED) | 지원자가 수락 |
| 오퍼 거절 (OFFER_DECLINED) | 지원자가 거절 (종료) |
| 입사 (HIRED) | 직원으로 전환 완료 (종료) |
| 불합격 (REJECTED) | 모든 단계에서 가능 (종료) |

**진행 규칙**:
- **전진만 가능**: 뒤 단계로 되돌릴 수 없음 (실수 시 새 Application 생성 또는 인사담당자 수동 정정)
- **건너뛰기 허용** (특정 경로만):
  - 지원 → 1차 면접 (서류 통과 명백한 경우)
  - 1차 면접 → 최종 면접 (2차 생략)
- **종료 상태**: 입사·오퍼 거절·불합격 (이후 변경 불가)

### 2.5 면접 (Interview)

| 속성 | 설명 |
|------|------|
| 면접 종류 | 전화 / 화상 / 대면 / 패널 |
| 회차 | 1차 / 2차 / 최종 |
| 면접관 | 직원 (1명) |
| 일시·소요시간 | 시작 시각 + 분 단위 |
| 장소 / 회의 링크 | 대면 장소 또는 화상 회의 URL |
| 캘린더 이벤트 | Teams 자동 일정 등록 ID |
| 상태 | 예정 / 완료 / 취소 / 무단 불참 |

**면접 평가**:
- 전체 점수 (0~100)
- 역량별 점수 (JSON)
- 강점·우려점·종합 의견
- **추천** 5단계: 매우 추천 / 추천 / 중립 / 비추천 / 매우 비추천

### 2.6 오퍼 (Offer)

| 항목 | 설명 |
|------|------|
| 제안 연봉 | offeredSalary |
| 제안일 | offeredDate |

### 2.7 인재 풀 (Talent Pool)

합격 보류 또는 자진 사퇴한 우수 지원자를 향후 채용에 재활용.

| 항목 | 설명 |
|------|------|
| 등록 사유 | 자격 충족했으나 보류 / 자진 사퇴 / 과스펙 / 수동 등록 |
| 동의 여부 | 인재 풀 동의 (consentGiven) |
| 만료일 | 인재 풀 유효 기간 |
| 태그 | 분류 태그 (자유) |
| 상태 | 활성 / 연락 진행 중 / 만료 / 채용 완료 |

### 2.8 채용 비용 (Recruitment Cost) 8종

## 3. 핵심 정책

### 3.1 충원 요청 → 공고 → 입사 흐름

```
[1] 충원 요청 작성
    │
    ↓
[2] 결재 (본부장 1단계 — 결재 매뉴얼 §2.1 채용)
    │
    ↓
[3] 승인 → 공고 작성·발행
    │
    ↓
[4] 지원자 모집 + 단계별 진행
    │
    ↓
[5] 최종 합격 → 오퍼 → 수락
    │
    ↓
[6] 입사 전환 → 인사관리 모듈에 직원 등록 + HIRE 발령 생성
    │
    ↓
[7] 충원 요청 상태 "충원 완료"로 변경
```

### 3.2 결재 라우팅 (충원 요청)

위임전결규정상 충원 요청은 본부장 1단계 결재가 표준입니다 (HR 합의 후 본부장 전결).

- **결재자 자동 지정**: 시스템이 신청자의 부서 → 부서장(`headEmployeeId`)을 결재자로 자동 지정
- 결재 매뉴얼 §3.1 라우팅 알고리즘 적용
- 결재 매뉴얼 §11.2 `resolve-approval-flow.ts`의 `dept_head` 역할로 변환

**중요**: 충원 요청 결재는 `RequisitionApproval` 별도 모델 사용 (결재 매뉴얼 §11.1 모델 참조).

### 3.3 단계 전환 규칙 (Stage Machine)

지원 단계는 정해진 전환 경로만 허용됩니다.

**허용 전환**:
- 지원 → 서류 / 1차 면접 / 불합격
- 서류 → 1차 면접 / 불합격
- 1차 면접 → 2차 면접 / 최종 / 불합격
- 2차 면접 → 최종 / 불합격
- 최종 → 오퍼 / 불합격
- 오퍼 → 오퍼 수락 / 오퍼 거절 / 불합격
- 오퍼 수락 → 입사

**불가**:
- 뒤로 되돌리기 (잘못 진행 시 인사담당자가 정정 처리 또는 새 Application 생성)
- 종료 상태(입사·오퍼 거절·불합격)에서 어디로든

**불합격 시 사유 필수**: 사유 미입력 시 시스템이 차단.

### 3.4 사내 공모 (Internal Jobs)

사내 직원이 다른 부서·직무로 이동하기 위한 공모.

- 공고에 "사내 공개" 활성화 → 직원이 사내 공모 화면에서 조회·지원
- 직원이 신청하면 일반 Application과 동일 단계 진행 (다만 출처가 INTERNAL)
- 합격 시 입사 전환 대신 **부서이동 발령**(인사관리 매뉴얼 §3.2 TRANSFER) 자동 생성

### 3.5 중복 지원자 검출

이메일·전화·이름 조합으로 중복 지원자를 자동 감지.

- 새 지원자 등록 시 기존 마스터 검색 → 일치율 표시
- 중복 가능성 있으면 채용담당자에게 통합 또는 별도 등록 선택 안내
- 통합 시 `CandidateDuplicateLog`에 기록

### 3.6 AI 스크리닝 (서류 자동 평가)

지원서 제출 후 AI가 자동으로 평가:
- 점수(0~100), 강점·약점 요약 생성
- 채용담당자가 검토 후 다음 단계(서류 → 1차 면접) 결정
- 회사 정책으로 AI 점수 임계값 자동 필터링 가능 (예: 50점 미만 자동 서류 탈락 — 현재는 수동 검토)

### 3.7 인재 풀 활용

- 합격 보류 또는 자진 사퇴한 지원자 중 우수 인재를 풀에 등록 (지원자 동의 필요)
- 향후 유사 직무 공고 발행 시 인재 풀에서 후보 자동 추천 (현재 수동 조회)
- 풀 만료일 도래 시 자동 만료 처리 + 지원자에게 알림

### 3.8 자동 알림

- **지원 접수**: 지원자에게 접수 확인 이메일
- **단계 전환**: 지원자에게 단계 변경 알림 (서류 통과·면접 일정 확정 등)
- **면접 일정**: 지원자·면접관에게 캘린더 초대 + 리마인더 (1일 전)
- **오퍼 발송**: 지원자에게 오퍼 레터 이메일
- **결재 도착**: 충원 요청 결재자에게 알림 (결재 매뉴얼 §4.3)
- **불합격 통지**: 지원자에게 사유와 함께 통지

### 3.9 입사 전환 (Convert to Employee)

> HIRE 발령 정의는 [employee.md §3.2](employee.md) 참조

오퍼 수락 후 시작 예정일 도래 시 또는 채용담당자 수동 실행:

```
[지원자 + 지원 정보]
        │
        ↓
[Employee 마스터 생성]
        │     · employeeNo 자동 발급 (회사별 번호 체계)
        │     · 인적사항: 지원자 정보에서 이관
        │     · hireDate = expectedStartDate
        │
        ↓
[EmployeeAssignment 생성 (HIRE 발령)]
        │     · 부서: Requisition.department
        │     · 직위: Requisition.position (또는 공고 직위)
        │     · 직급: 충원 요청·공고에서
        │     · 고용형태: 충원 요청
        │     · 급여: offeredSalary
        │
        ↓
[Application.stage = HIRED, convertedEmployeeId 연결]
        │
        ↓
[온보딩 자동 시작] (선택, 온보딩 모듈 가동 시)
```

### 3.10 채용 비용 분석

채용 비용은 공고별·출처별·유형별로 집계됩니다.

- **공고당 평균 채용 비용**: 광고 대비 효율성
- **출처별 효율**: 추천 vs 에이전시 vs 채용사이트 vs 직접지원 → 어디가 비용 대비 효과 좋은가
- **소요일**: 공고 게재 ~ 입사까지 일수
- **단계별 전환율**: 지원 → 서류 → 면접 → 합격 비율

### 4.5 자동 알림 정리

- **지원 접수**: 지원자 이메일
- **단계 전환**: 지원자 이메일 (서류 통과·면접·합격·불합격)
- **면접 일정**: 지원자·면접관 캘린더 초대 + Teams 자동 일정 (옵션)
- **면접 평가 마감 임박**: 면접관에게 평가 입력 알림
- **오퍼 발송 / 응답 임박**: 지원자에게 응답 마감 안내
- **충원 요청 결재**: 결재자 알림
- **인재 풀 만료 임박**: 지원자·채용담당자 알림
- **신규 사내 공모**: 직원 사이드바·메일 알림 (옵션)

### 5.2 결재 라우팅

채용 모듈의 결재 흐름은 **결재 매뉴얼 §2.1 채용**에서 정의됩니다.

| 작업 | 결재자 | 단계 |
|------|--------|------|
| 충원 요청 | 본부장 (부서장 또는 임원) | 1단계 (HR 합의 후 본부장 전결) |

**참고**:
- 위임전결규정상 채용은 본부장 1단계 결재. HR 합의는 시스템 별도 결재가 아닌 사전 협의 단계
- 충원 요청 결재는 `RequisitionApproval` 별도 모델 사용 (결재 매뉴얼 §11.2 `validate-requisition-approver.ts`)
- 셀프 결재 차단 별도 가드 (결재 매뉴얼 §3.3) — 본부장이 본인 부서 충원 요청 직접 결재 불가

### 5.3 데이터 가시성

- **직원**: 사내 공모 + 본인 지원 이력만 조회 (다른 지원자 정보 절대 조회 불가)
- **팀장**: 본인 부서 진행 중 채용 + 본인 면접만 조회
- **채용 담당자 (인사담당자)**: 본인 법인 전체 지원자·면접·비용
- **전사관리자**: 13개 법인 전체

지원자 개인정보는 채용담당자만 접근. 면접관은 본인 면접 대상자만 일부 정보 조회.

## 6. 화면 안내

### 6.1 채용 대시보드 (인사담당자)

위치: `채용` 메인 또는 `채용 → 대시보드`

- **KPI 카드**: 활성 공고 / 신규 지원자 (이번 주) / 진행 중 단계별 인원 / 평균 채용 소요일
- **충원 요청 현황**: 결재 진행 중·승인된 요청 목록
- **공고 현황**: 활성·임박 마감 공고
- **단계별 파이프라인**: 칸반 보드 형식 (서류·1차·2차·최종·오퍼)
- **채용 비용 차트**: 출처별·유형별·월별 추이

### 6.2 채용 보드 (Kanban)

위치: `채용 → 보드`

- 단계별 컬럼 (지원 / 서류 / 1차 / 2차 / 최종 / 오퍼)
- 지원자 카드 드래그앤드롭으로 단계 이동
- 카드 클릭 → 지원자 상세 모달
- 필터: 공고·기간·출처

### 6.3 충원 요청 (Requisition)

위치: `채용 → 충원 요청`

- 요청 목록 + 상태별 필터
- 신규 요청 작성 폼
- 결재 대기 탭 (본부장이 본인 결재 건만 조회)
- 요청 상세: 진행 이력 + 연결된 공고

### 6.4 지원자 상세

위치: `채용 → 보드 → [지원자 카드 클릭]`

탭 구조:
1. **요약** — 지원자 기본 정보 + 현재 단계 + AI 스크리닝
2. **이력서** — 첨부 파일 미리보기 + 다운로드
3. **타임라인** — 단계 전환·면접·평가·메모 이력
4. **면접** — 일정·평가 결과
5. **메모** — 채용담당자 내부 메모
6. **오퍼** — 오퍼 정보·응답 추적

### 6.5 면접 일정 등록

위치: 지원자 상세 → 면접 탭 → 신규 일정

- 면접관 선택 (가능한 시간대 자동 추천)
- 일시·장소·종류 (전화/화상/대면/패널)
- Teams 자동 일정 등록 옵션
- 면접관·지원자 자동 알림

### 6.6 인재 풀

위치: `채용 → 인재 풀`

- 풀 멤버 목록 + 상태·태그 필터
- 만료 임박 후보 강조
- 후보 카드 → 연락 이력·메모 관리
- 인재 풀 → 새 공고에 후보 추천 (드래그 또는 버튼)

### 6.7 사내 공모

위치: `채용 → 사내 공모` (직원 메뉴)

- 직원 본인이 지원 가능한 사내 공석 목록
- 공고 카드 + 지원 버튼
- 본인 지원 이력 별도 탭

### 6.8 채용 비용 분석

위치: `채용 → 비용 분석`

- 공고별·출처별·유형별 차트
- 채용 효율 KPI (출처별 합격률·비용·소요일)
- 월간 추이

## 7. 특수 상황 처리

### 7.1 같은 지원자가 여러 공고에 동시 지원

- 시스템이 자동 허용 — 한 사람이 여러 공고에 동시 지원 가능
- 각 공고마다 별도 Application 생성
- 단, 같은 공고에 같은 이메일로 재지원 불가 (unique 제약)

### 7.2 중복 지원자 감지

- 새 지원자 이메일·전화·이름이 기존 마스터와 일치율 높음
- 채용담당자에게 "중복 가능성: 75%" 등 표시
- 통합 처리 시 두 지원자의 이력을 하나로 합침
- `CandidateDuplicateLog`에 통합 이력 기록

### 7.3 지원자가 단계 진행 중 자진 사퇴

- 채용담당자가 단계를 "불합격(REJECTED)"으로 전환 + 사유에 "자진 사퇴" 입력
- 우수 지원자였다면 인재 풀 등록 제안

### 7.4 오퍼 거절 시 인재 풀 등록

- 오퍼 거절 시 채용담당자가 인재 풀 등록 검토
- 지원자 동의 필요 (이메일로 동의 요청 또는 이미 인재 풀 동의된 경우 자동)
- 만료일 1~2년 후 자동 설정

### 7.5 충원 요청 후 헤드카운트 변경

- 충원 요청 승인 후 인원 변경 필요한 경우 (예: 3명 → 2명)
- 결재 완료 후 변경은 인사담당자가 수동 수정 + 사유 기록
- 인원 증가는 새 충원 요청 필요

### 7.6 사내 공모로 부서 이동 시 기존 직위 처리

- 사내 공모 합격 시 새 부서·직위로 부서이동 발령 자동 생성
- 기존 직위는 비어 있게 됨 → 후임자 충원 필요 시 새 충원 요청
- 자동 후임자 충원 트리거는 없음 (인사담당자 수동)

### 7.7 면접관 부재·교체

- 면접 일정 등록 후 면접관 출장·휴가 등으로 부재
- 채용담당자가 일정 수정 → 새 면접관 지정
- 지원자에게 자동 변경 알림

### 7.8 면접 무단 불참 (No-show)

- 지원자가 예정 시각에 불참
- 면접 상태 "무단 불참(NO_SHOW)"으로 변경
- 채용담당자가 후속 처리 (재일정 또는 불합격)
- 무단 불참 이력은 인재 풀 등록 검토 시 참고

### 7.9 입사 시작일 변경

- 오퍼 수락 후 시작 예정일 변경 필요 (이전 회사 인수인계 등)
- 채용담당자가 expectedStartDate 수정
- 입사 전환은 새 시작일 도래 후 실행

### 7.10 입사 전환 후 취소 (입사 철회)

- 입사 전환 완료 후 입사 전 취소 (드문 케이스)
- 인사담당자가 직원 마스터를 "취소(CANCELLED)" 처리 + Assignment 삭제 (감사 대상)
- 일반적 운영 권고: 입사 전환은 시작일 이후 실행하여 위험 최소화

### 7.11 해외 법인 지원자

- 해외 6개 법인 공고에 지원하는 지원자
- 현지 언어 지원서 (현재 한국어·영어 중심, 다국어 부분 지원)
- 면접·오퍼는 현지 통화 (RecruitmentCost·Application.offeredSalary는 통화 필드 보유)

## 8. 자주 묻는 질문

**Q. 충원 요청 없이 바로 공고를 낼 수 있나요?**
A. 시스템상 가능하지만 권장하지 않습니다. 충원 요청은 본부장 결재로 인력 충원 정당성을 검증하는 단계입니다. 결재 매뉴얼 §2.1 채용 정책에 따라 충원 요청 후 공고 발행이 표준입니다.

**Q. 지원 단계를 잘못 진행했어요. 되돌릴 수 있나요?**
A. 일반적으로 뒤로 되돌릴 수 없습니다 (단계 머신 정책). 인사담당자가 수동 정정 또는 새 Application 생성으로 처리합니다. 빈번한 정정이 필요하면 단계 정책 검토가 필요합니다.

**Q. 같은 사람이 여러 공고에 지원하면 어떻게 되나요?**
A. 가능합니다. 각 공고마다 별도 Application으로 진행됩니다. 단, 같은 공고에 동일 이메일로는 1회만 지원 가능합니다.

**Q. AI 스크리닝 점수가 낮은데 면접 보고 싶어요. 어떻게 하나요?**
A. AI 점수는 참고용이며 채용담당자가 최종 판단합니다. 점수와 무관하게 단계 진행이 가능합니다. 회사 정책으로 자동 필터링은 현재 비활성입니다.

**Q. 사내 공모 지원하면 현재 부서장이 알게 되나요?**
A. 현재 시스템 정책상 자동 통지는 없으나, 사내 공모 합격 시 부서이동 발령에서 양쪽 부서가 노출됩니다. 사전 협의 권장.

**Q. 인재 풀에 등록되려면 어떻게 해야 하나요?**
A. 지원 시 또는 단계 진행 중 채용담당자가 인재 풀 등록을 제안하고, 지원자가 동의해야 합니다. 동의 없이 등록되지 않습니다.

**Q. 면접관으로 배정됐는데 일정 충돌이 나요. 어떻게 하나요?**
A. 채용담당자에게 알려 일정 변경 또는 면접관 교체를 요청하세요. 시스템에서는 채용담당자가 면접 일정을 직접 수정합니다.

**Q. 오퍼 거절했는데 다시 지원할 수 있나요?**
A. 가능합니다. 새 공고에 새 Application 생성으로 지원하면 됩니다. 단, 거절 사유가 시스템에 기록되어 채용담당자가 참고할 수 있습니다.

**Q. 채용 비용은 어디서 입력하나요?**
A. `채용 → 비용 분석`에서 공고·출처·유형별로 등록합니다. 광고비·에이전시 수수료·추천 보너스 등 8종 유형 지원. 자동 연동(예: 에이전시 청구서 자동 입력)은 현재 미지원이며 수동 입력입니다.

**Q. 합격자가 입사 직전 취소하면 어떻게 처리되나요?**
A. 인사담당자가 직원 마스터를 취소 처리하고 발령을 삭제합니다. 운영 권고는 입사 시작일 이후 입사 전환을 실행하여 이런 위험을 최소화하는 것입니다.

**Q. 해외 법인도 같은 시스템으로 채용하나요?**
A. 네. 13개 법인 모두 동일 시스템 사용. 다만 현지 언어 지원서·면접·오퍼 등 일부 다국어 지원이 부분적입니다 (§9 #6 참조).

**Q. 본부장 결재 대신 다른 사람이 충원 요청 결재할 수 있나요?**
A. 위임 설정으로 가능합니다. 결재 매뉴얼 §3.4 위임(대결) 참조. 단, 현재 위임의 "전 업무" 범위는 휴가에만 동작하므로 채용 결재 위임은 인사담당자가 수동 처리해야 합니다 (결재 매뉴얼 §9 #1).

### 11.5 알려진 위험 코드 경로

- `stage-machine.ts` — **PROTECTED**. 단계 전환 매트릭스. 새 단계 추가·전환 경로 변경 시 architecture review 필수
- 입사 전환 트랜잭션 — Application → Employee + EmployeeAssignment(HIRE) 동시 생성. 트랜잭션 격리 + 실패 시 부분 데이터 정리 필수
- 충원 요청 결재 트랜잭션 — RequisitionApproval 단계별 진행. 결재 매뉴얼 §11.5 race condition 동일 위험
- 중복 지원자 통합 — 두 Applicant ID 통합 시 모든 관련 Application·Interview·Evaluation 이관. 데이터 정합성 위험
- 사내 공모 → 부서이동 자동 발령 — 합격 시 직원의 기존 발령 종료(endDate 설정) + 새 발령 생성. Append-only 원칙 준수 필수 (인사관리 매뉴얼 §3.1)
- `JobPosting.requirementCompetencies` JSON 자유 형식 — 스키마 검증 없음. 면접 평가 시 매칭 보장 안 됨
- `Application.aiScreeningScore` Range — 0~100 보장 안 됨 (DB 레벨 constraint 없음, 코드 책임)
- `TalentPoolEntry.expiresAt` 자동 만료 처리 — cron으로 status='expired' 전환 필요 (현재 구현 확인 필요)
