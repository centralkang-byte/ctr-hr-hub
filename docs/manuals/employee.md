# 인사관리 매뉴얼 (Employee + Org)

> **최종 갱신**: 2026-05-11
> **담당**: 인사관리팀
> **상태**: Draft v1 (CEO 리뷰 전)

---

## 1. 개요

### 1.1 무엇을 다루는가

CTR HR Hub의 인사관리 모듈은 다음을 통합 관리합니다.

- **조직 구조** — 회사·부서·직위 트리와 보고 라인 (실선·점선)
- **직원 정보** — 인적사항, 입사·퇴사일, 사번, 비상연락처, 사진, 다국어 이름
- **발령 이력 (Append-only)** — 입사부터 퇴사까지 모든 부서·직급·직위·법인·신분 변경의 시점별 이력
- **직급·호칭 체계** — 법인별 직급(rank) + 국가별 호칭(title) 분리 관리
- **고용 형태** — 정규직 / 계약직 / 파견 / 인턴, 수습 기간 추적, 계약 자동 전환 알림
- **법인 전환** — 그룹사 내 법인 이동 시 연차 정산·급여 재조정·근속 통산
- **대량 인사이동** — CSV 업로드로 부서이동·승진·법인전환·퇴직·급여변경 일괄 처리
- **권한 관리** — 직원별 역할(role) 부여, 부서장 지정
- **변경 이력 감사** — 모든 인사 발령의 누가·언제·왜 기록

### 1.2 도입 의도

- **인사관리규정(CP-G-02-01) 디지털화** — 9장 41조의 채용·수습·전보·승진·면직·발령·문서 절차를 시스템에서 운영
- **단일 진실 원천(SSOT)** — 직원의 현재 부서·직급·직위·법인은 항상 가장 최근 활성 발령 한 건에서 결정. 엑셀·이메일에 흩어진 인사 정보를 통합
- **다국적 법인 대응** — 13개 법인(국내 7 + 해외 6)에서 각각 다른 직급 체계·호칭·고용형태·노동법 준수
- **시점별 이력 추적** — Append-only 발령으로 "2025년 3월 1일 시점의 그 직원 부서는 어디였나?" 같은 시점 질의 가능
- **법규 정합성** — 한국 근로기준법, 각국 노동법, 그룹사 근속 통산 판례(대법원 2006다24098) 등 반영

### 1.3 휴가·휴직 매뉴얼과의 관계

| 항목 | 본 매뉴얼 (인사관리) | 휴가 / 휴직 매뉴얼 |
|------|---------------------|------------------|
| 직원 상태 변경 | 입사·퇴사·법인전환·발령 직접 변경 | 휴직 활성화 시 "휴직 중" 자동 전환 (간접) |
| 발령 생성 | 입사·전보·승진·퇴직 등 모든 발령 | 휴직 시작·복직 시 자동 발령 (휴직 매뉴얼 §4.2) |
| Assignment 모델 | 본 매뉴얼이 정의 (SSOT) | 휴직·휴가가 참조 |
| 근속 산입 | 회사·법인 전환 정책 정의 | 휴직 유형별 산입 여부는 휴직 매뉴얼 §3.4 |

| 코드 | 이름 | 국가 |

각 법인은 별도의 부서 트리·직급 체계·고용형태·노동법을 가집니다.

### 2.2 부서 (Department) — 트리 구조

부서는 회사 내에서 트리 구조로 구성됩니다.

- **계층**: 본부 → 실 → 팀 → 파트 등 (회사별 자유)
- **부서장**: 각 부서에 부서장 직원 1명 지정 (결재의 부서장 역할에 사용)
- **부서 코드**: 회사 내 유일 (예: `DEV-01`, `CN-DEV-01`)
- **비용센터 코드**: ERP 연계용 부서별 비용 코드
- **부서 이동 시**: 직원의 새 발령이 자동 생성됨 (§4.2)

### 2.3 직급 (JobGrade)·호칭 (EmployeeTitle) — 분리 관리

CTR HR Hub은 **직급(rank)**과 **호칭(name)**을 분리합니다.

- **직급**: 보상·승진의 기준이 되는 등급 (예: L1, L2, L3) — 회사별 정의, rankOrder로 정렬
- **호칭**: 명함·이메일에 사용되는 이름 (예: 매니저, 책임매니저, 전문리더) — 국가별 다른 명칭 가능
- **매핑**: 법인별로 직급↔호칭 1:1 매핑 (한국은 "L1=매니저", 미국은 "L1=Manager" 등)

**현재 직급 체계** (CEO 확인, 2026-03-27):
- L1 (매니저) / L2 (책임매니저)
- 전문리더 / 경영리더 (총 4단계)
- **향후 확장**: L1~L5 + 임원 + 호칭 분리 관리 예정

> **데이터 흐름 주의**: 현재 시스템은 L1/L2/전문리더/경영리더 4단계로 운영 중이지만, 인사관리규정 문서(CP-G-02-01 Rev.11)는 L1/L2 2단계만 명시합니다. 향후 L1~L5로 확장하려면 직급 리팩터링 작업이 필요합니다 (§9 #1 참조).

### 2.4 직위 (Position) — 보고 라인

직위는 조직도의 노드입니다. 직급보다 구체적이며 보고 관계를 갖습니다.

| 속성 | 설명 |
|------|------|
| 코드 | 회사 내 유일 (예: `POS-DEV-LEAD`) |
| 한국어/영문 이름 | 화면 표시용 |
| 부서 | 어느 부서에 속하는지 |
| 직급 | 어느 직급에 해당하는지 (선택) |
| 보고 직위 (실선) | 상위 보고 라인 |
| 점선 보고 직위 | dotted-line 매트릭스 보고 (선택) |
| 헤드카운트 여부 | 정식 정원 직위인지 |
| 충원 완료 여부 | 현재 충원되어 있는지 |

**조직도 시각화**: 회사별 조직도 화면에서 직위 노드들이 실선·점선 관계로 표시됩니다.

### 2.5 고용 형태 (EmploymentType)

| 형태 | 의미 |
|------|------|
| 정규직 | 무기계약 |
| 계약직 | 기간제 계약 (계약 종료일 추적) |
| 파견 | 파견업체 소속 직원 |
| 인턴 | 인턴십 |

**계약직 추적**:
- 계약 차수 (몇 번째 갱신인지)
- 계약 시작·종료일
- 자동 전환 예정일 (계약직 → 정규직 자동 전환 시점, 사규 또는 법정 기간 도래)
- 수습 시작·종료일
- 수습 상태 (진행 중 / 통과 / 실패 / 면제)

### 2.6 재직 상태 (EmployeeStatus)

| 상태 | 의미 |
|------|------|
| 정상 | 현재 정상 근무 중 |
| 휴직 중 | 휴직 모듈에서 활성화 (휴직 매뉴얼 §4.2 참조) |
| 사직 | 자발적 퇴사 |
| 면직 | 회사가 종료 (해고·계약만료) |

직원의 현재 상태는 가장 최근 활성 발령(`endDate=null`)의 `status` 필드에 저장됩니다.

## 3. 핵심 정책

### 3.1 Append-Only 발령 이력

이 시스템의 가장 중요한 원칙입니다.

- **모든 인사 변경은 기존 발령을 수정하지 않고 새 발령을 생성**합니다
- 기존 활성 발령은 `종료일 = 변경 시점 전날`로 닫고, 새 발령은 `시작일 = 변경 시점`으로 생성
- 같은 직원의 발령은 시간순으로 정렬되며, "현재 발령"은 `endDate = null`인 한 건
- 과거 발령은 절대 수정하지 않음 — 시점별 시계열이 보존됨

**효과**:
- "2025년 3월 1일 시점에 김부장은 어느 부서였나?" → 그날 활성이었던 발령 조회 → 답
- 발령 이력 = 직원 커리어 타임라인
- 잘못된 발령 수정 시: 새 발령(정정)을 생성 (수정 X, 추가 O)

### 3.2 발령 유형 8종

각 발령은 변경 유형(`changeType`) 한 가지를 가집니다.

| 유형 | 의미 | 사용 예 |
|------|------|--------|
| 입사 | 신규 입사 첫 발령 | 신입·경력 채용 |
| 부서이동 (TRANSFER) | 부서·직위 변경 (같은 법인 내) | 조직 개편, 인사이동 |
| 승진 | 직급 상향 | 연차별 승진, 특진 |
| 강등 | 직급 하향 | 징계 |
| 조직개편 (REORGANIZATION) | 부서 폐지·신설로 인한 자동 이관 | 본부 통폐합 |
| 상태 변경 (STATUS_CHANGE) | 재직 상태만 변경 (휴직·복직 등) | 휴직 시작·종료 (휴직 매뉴얼 §4.2) |
| 계약 변경 (CONTRACT_CHANGE) | 고용 형태·계약 변경 | 계약직 갱신, 정규직 전환 |
| 법인전환 (COMPANY_TRANSFER) | 법인 이동 | 한국→중국 파견, 자회사 이동 |

### 3.3 직원의 현재 정보 결정 방식

직원의 현재 부서·직급·직위·법인·재직상태는 **가장 최근 활성 발령** 한 건에서 결정됩니다.

**활성 발령 조건**:
- `endDate = null` (종료되지 않음)
- `isPrimary = true` (주직책 — 겸직의 경우 보조 직책은 별도)

**조회 패턴**: 모든 화면·API는 직원의 `assignments` 목록에서 `isPrimary && endDate=null` 한 건을 찾아 현재 정보를 표시합니다.

### 3.4 근속 통산 정책 (그룹사 내 법인 전환)

**원칙**: 동일 그룹 내 법인 이동 시 근속을 통산합니다.

- **법적 근거**: 대법원 판례 2006다24098 (동일 그룹 내 전적은 근속 통산 원칙)
- **시스템 적용**: 직원의 `최초 입사일`(`hireDate`)은 법인 전환 시에도 **변경하지 않음**
- **법인별 입사일이 필요한 경우**: 발령의 `effectiveDate`로 추적

**효과**:
- 퇴직금 산정: 최초 그룹 입사일 기준
- 연차 발생 기준: 최초 그룹 입사일 기준
- 승진 연한: 최초 그룹 입사일 기준

### 3.5 법인 전환 시 연차·급여 처리

법인이 바뀌면 노동법도 바뀌므로 다음 정책을 적용합니다.

**연차**:
- 출발 법인에서 **미사용 연차 수당 정산** (비례 계산)
- 도착 법인에서 **입사일 기준 새로 부여** (현지 노동법 적용)
- 근거: 한국↔해외는 물론 한국 내 법인 간에도 노동법·휴가 정책 상이

**급여**:
- 도착 법인의 **직급별 급여 밴드(salary band)에 맞춰 재조정**
- 출발 법인 급여를 참고하되, 도착 법인 밴드 범위 내에서 결정
- 근거: 국가·법인별 물가·시장 차이, 현지 형평성
- 한국 내 법인 이동도 동일 적용

**근속**: §3.4 통산 정책 그대로 적용 (최초 입사일 유지)

> 급여 처리는 [payroll.md §7.4 법인 전환자](payroll.md) 참조

### 3.6 수습 기간

신규 채용 시 수습 기간을 운영합니다.

- **기본 기간**: 3개월 (회사별 사규에 따라 조정 가능)
- **수습 종료 시점**: 시스템이 7일 전 인사담당자·직속 팀장에게 알림
- **수습 전환 평가**: 팀장이 평가 → 본부장 결재 (결재 매뉴얼 §2.1 수습 전환)
- **수습 상태**: 진행 중 / 통과 / 실패 / 면제
- **추후 개선** 항목: 수습 종료 자동 알림 + 결재 흐름 (§9 #3 참조)

### 3.7 계약직 자동 전환

계약직 직원의 정규직 자동 전환을 추적합니다.

- **자동 전환 예정일**: 법정 한도(한국 2년) 이내에서 회사별 사규로 단축 가능. 법정 한도 초과 운영은 정규직 의제로 자동 전환됨
- **시점 도래 시 알림**: 자동 알림으로 인사담당자에게 전환 검토 요청
- **결재 흐름**: 팀장 평가 → 본부장 결재 (결재 매뉴얼 §2.1 계약직 전환)
- **추후 개선** 항목: 자동 전환 시점 도래 시 자동 결재 흐름 진입 (§9 #4 참조)

### 3.8 직위 보고 라인 (실선·점선)

직위 간 보고 관계는 두 가지로 구분됩니다.

- **실선 (Solid Line)**: 기본 보고 라인. 직원의 직속 팀장은 본인 직위의 상위 실선 직위 보유자
- **점선 (Dotted Line)**: 매트릭스 조직의 부가 보고 라인 (예: 프로젝트 매니저)
- **결재의 직속 팀장 역할**: 실선 우선, 보조 직책의 매니저도 결재 권한 인정 (결재 매뉴얼 §11.5 참조)

## 4. 워크플로

### 4.1 직원 생애주기 (Employee Lifecycle)

> 신규 입사: 채용 매뉴얼 [recruitment.md §3.9 입사 전환](recruitment.md) 참조

```
[채용] → [입사 발령] → [수습 중] → [수습 종료] → [정규직 전환] → ...
   │           │             │           │
   │           ↓             ↓           ↓
   │       Assignment    상태 추적   결재 매뉴얼
   │      (HIRE)        (수습 평가)  §2.1 수습 전환
   │
   │   [재직 중 변경]
   │       ├── 부서이동 (TRANSFER 발령)
   │       ├── 승진 (PROMOTION 발령)
   │       ├── 계약 갱신 (CONTRACT_CHANGE 발령)
   │       ├── 법인 이동 (COMPANY_TRANSFER 발령)
   │       ├── 휴직 시작 (STATUS_CHANGE 발령, 휴직 매뉴얼)
   │       └── 휴직 종료 (STATUS_CHANGE 발령, 휴직 매뉴얼)
   │
   ↓
[퇴직 결정] → [퇴직 발령] → [정산] → [퇴직 완료]
                  │              │
                  ↓              ↓
              Assignment    오프보딩 모듈
            (RESIGN/        (별도 매뉴얼)
            TERMINATE)
```

### 4.2 발령 생성의 표준 패턴

새 발령을 생성할 때 시스템은 다음을 자동으로 처리합니다.

```
1. 기존 활성 발령 조회 (endDate=null && isPrimary=true)
2. 기존 발령의 endDate = 신규 발령 시작일 - 1일
3. 새 발령 생성 (effectiveDate, changeType, 변경된 필드 + 미입력 필드는 기존값 carry forward)
4. EmployeeHistory에 변경 이력 기록 (감사 로그)
```

**Carry Forward 규칙** (대량 인사이동 시):

| 발령 유형 | 미입력 시 현재값 유지 필드 |
|----------|---------------------------|
| 부서이동 | 직급, 직군, 직위, 고용형태, 근무지 |
| 승진 | 부서, 직군, 고용형태, 근무지 |
| 법인전환 | 직군 (나머지는 새 법인 기준 필수) |
| 퇴직 | 없음 (기존 발령 종료만) |

### 4.3 대량 인사이동 워크플로 (3단계 마법사)

CSV 파일로 다수 직원의 발령을 일괄 처리하는 기능입니다.

**3단계**:

```
Step 1: 유형 선택 (5종 중 1개)
   ├── 부서이동
   ├── 승진
   ├── 법인전환 (전사관리자 전용)
   ├── 퇴직
   └── 급여변경
   
Step 2: CSV 업로드 + 검증 + 프리뷰
   ├── 파일 형식 검증
   ├── 행별 필수 필드 검증
   ├── 참조 데이터 검증 (사번·부서코드·직급코드 등)
   ├── 비즈니스 룰 검증 (직급 비교, 발효일 역전 등)
   └── 프리뷰 테이블 (각 행의 변경 전·후 표시)
   
Step 3: 실행 확인
   ├── 트랜잭션 내 일괄 실행 (All-or-Nothing)
   ├── 1건 실패 → 전체 롤백
   ├── 트랜잭션 직전 재검증 (사이에 다른 관리자가 직원 상태 변경했을 수 있음)
   └── 실행 감사 로그 기록
```

**5가지 이동 유형의 정책**:

| 유형 | 처리 |
|------|------|
| 부서이동 | 새 발령(TRANSFER) 생성, 미입력 필드 carry forward |
| 승진 | 새 발령(PROMOTION) 생성, 직급이 현재보다 높은지 경고만 표시 |
| 법인전환 | 새 발령(COMPANY_TRANSFER) 생성, 부서·직위는 새 법인 기준 필수, 연차·급여 정책(§3.5) 적용 |
| 퇴직 | 기존 발령 종료, 새 발령(RESIGN/TERMINATE) 생성, 오프보딩 자동 시작 |
| 급여변경 | 발령 변경 없음, 급여 이력만 기록 |

**날짜 처리 규칙**:
- 발효일 = 변경이 적용되는 날
- 기존 발령 종료일 = 발효일 - 1일
- 퇴직의 경우: 기존 발령 종료일 = 마지막 근무일, 퇴직 발령 시작일 = 마지막 근무일 + 1일

### 4.4 자동 알림

- **계약 자동 전환 예정일 D-30**: 인사담당자에게 전환 검토 알림
- **수습 종료일 D-7**: 직속 팀장·인사담당자에게 평가 요청 알림
- **법인전환 시작 D-3**: 직원·인사담당자에게 절차 안내 알림
- **추후 개선** 항목: 계약 갱신·수습 평가 결재 흐름 자동 시작 (§9 #3·#4)

## 7. 특수 상황 처리

### 7.1 겸직 (Multiple Concurrent Assignments)

직원이 동시에 여러 직책을 보유하는 경우:

- 주직책(`isPrimary=true`) 1건 + 보조 직책 여러 건
- 휴가·결재 라우팅·급여는 **주직책 기준으로만** 처리
- 보조 직책의 부서·매니저는 결재 매뉴얼 §3.3에서 보조 매니저로 인정
- 직원 정보 표시 시: 주직책의 부서·직위가 기본 표시

### 7.2 점선 보고 (Dotted Line) — 매트릭스 조직

- 직위에 실선 상사 + 점선 상사 동시 지정 가능
- 결재 시 실선·점선 매니저 모두 결재 권한 보유 (결재 매뉴얼 §11.5)
- 조직도에서는 점선이 다른 색으로 시각화

### 7.3 부서 폐지·신설 시 직원 이관

- 부서를 폐지·합병할 때 해당 부서에 있던 직원들의 발령을 자동 생성
- 발령 유형: 조직개편 (REORGANIZATION)
- 새 부서로의 이관은 인사담당자가 일괄 수동 처리 또는 CSV 업로드로 처리

### 7.4 직급·호칭 체계 변경

- 직급 체계 자체를 바꾸는 경우 (예: L1/L2 → L1/L2/L3/L4/L5 확장)
- 새 직급 추가 → 기존 직원의 직급은 매핑 규칙대로 일괄 변환
- 호칭 매핑도 함께 업데이트
- **현재 상태**: 시스템은 4단계(L1/L2/전문리더/경영리더), 향후 L1~L5로 리팩터링 예정

### 7.5 시간 역행 발령 (Backdated Assignment)

원칙적으로 불가능하지만 정정 목적으로 필요한 경우:

- 새 발령의 `effectiveDate`가 직원의 현재 활성 발령보다 과거인 경우
- 시스템이 차단 (대량 인사이동의 발효일 역전 검증)
- 우회 방법: 인사담당자가 단건 발령 등록 시 별도 정정 절차 필요 (감사 대상)
- **권장**: 정정은 새 시점에 새 발령으로 처리, 사유에 정정 내용 기록

### 7.6 휴직 중 발령

- 휴직 활성 상태 동안 다른 발령(부서이동·승진 등) 일반적으로 불가
- 예외: 부서 폐지·법인 이관 등 불가피한 경우 인사담당자가 수동 처리
- 휴직 자체의 시작·종료 발령은 휴직 매뉴얼 §4.2 참조

### 7.7 법인전환 직원의 진행 중 결재

- 법인 이동 시점에 다른 모듈의 진행 중 결재가 있는 경우
- 원칙: 출발 법인에서 결재 완료 후 이동
- 불가피한 경우: 인사담당자가 결재 우회 또는 이관 처리

### 7.8 퇴사 후 재입사

- 퇴사한 직원이 다시 입사하는 경우 시스템은 새 직원으로 등록 또는 기존 직원 재활용 둘 다 가능
- **권장**: 기존 직원 재사용 (사번·이력 보존) — 새 발령(HIRE) 추가
- 근속 통산: 회사 사규에 따라 재입사 시점 또는 최초 입사 기준 (인사담당자 판단)
- **추후 개선** 항목: 재입사 시 근속 통산 정책 시스템화 (§9 #7 참조)

### 7.9 사번 변경

- 기본적으로 사번(`employeeNo`)은 변경하지 않음
- 사번 체계 전면 개편 등 예외 상황: 전사관리자만 수정 가능, 모든 외부 시스템 연동(급여·SSO·이메일 등) 영향 확인 필요

## 8. 자주 묻는 질문

**Q. 직원의 부서·직급을 바꾸려면 어떻게 하나요?**
A. 직원 상세 → 발령 이력 탭에서 새 발령을 생성하거나, 대량 인사이동 화면에서 CSV로 일괄 처리합니다. 기존 정보를 직접 수정하지 않고 항상 새 발령으로 추가됩니다 (Append-Only 원칙).

**Q. 발령 이력이 너무 많아 보기 어려워요. 정리할 수 있나요?**
A. 발령 이력은 감사 목적으로 보존되며 삭제할 수 없습니다. 화면에서는 carry forward로 묶여 시간순으로 표시되며, 필터로 특정 유형(승진만, 법인전환만 등)만 볼 수 있습니다.

**Q. 직급과 호칭의 차이가 뭔가요?**
A. 직급은 보상·승진의 기준(L1, L2 등)이고, 호칭은 명함·이메일에 사용되는 이름(매니저, 책임매니저 등)입니다. 같은 직급도 국가별로 호칭이 다를 수 있어 분리 관리합니다.

**Q. 본부장은 어떤 시스템 역할인가요?**
A. 현재 시스템 역할 체계(일반직원·팀장·인사담당자·임원·전사관리자)에는 본부장에 정확히 대응하는 역할이 없습니다. 운영상은 해당 부서의 부서장(`headEmployeeId`)으로 등록하여 결재 흐름의 "부서장" 역할로 동작시킵니다 (결재 매뉴얼 §9 #5 참조).

**Q. 그룹 내 법인을 이동하면 근속이 초기화되나요?**
A. 아닙니다. 대법원 판례에 따라 동일 그룹 내 법인 전환 시 근속은 통산됩니다. 최초 그룹 입사일이 유지되며 퇴직금·연차 발생·승진 연한 모두 최초 입사일 기준입니다.

**Q. 법인 이동 시 연차는 어떻게 되나요?**
A. 출발 법인에서 미사용 연차 수당으로 정산받고, 도착 법인에서 입사일 기준으로 새로 부여됩니다. 법인별 노동법이 달라 연차 정책이 다르기 때문입니다 (§3.5 참조).

**Q. 법인 이동 시 급여는 어떻게 되나요?**
A. 도착 법인의 직급별 급여 밴드(salary band)에 맞춰 재조정됩니다. 출발 법인 급여를 참고하지만 도착 법인 밴드 범위 내에서 결정됩니다. 국가·법인별 물가 차이 때문입니다 (§3.5).

**Q. 대량 인사이동 중 1건만 에러가 나면 어떻게 되나요?**
A. 전체가 롤백됩니다 (All-or-Nothing). 1건이라도 검증·실행 단계에서 실패하면 모든 변경이 취소됩니다. 안전성을 위한 정책이며 프리뷰 단계에서 미리 모든 에러를 확인할 수 있습니다.

**Q. 법인전환 CSV를 인사담당자가 업로드할 수 있나요?**
A. 불가능합니다. 법인전환은 출발 법인 + 도착 법인 양쪽에 쓰기 권한이 필요하므로 전사관리자만 실행할 수 있습니다. 인사담당자는 본인 법인 권한만 보유하므로 화면에서 차단됩니다.

**Q. 직원 본인이 바꿀 수 있는 정보는 무엇인가요?**
A. 비상연락처, 사진, 본인 연락처 등 인적사항 일부만 수정 가능합니다. 부서·직급·고용형태·재직상태 등 인사 발령에 관련된 정보는 인사담당자만 변경할 수 있습니다.

**Q. 퇴사 후 다시 입사하면 사번이 어떻게 되나요?**
A. 기존 직원 정보를 재사용하여 새 발령(입사)을 추가하는 방식을 권장합니다. 사번과 이력이 보존됩니다. 단, 근속 통산 여부(재입사 시점 또는 최초 입사 기준)는 회사 사규에 따라 인사담당자가 판단합니다.

**Q. 수습 직원의 평가가 결재 흐름에 자동으로 들어가나요?**
A. 현재 시스템은 수습 종료 7일 전 알림은 발송하지만, 평가 결과를 결재 흐름에 자동 진입시키지는 않습니다. 인사담당자가 수동으로 결재 신청해야 합니다. **추후 개선** 항목입니다 (§9 #3).

## 10. 추후 개선

**기능 개발 트랙 (인사관리규정 정합성, A2 Gap)**:
1. 직급 체계 L1~L5 + 임원 + 호칭 확장 (§9 #1, ADR A2 Gap #1)

8. 본부장 시스템 역할 매핑 정립 (§9 #5, 결재 매뉴얼 §9 #5와 통합)

**워크플로**:
9. 재입사 시 근속 통산 정책 시스템화 (§9 #7)

11. 시간 역행 발령 정정 절차 (§9 #11)
12. 발령 자동 알림 정책 정립 (§9 #12)

**대량 인사이동 (Phase 4)**:

### 11.5 알려진 위험 코드 경로

- `EmployeeAssignment` Append-only 원칙 — `UPDATE`로 기존 발령 수정 시 시점별 이력 깨짐. 모든 변경은 새 row 생성. 라이브 코드 리뷰에서 강제
- `extractPrimaryAssignment` SSOT — 직원의 현재 정보 조회 시 `assignments[0]` 직접 접근 금지, 헬퍼만 사용 (`.claude/rules/assignments.md` 강제)
- `resolveApproverByRole.direct_manager` — primary + secondary assignment 비대칭 정책 (결재 매뉴얼 §11.5 정합)
- 대량 인사이동 트랜잭션 — `prisma.$transaction()` 내에서 모든 row 처리, 실패 시 전체 롤백. 트랜잭션 격리 수준 확인 필요
- 대량 인사이동의 발효일 역전 검증 (Gemini Patch 2) — `validate` 통과 후 `execute` 시점에 직원 상태가 바뀌었을 수 있음 → 트랜잭션 내 재검증 필수 (Patch 3)
- 퇴직 발령 날짜 처리 (Gemini Patch 4) — 기존 Assignment endDate = 마지막근무일, 새 발령 effectiveDate = 마지막근무일 + 1일. `addDays(parseDateOnly(...), 1)` 공식 강제
- 법인전환 권한 (Gemini Patch 1) — `ENTITY_TRANSFER`는 SUPER_ADMIN 전용, HR_ADMIN 차단 (UI + API 양쪽). 타법인 쓰기 권한 침범 방지
- `hireDate` 불변성 — 법인 전환 시 절대 변경 금지 (근속 통산 정책). DB constraint은 없으므로 코드 레벨에서 강제 필요
