> **최종 갱신**: 2026-05-12
> **담당**: 인사관리팀
> **상태**: Draft v1 (CEO 리뷰 전)

## 1. 개요

### 1.1 무엇을 다루는가

CTR HR Hub의 온보딩·오프보딩 모듈은 다음을 통합 관리합니다.

- **온보딩** — 신규 입사·전보·재입사 직원의 시점별(Day 1/7/30/90) 체크리스트 진행
- **오프보딩** — 5종 퇴직 유형(자발·비자발·정년·계약만료·상호합의)별 체크리스트 + 인수인계 + 자산 회수
- **크로스보딩** — 법인 전환 시 출발 법인·도착 법인 양쪽 동시 진행하는 특수 워크플로
- **체크리스트 템플릿** — 회사·시나리오별 표준 작업 정의 (10개 카테고리)
- **버디 배정 (Buddy)** — 신규 입사자를 안내할 멘토 직원 매칭
- **온보딩 체크인 (Check-in)** — Day 1/7/30/90 마일스톤별 직원 만족도·기분 측정
- **퇴직 면담 (Exit Interview)** — 퇴직 사유 6종 분류 + 만족도 + AI 요약
- **자산 회수 (Asset Return)** — 노트북·휴대폰 등 자산 회수 추적 + 미반환 시 임금 차감
- **M365 계정 프로비저닝** — 마이크로소프트 계정 생성·라이선스 회수 (**현재 Mock 구현** `src/lib/m365-account.ts`, 실제 Azure AD Graph API 연동은 별도 트랙)
- **재입사 차단 (Do Not Rehire)** — 비자발 퇴직자 재입사 차단 플래그

### 1.2 도입 의도

- **첫 90일 경험 표준화** — 모든 신규 입사자가 동일 수준의 안내를 받도록 체크리스트 강제
- **퇴직 절차 누락 방지** — 자산 회수·계정 정지·인수인계·정산 등 누락 가능성 높은 항목을 시스템 게이트로 강제
- **인수인계 추적** — 퇴직자가 누구에게 무엇을 인수인계했는지 명문화
- **법인 전환 충돌 방지** — 출발 법인 오프보딩과 도착 법인 온보딩을 한 인스턴스로 연결하여 데이터 일관성 보장
- **퇴직 인사이트** — 면담 데이터 + AI 요약으로 이직 사유 패턴 분석
- **법규 준수** — 자산 회수 미반환 시 임금 차감은 본인 동의 + 사유 기록 (한국 근로기준법 §43 임금 전액 지급 원칙 + 대법원 판례에 따른 상계 합의 법리)

### 1.3 채용·인사관리·급여 매뉴얼과의 관계

| 항목 | 본 매뉴얼 (온보딩·오프보딩) | 관련 매뉴얼 |
|------|-------------------------|-----------|
| 입사 발령 생성 | 본 매뉴얼이 트리거 후 생성 | 인사관리 매뉴얼 §4.1 HIRE 발령 (채용 매뉴얼 §3.9 입사 전환에서 생성) |
| 신규 입사 트리거 | 채용 매뉴얼이 트리거 | 채용 매뉴얼 §3.9 입사 전환 |
| 퇴직 발령 생성 | 본 매뉴얼이 시작 → 인사관리 발령 트리거 | 인사관리 매뉴얼 §4.1 RESIGN/TERMINATE 발령 |
| 퇴직금 정산 | 본 매뉴얼이 트리거 | 급여 매뉴얼 §3.13 퇴직금 |
| 자산 회수 시 임금 차감 | 본 매뉴얼이 계산·기록 | 급여 매뉴얼 §3.7 수동 조정 (환수) |
| 법인 전환 | 본 매뉴얼이 양쪽 동시 진행 | 인사관리 매뉴얼 §3.5 COMPANY_TRANSFER + 휴가·보상 정책 cross-ref |
| 평가·휴가 잔액 정산 | 본 매뉴얼이 트리거 | 휴가 매뉴얼 §7.2 / 성과 매뉴얼 §7.1 |

## 3. 핵심 정책

### 3.1 온보딩 자동 시작

채용 매뉴얼 §3.9 입사 전환 완료 시 시스템이 자동으로 온보딩 시작:

1. **OnboardingTemplate 선택**: 회사·시나리오(NEW_HIRE/TRANSFER/REHIRE) 매칭 템플릿
2. **EmployeeOnboarding 인스턴스 생성** + 모든 작업을 EmployeeOnboardingTask로 복제
3. **마감일 계산**: 각 작업의 `dueDaysAfter` × 입사일 기준
4. **담당자 자동 배정**: assigneeType 기반 (HR/MANAGER/IT 등 자동 매핑)
5. **버디 추천**: 같은 부서 + 비슷한 직급 직원 추천 (인사담당자 확정)

### 3.2 작업 진행 + 게이트

직원과 담당자가 작업을 단계별 수행:

- **상태**: 대기 → 진행 중 → 완료 (또는 차단·건너뛰기)
- **차단 (BLOCKED)**: 다른 작업·외부 사유로 진행 불가 시 사유 입력
- **건너뛰기 (SKIPPED)**: 해당 작업이 본인에게 해당 안 됨 (사유 필수)
- **사인오프 (Sign-off)**: 특정 작업에 인사담당자 최종 확인 필수 (예: 근로계약서)

### 3.3 마일스톤 체크인

Day 1/7/30/90 마일스톤은 **대시보드에 표시** (`milestone-helpers.ts`에서 도래 시점 계산). **자동 알림 발송 cron은 미등록** — `nudge-batch`에 룰 추가 필요. 현재는 인사담당자·매니저가 대시보드에서 확인 후 수동 알림. 체크인 응답 항목:

| 항목 | 측정 |
|------|------|
| 기분 (Mood) | 매우 좋음 / 좋음 / 보통 / 힘듦 / 나쁨 (5단계) |
| 에너지 | 1~10 |
| 소속감 | 1~10 |
| 자유 코멘트 | 텍스트 |

응답·우려 표시 분석: **현재 미구현** — 응답 텍스트는 시스템에 보존되며 인사담당자·매니저가 직접 검토. AI 자동 요약·패턴 분석은 §10 추후 개선.

### 3.4 버디 (Buddy) 시스템

- 매니저·인사담당자가 버디 직원 1명 배정
- 버디 역할: 비공식 멘토, 회사 적응 도움
- 시스템: 버디용 작업(BUDDY assignee) 자동 생성 (점심 식사·1주차 커피챗 등)
- 버디 부재·휴직 시 대체 버디 수동 배정

### 3.5 오프보딩 자동 시작

인사관리 매뉴얼 §4.1 RESIGN/TERMINATE 발령 등록 시 자동:

1. **OffboardingChecklist 선택**: 회사·퇴직 유형 매칭
2. **EmployeeOffboarding 인스턴스 생성** + 작업 복제
3. **마지막 근무일 (lastWorkingDate) 기준 마감일 계산**:
   - 인수인계: 마지막 근무일 전 1~2주
   - 자산 회수: 마지막 근무일 +1일 이내
   - 면담: 마지막 근무일 7일 전~당일
4. **인수인계 대상자 지정** (handoverToId): 후임자 또는 매니저
5. **자동 게이트 발동**:
   - 자산 회수 안 끝나면 퇴직 완료 불가
   - 인수인계 안 끝나면 경고
   - 면담 안 끝나면 자동 표시

### 3.6 인수인계 (Handover)

- **handoverToId**: 인수인계 대상자 (후임자 또는 매니저)
- 인수인계 작업은 HANDOVER 카테고리로 별도 표시
- 인수인계 문서: 오프보딩 문서(OffboardingDocument)로 첨부
- 완료 게이트: 매니저·인사담당자가 인수인계 완료 사인오프

### 3.7 자산 회수 (Asset Return)

직원이 회사 지급 자산(노트북·휴대폰·법인카드 등)을 반납.

**상태 흐름**:
```
대기 (PENDING) ──→ 회수 완료 (RETURNED) ──→ 종료
                       │
                       └─→ 미반환 (UNRETURNED)
                              ├─→ 임금 차감 (DEDUCTED) ── 직원 동의 + 잔존가치 계산
                              └─→ 민사 청구 (CIVIL_CLAIM) ── 차감 불가 시
```

**잔존가치 계산** (예시):
```
잔존가치 = 구매가 × (1 - 사용월수/내용연수×12) × 사용 가능 비율
```

**임금 차감 조건** (한국 근로기준법 §43 + 대법원 판례 상계 합의 법리):

한국 근로기준법 §43은 임금 전액 지급 원칙을 정합니다 (본문 예외는 법령·단체협약 두 가지뿐). 대법원 판례(2001다25184 등)는 근로자의 자유로운 의사에 기한 명확한 서면 동의가 있고 객관적 합리성이 인정되는 경우에 한해 임금 상계를 제한적으로 허용합니다. 따라서 시스템은 다음을 모두 보관해야 합니다:

- (1) 잔존가치 산정 근거 (구매가·내용연수·사용 기간 등 객관적 산식)
- (2) 본인 자필 서면 동의서 (consentDocExists) — 자유로운 의사 입증
- (3) 차감 금액·사유 명문화
- 마지막 급여에서 차감 (급여 차감 실행은 [payroll.md §3.7](payroll.md) 참조)

**차감 불가 시**: 민사 청구로 전환 (법무 협조 필요)

### 3.8 퇴직 면담 (Exit Interview)

- 마지막 근무일 7일 전~당일에 인사담당자·HR 매니저가 면담
- **점수**: 종합 만족도(1~5) + 4개 영역별 점수(보상·문화·매니지먼트·성장)
- **사유**: 6종 분류 + 상세 텍스트 + 개선 제안
- **재추천 여부**: "다시 다닐 의향" yes/no
- **익명성**: `isConfidential=true` 시 매니저·팀에 비공개, 인사·임원만 조회
- **AI 요약**: **현재 미구현** — 면담 텍스트는 시스템에 보존되며 인사담당자가 직접 검토. AI 자동 요약·핵심 사유 추출은 §10 추후 개선.

### 3.9 재입사 차단 (Do Not Rehire)

비자발 퇴직자 또는 심각한 사유 직원의 재입사 차단.

- `isDoNotRehire=true` + `doNotRehireReason` (사유) + `doNotRehireSetById` (설정자)
- 채용 모듈에서 지원자 이메일 매칭 시 자동 경고
- 인사담당자만 설정·해제 가능 (감사 대상)

### 3.10 M365 계정 프로비저닝

마이크로소프트 365 계정 관리. **현재 Mock 구현** (`src/lib/m365-account.ts`). 실제 Azure AD Graph API 연동은 별도 트랙. HR이 IT 담당자에게 수동 요청 후 결과를 시스템에 입력해야 실제 운영 가능.

**입사 시 (PROVISION)** — 정책 정의 (Mock 호출, 실제 처리는 IT 수동):
- 회사 도메인 이메일 생성
- 라이선스 할당 (E1/E3/E5 등 회사 정책)
- 기본 그룹·OU 배정

**퇴직 시 (DEPROVISION)** — 정책 정의 (Mock 호출, 실제 처리는 IT 수동):
- 로그인 비활성화 (마지막 근무일 +1일)
- 라이선스 회수
- 메일박스 공유 메일박스로 전환 (선택)

**상태**: 대기 / 진행 중 / 완료 / 실패

> **데이터 흐름 주의**: 현재 Mock 단계에서는 시스템이 작업 로그만 기록하고 실제 계정은 IT 담당자가 수동 처리합니다. 신규 입사·퇴직 시 HR이 별도 요청 → IT 처리 → 결과 시스템 입력 흐름. Azure AD Graph API 실제 연동은 §10 추후 개선.

### 5.2 결재 라우팅

온보딩·오프보딩 모듈은 일반적으로 결재 흐름이 없으며, 트리거 결재는 채용·인사관리 모듈에서 이미 완료된 상태로 진입합니다.

| 작업 | 결재 |
|------|------|
| 온보딩 시작 | 채용 매뉴얼 §3.9 입사 전환에서 이미 결재 완료 |
| 오프보딩 시작 | 인사관리 매뉴얼 §4.1 퇴직 발령 결재(본부장→대표) 완료 |
| 자산 임금 차감 | 직원 서면 동의 (시스템 내 결재 흐름 없음, 동의 문서 첨부) |
| 재입사 차단 설정 | 인사담당자 직접 (감사 이력만) |

**시스템 결재 흐름 (현재 시드)** — 자발/비자발 단일 흐름:
- 시드: `[본부장(dept_head), 대표(ceo)]` 2단계 (`prisma/seeds/42-approval-flow-defaults.ts:64-69`)
- 자발(VOLUNTARY) / 계약만료(CONTRACT_END) / 정년(RETIREMENT): 시스템 결재로 충분
- **비자발(INVOLUNTARY)**: 시스템 결재는 동일하나 **한국 근로기준법 별도 요건 적용 필수**:
  - §23 정당한 사유 (해고 사유 명확화)
  - §24 정리해고 요건 (긴박한 경영상 필요, 해고 회피 노력 등)
  - §26 30일 전 해고 예고
  - §27 서면통지 (해고 사유·시기)

**현재 시스템 한계**:
- 비자발 케이스 별도 결재 흐름 없음 (시스템은 자발과 동일하게 2단계 결재만)
- §23/§24/§26/§27 요건 시스템 자동 검증·기록 없음
- 정리해고 사유 입력 필드, 서면통지서 자동 생성 미구현

**운영 권장**: 비자발 케이스는 시스템 결재 외에 노무사 검토 + 서면통지서 별도 작성 + 30일 전 통지 별도 관리.

상세는 [employee.md §5.2](employee.md) + 한국 근기법 §23/§24/§26/§27 참조.

### 5.3 데이터 가시성

- **직원**: 본인 온보딩·오프보딩만 조회
- **버디**: 배정된 신규 입사자의 일부 정보 (이름·부서·작업 진척)

## 7. 특수 상황 처리

### 7.1 매니저·버디 부재 시

- 매니저 휴가·휴직 시 매니저 작업은 대체 매니저에게 임시 재배정 (인사담당자 수동)
- 버디 휴가·휴직 시 대체 버디 추천 (인사담당자 결정)

### 7.2 작업 차단 (BLOCKED)

- 다른 작업·외부 사유로 진행 불가 (예: 노트북 재고 없음 → SETUP 차단)
- 사유·차단 시점·해제 시점 기록
- 차단 장기화 시 인사담당자 알림

### 7.3 입사 취소 (Onboarding Cancel)

- 신규 입사자가 입사 직전·직후 취소
- 인사담당자가 온보딩 인스턴스 ARCHIVED 처리
- 직원 마스터·발령 처리는 채용 매뉴얼 §7.10 참조

### 7.4 마지막 근무일 변경

- 퇴직 협의 도중 마지막 근무일 변경 (연장·단축)
- 인사담당자가 `lastWorkingDate` 수정 → 모든 작업 마감일 자동 재계산
- 큰 변경 시 직원·인수인계 대상에게 알림

### 7.5 자산 미반환 (UNRETURNED)

- 마지막 근무일 후 자산 미반환
- 인사담당자가 직원에게 반환 요청
- 일정 기간 후에도 미반환:
  - **직원 동의 시**: 잔존가치 계산 → 임금 차감 (마지막 급여에서)
  - **동의 거부 시**: 민사 청구로 전환 (법무팀 협조)

### 7.6 퇴직 면담 거부

- 직원이 면담 거부 권리 있음 (강제 불가)
- `isExitInterviewCompleted=false` 유지 + 사유 기록
- 거부해도 오프보딩 완료 가능 (단, 사유 메모 필수)

### 7.7 비밀유지 (isConfidential) 면담

- 직원이 매니저 비공개 요청 시 isConfidential=true
- 매니저·팀에 면담 내용 비공개
- 인사·임원·법무만 조회
- 통계 분석은 익명화 후 매니저 제공 가능

### 7.8 크로스보딩 도중 한쪽 실패

- 출발 법인 인수인계 지연 → 도착 법인 도착 지연
- 시스템이 양쪽 연결(linkedPlanId)로 진척 동기화
- 한쪽 실패 시 인사담당자가 수동 조율

### 7.9 재입사자 (REHIRE) 처리

- 과거 퇴직자가 재입사
- 시스템이 DNR(do-not-rehire) 자동 검증 → DNR 시 채용 차단
- DNR 아니면 REHIRE 시나리오 온보딩 시작 (일부 작업 면제 가능 — 회사 정책)

### 7.10 정년 퇴직 자동 알림

- 정년 도래 직원 자동 식별 (Employee.birthDate + 회사 정년 정책)
- 자동 알림은 **현재 미구현** — 인사관리 매뉴얼 §9 #9 추후 개선 항목

### 7.11 M365 프로비저닝 실패

- Azure AD API 일시 장애로 PENDING 또는 FAILED
- IT 담당자 수동 처리 (재시도 또는 수동 계정 생성·삭제)
- 현재 자동 긴급 알림 cron 없음 — 대시보드에서 진행률·미완료 항목 확인 + 인사담당자 수동 follow-up. **추후 개선** §10 참조

### 7.12 비자발 퇴직의 DNR 자동 설정

- 비자발(INVOLUNTARY) 퇴직 시 isDoNotRehire 기본값 검토 권고
- 인사담당자가 수동 결정 (자동 설정 안 함 — 회사 정책)

## 8. 자주 묻는 질문

**Q. 입사 첫날 뭘 해야 하나요?**
A. `내 정보 → 온보딩`에서 Day 1 체크리스트를 확인하세요. 사원증·노트북 수령, M365 계정 활성화, 환영 점심·오리엔테이션 등이 표시됩니다. 작업마다 담당자(HR·IT·매니저·버디)가 다르며 본인이 직접 해야 하는 항목은 별도 표시됩니다.

**Q. 마일스톤 체크인이 뭔가요?**
A. Day 1/7/30/90 마일스톤이 도래하면 대시보드에 만족도·기분·에너지·소속감 응답 폼이 표시됩니다. 현재 자동 알림 발송 cron은 미등록(§9)이라 인사담당자·매니저가 안내해 응답을 유도합니다. 답변은 매니저·인사가 모아 보지만 익명이며, AI 요약 패턴 분석은 §10 추후 개선입니다.

**Q. 버디는 누구인가요?**
A. 신규 입사 시 매니저·인사가 배정하는 멘토 직원입니다. 같은 부서의 비슷한 직급 직원이 추천되며, 비공식적으로 회사 적응을 돕습니다. 점심 함께 먹기·1주차 커피챗 등이 버디 작업입니다.

**Q. 퇴사할 때 뭘 해야 하나요?**
A. 인사담당자가 퇴직 발령 등록 후 자동으로 오프보딩 체크리스트가 생성됩니다. `내 정보 → 오프보딩`에서 인수인계·자산 반납·면담 일정 등을 확인할 수 있습니다.

**Q. 인수인계를 누구한테 하나요?**
A. 매니저나 인사담당자가 후임자(또는 임시 인수인계 대상)를 지정합니다. 시스템에서 대상자를 확인할 수 있으며, 인수인계 미팅·문서를 통해 진행합니다.

**Q. 자산을 반납 안 하면 어떻게 되나요?**
A. 일정 기간 후에도 미반환 시 직원 동의를 받아 잔존가치만큼 마지막 급여에서 차감합니다. 한국 근로기준법 §43은 임금 전액 지급 원칙을 정하며, 대법원 판례는 근로자의 자유로운 의사에 기한 명확한 서면 동의(상계 합의)가 있고 객관적 합리성이 인정될 때에 한해 임금 상계를 제한적으로 허용합니다. 따라서 본인 자필 서면 동의서와 잔존가치 산정 근거가 모두 보관되어야 합니다. 동의를 거부하면 민사 청구로 전환됩니다.

**Q. 퇴직 면담은 꼭 해야 하나요?**
A. 강제는 아닙니다. 거부 권리가 있으며, 거부해도 오프보딩은 완료 가능합니다. 다만 회사가 직원 경험·이직 패턴을 개선하기 위해 권장합니다.

**Q. 매니저가 내 퇴직 면담을 볼 수 있나요?**
A. 비공개(`isConfidential=true`)로 표시한 면담은 매니저·팀에 비공개입니다. 인사·임원만 조회하며, 매니저 피드백을 자유롭게 작성할 수 있도록 보호됩니다.

**Q. M365 계정은 언제 만들어지나요?**
A. 현재 M365 프로비저닝은 Mock 구현(`src/lib/m365-account.ts`) 단계입니다. 입사 발령 등록 시 시스템에 작업 로그가 기록되며, 실제 계정 생성·라이선스 할당은 HR이 IT 담당자에게 요청한 뒤 IT가 수동 처리하고 결과를 시스템에 입력해야 합니다. Azure AD Graph API 실제 연동은 §10 추후 개선입니다.

**Q. 법인 전환할 때 양쪽 다 작업해야 하나요?**
A. 네. 출발 법인에서 인수인계·자산·정산 작업과 도착 법인에서 신규 입사 작업이 동시 진행됩니다. 시스템이 양쪽을 연결하여 추적합니다.

**Q. 입사 취소되면 다시 입사할 수 있나요?**
A. 입사 취소 자체로는 차단되지 않습니다. 다만 비자발 퇴직 등으로 인사담당자가 "재입사 차단(Do Not Rehire)" 설정한 경우 새 지원 시점에서 채용 시스템이 자동 경고합니다.

**Q. 정년이 다가오면 자동으로 알려주나요?**
A. 현재 시스템은 정년 자동 알림을 제공하지 않습니다 (인사관리 매뉴얼 §9 #9 추후 개선). 인사담당자가 직원 생년월일·회사 정년 정책으로 별도 확인합니다.

## 11. 부록: 기술 메모

> 본문이 정책 설명, 이 섹션은 개발/QA 참고용. HR은 건너뛰어도 무방합니다.

### 11.1 데이터 모델

핵심 모델 (모두 `prisma/schema.prisma`):

- `OnboardingTemplate` — 회사·시나리오별 마스터 템플릿 (NEW_HIRE/TRANSFER/REHIRE × ONBOARDING/OFFBOARDING/CROSSBOARDING)
- `OnboardingTask` — 템플릿 내 작업 정의 (assigneeType, dueDaysAfter, category 10종)
- `EmployeeOnboarding` — 직원별 온보딩 인스턴스 (status 5종, buddyId, linkedPlanId 크로스보딩)
- `EmployeeOnboardingTask` — 직원별 작업 인스턴스 (status 5종 TaskProgressStatus, assignee, blocked 추적)
- `OnboardingCheckin` — Day 1/7/30/90 마일스톤 체크인 (mood/energy/belonging, AI 요약)
- `OffboardingChecklist` — 회사·퇴직 유형별 마스터
- `OffboardingTask` — 오프보딩 작업 정의 (dueDaysBefore 기준)
- `EmployeeOffboarding` — 직원별 오프보딩 (ResignType 5종, handoverToId, isDoNotRehire 플래그)
- `EmployeeOffboardingTask` — 직원별 작업 인스턴스
- `OffboardingDocument` — 인수인계서·동의서·면담 보고서 등 첨부
- `AssetReturn` — 자산 회수 (5-state, 잔존가치, 임금 차감 동의)
- `ExitInterview` — 퇴직 면담 (ExitReason 6종, satisfactionDetail JSON 4영역, isConfidential, AI 요약)
- `M365ProvisioningLog` — Azure AD 프로비저닝 로그 (PROVISION/DEPROVISION/UPDATE × 3-state)
- `OnboardingSetting` — 회사별 온보딩 설정 (checklistTemplate JSON)

**핵심 enum**:
- `OnboardingTargetType`: NEW_HIRE / TRANSFER / REHIRE

### 11.5 알려진 위험 코드 경로

- 온보딩·오프보딩 자동 시작 — 채용/인사관리 모듈의 트리거 누락 시 인스턴스 미생성. 트리거 누락 모니터링 필요
- 크로스보딩 linkedPlanId — 한쪽 인스턴스 삭제·취소 시 다른 쪽의 참조 무결성 깨질 수 있음. 트랜잭션 처리 필수
- 자산 미반환 → 임금 차감 — 한국 근로기준법 §43은 임금 전액 지급 원칙을 정하며, 대법원 판례(2001다25184 등)는 자유로운 의사에 기한 명확한 서면 상계 합의 + 객관적 합리성이 있을 때만 임금 상계를 제한적으로 허용. consentDocExists(본인 자필 서면 동의서) + 잔존가치 산정 근거 + 차감 금액·사유를 모두 보관하도록 강제 검증 필요
- 퇴직 면담 isConfidential — 매니저 UI에서 노출되면 안 되는데 권한 가드 누락 시 노출 가능. UI·API 양쪽 검증 필수
- M365 API 실패 — Azure AD API 일시 장애 시 PENDING으로 남음. 재시도 로직·수동 처리 흐름 명확화 필요
- AssetReturn 상태 전이 — PENDING → RETURNED → DEDUCTED → CIVIL_CLAIM 순서 강제. 역행 차단 필요
- 재입사 (REHIRE) 시점에 DNR 검증 — 채용 모듈에서 자동 검증 누락 시 차단 직원이 다시 입사 가능 위험
- 작업 마감일 재계산 — `lastWorkingDate` 변경 시 모든 작업 마감일 자동 재계산. 누락 시 stale due date 위험
