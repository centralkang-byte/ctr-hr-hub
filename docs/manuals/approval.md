> **최종 갱신**: 2026-05-11
> **담당**: 인사관리팀
> **상태**: Draft v1 (CEO 리뷰 전)

## 1. 개요

### 1.1 무엇을 다루는가

CTR HR Hub의 결재·승인 모듈은 다음을 통합 관리합니다.

- **결재 흐름 정의** — 모듈별·법인별 결재 단계와 결재자 역할 등록
- **결재 라우팅** — 직원이 신청한 건의 결재자를 자동으로 지정 (직속 팀장, 부서장, 인사담당자, 대표 등)
- **다단계 결재** — 1단계 결재 후 다음 단계로 자동 이관 (예: 본부장 → 대표)
- **위임(대결)** — 결재자가 부재일 때 다른 사람이 대신 결재
- **통합 결재함** — 한 화면에서 휴가/성과/급여 등 여러 모듈의 결재 대기 건을 일괄 확인
- **자동 승인** — 일정 기간 결재가 진행되지 않으면 자동 승인 (선택 정책, **현재 비활성** — §3.5 참조)
- **감사 이력** — 누가 언제 어떤 의견으로 결재했는지 기록

### 1.2 도입 의도

- **위임전결규정(CP-A-03-04) 디지털화** — 종이/엑셀로 관리하던 결재 권한을 시스템에서 통합 운영
- **누락·지연 방지** — 결재자가 자동 지정되어 "누구에게 보내야 하나" 고민 없이 신청 가능
- **부재 대응** — 출장·휴가 중 결재자가 부재일 때 자동으로 대결자가 처리
- **감사 가능성** — 모든 결재 의사결정에 대한 이력 추적 (누가·언제·어떤 의견)
- **법인별 차등** — 13개 법인이 각자의 위임전결규정을 따를 수 있도록 법인별 결재 흐름 설정 지원

## 2. 결재 모듈

결재는 모듈 단위로 정의됩니다. 각 모듈마다 별도의 결재 흐름이 존재하며, 모듈별로 단계 수와 결재자 역할이 다릅니다.

### 2.1 모듈 12종 (기본 결재 흐름 시드)

| 모듈 | 기본 결재 흐름 | 단계 수 | 비고 |
|------|---------------|--------|------|
| 휴가 | 직속 팀장 | 1단계 | 일반 휴가 (연차·병가·경조 등) |
| 휴직 | **결재 흐름 엔진 미사용** (RBAC 권한 게이트) | — | `ApprovalFlow` 시드 없음. `PATCH /api/v1/leave-of-absence/[id]` action=approve가 `MODULE.LEAVE` UPDATE 권한자(HR_ADMIN 등)에 의해 직접 승인. loa.md §5.2 참조 |
| 급여 | 본부장 → 대표 | 2단계 | 월 급여 마감 후 결재 |
| 채용 (충원 요청) | 본부장 | 1단계 | **현재 시스템 운영 흐름**. 위임전결규정 Rev13은 CEO 전결 + 본부장·HR 합의 → 시스템과 불일치 (§9 참조) |
| 징계 | 인사담당자 → 대표 | 2단계 | HR 기안 → 대표 전결 |
| 증명서 발급 | 직속 팀장 | 1단계 | 재직증명서·경력증명서 등 |
| 근태 (정정·연장근무) | 직속 팀장 | 1단계 | 출퇴근 정정, 시간외근무 |
| 퇴직·사직 | 본부장 → 대표 | 2단계 | 사직서 결재 |
| 인사발령 | 대표 | 1단계 | 발령 통지 결재 |
| 수습 전환 | 직속 팀장 → 본부장 | 2단계 | 수습 → 정규직 전환 평가 |
| 계약직 전환 | 직속 팀장 → 본부장 | 2단계 | 계약 갱신·정규직 전환 |
| 비정기 급여 조정 | 직속 팀장 → 인사담당자 | 2단계 | 승급·특별인상 등 |

**근거**: 위임전결규정(CP-A-03-04, Rev13) 부표#1 업무전결권한기준표(Rev60~64).

### 2.2 결재자 역할 5종

각 결재 단계는 다음 역할 중 하나가 결재합니다.

| 역할 코드 | 설명 | 시스템 결정 방식 |
|----------|------|-----------------|
| 직속 팀장 | 신청자의 직속 상관 | 신청자 직책의 보고 라인을 따라가 상위 직책 보유자 |
| 부서장 | 신청자가 소속된 부서의 장 | 부서에 등록된 부서장 |
| 인사담당자 | 해당 법인의 인사 담당자 | 인사담당자 권한 보유 직원 (법인별) |
| 재무 담당자 | 해당 법인의 급여 관리 권한 보유자 | 급여 관리 권한 보유 직원 |
| 대표 | 해당 법인의 대표 또는 임원 | 전사관리자 또는 임원 권한 보유 직원 |

**특정 사용자 지정**: 위 역할 대신 특정 직원을 결재자로 직접 지정할 수도 있습니다 (예: "이 모듈은 반드시 김부장이 결재").

### 2.3 법인별 차등

기본 결재 흐름은 본사 표준(13개 법인 공통)으로 등록되며, 각 법인이 자기 정책으로 덮어쓸 수 있습니다.

- 본사 표준: 모듈별로 1건씩 (법인 미지정)
- 법인 오버라이드: 특정 법인이 자기 결재 흐름을 등록하면 해당 법인 직원은 오버라이드 우선 적용
- 예: 한국 본사는 급여 → 본부장→대표 2단계, 베트남 법인은 급여 → 법인장 1단계로 단축 가능

## 3. 핵심 정책

### 3.1 결재 라우팅 (자동 지정)

직원이 신청서를 제출하면 시스템이 다음 순서로 결재자를 결정합니다.

```
1. 신청자가 속한 법인의 결재 흐름 조회
   - 해당 법인이 등록한 모듈별 흐름이 있으면 그것 사용
   - 없으면 본사 표준 흐름 사용
2. 첫 단계의 결재자 역할을 실제 직원으로 변환
3. 그 직원에게 결재 요청 발송 (알림 + 통합 결재함에 등록)
```

**역할 → 실제 사람 변환 방식**:

| 역할 | 변환 로직 |
|------|----------|
| 직속 팀장 | 신청자의 주직책 → 보고 라인 따라간 상위 직책 → 그 직책 보유자 |
| 부서장 | 신청자의 주직책 소속 부서 → 부서에 등록된 부서장 |
| 인사담당자 | 해당 법인에서 인사담당자 권한을 보유한 직원 (여럿이면 가장 먼저 등록된 사람) |
| 재무 담당자 | 해당 법인에서 급여 관리 권한을 보유한 직원 |
| 대표 | 해당 법인 소속 전사관리자 또는 임원 (여럿이면 가장 먼저 등록된 사람) |

> **데이터 흐름 주의**: 위 역할이 실제 직원으로 변환되지 않으면(예: 부서에 부서장이 등록 안 됨, 해당 법인에 인사담당자 없음) 결재가 중단됩니다. 마스터데이터 검토 시 부서장·인사담당자·대표 권한 보유자 누락을 우선 확인하세요.

### 3.2 다단계 결재 진행

여러 단계가 등록된 모듈(예: 급여 본부장→대표)의 결재 흐름:

```
[신청]
   ↓
[1단계: 본부장 결재 대기]
   ├── 승인 ──→ [2단계: 대표 결재 대기]
   │                ├── 승인 ──→ [최종 승인 완료]
   │                └── 반려 ──→ [최종 반려, 신청자에게 알림]
   └── 반려 ──→ [최종 반려, 신청자에게 알림]
```

- 한 단계가 승인되면 다음 단계 결재자에게 자동 알림
- 어느 단계든 반려되면 전체 신청이 반려 처리 (이전 단계 승인 이력은 보존)
- 모든 단계 승인 시에만 최종 승인

### 3.3 셀프 결재 방지

원칙적으로 신청자 본인은 결재자가 될 수 없습니다.

- **주직책 매니저가 본인인 경우**: 시스템에 등록된 자기 주직책이 본인의 상사 직책으로 잡혀 있는 드문 케이스에서는, 기존 동작 호환을 위해 자동 승인 경로로 처리 (예외 케이스, 일반적으로 발생하지 않음)
- **보조직책 매니저가 본인인 경우**: 차단됨. 보조 직책으로 자기 자신을 결재자로 만드는 우회 시도는 막힘
- **채용 충원 요청**: 신청자가 부서장 본인이어도 셀프 결재 추가 차단 (별도 가드)

### 3.4 위임 (대결)

결재자가 부재 시 다른 사람이 대신 결재할 수 있습니다.

**위임 설정 항목**:

| 항목 | 설명 |
|------|------|
| 위임자 | 결재 권한을 넘기는 사람 (원래 결재자) |
| 대결자 | 결재를 대신 수행할 사람 |
| 범위 | 휴가만 / 전 업무 |
| 시작일·종료일 | 위임이 유효한 기간 |
| 사유 | 출장·연수·휴가 등 부재 사유 (선택 입력) |

**상태 3종**:

- **활성**: 시작일~종료일 사이, 대결자가 결재 가능
- **만료**: 종료일 경과 → 만료된 위임으로 표시 (사용 시점 차단)
- **회수**: 위임자가 위임 종료 전에 직접 회수 (출장에서 조기 복귀 등)

**위임 자동 만료 cron 미확인**: `expireOverdueDelegations()` 함수는 코드에 존재하나 호출 cron이 `vercel.json` 또는 코드 흐름에 명시적이지 않습니다. 현재는 위임 사용 시점에 종료일 비교(만료된 위임 사용 차단)로 처리되어 운영상 안전하나, 종료일 지난 위임의 상태 자동 정리는 **검증 필요** 항목입니다. 인사담당자가 위임 설정 화면에서 수시 확인 권장 (§9 #11 참조).

> **현재 위임 범위 한계**: 위임의 "전 업무" 범위는 데이터 모델로 지원되지만, 실제로 결재 흐름이 위임을 참조하는 모듈은 **현재 휴가 모듈만**입니다. 다른 모듈(급여·채용·징계 등)은 위임 설정이 있어도 원래 결재자에게만 알림이 가고, 대결자에게는 이관되지 않습니다. **추후 개선** 예정 (§9 #1 참조).

### 3.5 자동 승인 (선택 정책)

각 결재 단계는 "N일 후 자동 승인" 옵션을 설정할 수 있습니다.

- 예: 휴가 신청 → 직속 팀장 결재 단계에 "7일 후 자동 승인" 설정 → 7일간 결재 안 하면 시스템이 자동 승인
- 결재 누락으로 인한 신청자 피해 방지가 목적
- 기본값은 비활성 (자동 승인 없음, 결재자가 명시적으로 결재해야 함)

> **현재 비활성**: `auto-acknowledge` cron 코드는 존재하나(`src/app/api/v1/cron/auto-acknowledge/route.ts`) `vercel.json`에 등록되지 않아 **현재 실행되지 않습니다**. 결재 흐름 설정에 N일 후 자동 승인을 입력해도 동작하지 않으며, 현재는 수동 결재만 가능합니다. cron 등록 후 활성화 시 옵션 동작 (§9 #12 참조).

### 3.6 통합 결재함

결재자가 자기에게 들어온 모든 결재 대기 건을 한 화면에서 처리할 수 있습니다.

**통합되는 모듈**: 휴가 신청, 성과목표 검토(MBO), 급여 결재

**필터**:
- 상태: 대기 / 승인 완료 / 반려 / 전체
- 모듈: 휴가 / 성과 / 급여 / 전체
- 기간: 최근 30일 (기본) 또는 사용자 지정

**우선순위 자동 부여**:
- 휴가 3일 이상 → 높음
- 휴가 3일 미만 → 보통
- 급여 결재 → 항상 높음

**사이드바 뱃지**: 미결재 건수가 사이드바 아이콘에 표시됩니다.

> **현재 통합 결재함의 범위**: 휴가·성과·급여 3개 모듈만 통합되어 있습니다. 채용 충원 요청, 근태 정정, 징계, 증명서 등은 통합되지 않고 각 모듈의 자체 화면에서 별도로 결재합니다. **추후 개선** 예정 (§9 #2 참조).

## 4. 워크플로

### 4.1 결재 라이프사이클

```
[신청자]              [시스템]                    [결재자]
신청 ─────────────→ 결재 흐름 조회 ─────────→
                    1단계 결재자 결정 ────────→ 결재함에 표시
                                                  │
                                                  ├── 승인 ──→ 다음 단계 or 완료
                                                  ├── 반려 ──→ 최종 반려
                                                  └── (N일 무응답) ──→ 자동 승인 (설정 시)
```

**상태 3종 (각 결재 단계)**: 대기 / 승인 / 반려

**전체 신청 상태**: 결재 진행 중 / 최종 승인 / 최종 반려 / 신청 취소

### 4.2 위임 적용 시 라우팅

```
[신청] → [원래 결재자: 본부장 출장 중, 위임 활성]
                    │
                    ↓
            [대결자: 부본부장에게 알림]
                    │
            ├── 대결자 승인 ──→ 다음 단계
            └── 대결자 반려 ──→ 최종 반려
```

- 결재 결과 이력에는 "대결자 X가 위임자 Y의 권한으로 결재"가 기록됨
- 위임 종료 후에도 결재 이력은 보존

### 4.3 자동 알림

- **결재 요청 도착**: 결재자에게 Teams + 이메일 + 푸시
- **결재 결과**: 신청자에게 알림 (승인/반려 모두)
- **위임 시작 알림**: 대결자에게 "오늘부터 X일까지 Y님의 결재 대결자입니다"
- **위임 만료 임박**: 위임자에게 종료 1일 전 알림 (출장 연장 시 갱신 유도)

## 5. 역할별 권한

### 5.1 행위 주체 매트릭스 — 각 작업을 **수행할 수 있는 역할**

> 이 표는 "누가 어떤 작업을 수행할 수 있는가"입니다. **결재 라우팅(누가 누구의 신청을 결재하는가)은 §5.2 참조**.

| 작업 | 직원 | 팀장 | 인사담당자 | 전사관리자 |
|------|:----:|:----:|:---------:|:---------:|

### 5.3 위임 권한

- **팀장 이상**: 본인이 부재 시 다른 사람에게 위임 가능 (휴가 범위만 현재 동작)
- **본인이 등록한 위임만 회수 가능** — 다른 사람의 위임은 인사담당자/전사관리자만 회수 가능
- **위임 중첩 금지**: A가 B에게 위임 활성 상태에서 B가 C에게 다시 위임은 막지 않으나, 결재 라우팅은 첫 위임만 따름

## 6. 화면 안내

### 6.1 결재자 — 통합 결재함

위치: `결재 → 결재함`

- **상단 필터**: 모듈(휴가/성과/급여/전체), 상태(대기/승인/반려/전체), 기간
- **카드 목록**: 신청자 / 모듈 / 요약 / 신청 일시 / 마감일 / 우선순위
- **일괄 처리**: 여러 건 선택 후 일괄 승인/반려 (휴가 모듈)
- **개별 처리**: 카드 클릭 → 상세 모달 → 의견 입력 후 승인/반려

### 6.2 위임 설정

위치: `결재 → 위임 설정`

- **현재 활성 위임**: 본인이 위임자로 등록한 활성 건 목록
- **신규 위임 등록**: 대결자 선택 → 범위(휴가만/전 업무) → 기간 → 사유 → 저장
- **위임 회수**: 활성 위임을 종료일 전에 회수
- **위임 이력**: 만료·회수된 과거 위임 조회

### 6.3 결재 흐름 설정 (인사담당자/전사관리자)

위치: `설정 → 시스템 → 결재 플로우`

- 좌측: 법인 선택 (전사관리자만 13개 법인 선택, 인사담당자는 본인 법인 고정)
- 우측: 모듈별 결재 흐름 카드 12개
- 각 모듈 카드: 단계 추가/삭제, 각 단계의 결재자 역할 선택, 자동 승인 일수 설정
- 본사 표준 vs. 법인 오버라이드 시각적 구분

### 6.4 모듈별 결재 화면 (참고)

통합 결재함에 통합되지 않은 모듈은 각자의 화면에서 결재합니다.

- 채용 충원 요청 → `채용 → 충원 요청 → 결재 대기 탭`
- 근태 정정·연장근무 → `근태 → 결재` 또는 별도 결재 화면
- 징계 → `인사관리 → 징계 → 결재` (모듈 가동 여부에 따라 다름)

## 7. 특수 상황 처리

### 7.1 결재자 부재 (위임 미설정)

- 결재 요청은 원래 결재자에게 그대로 발송
- 알림 누적 → 자동 승인 일수 설정이 있으면 그 일수 경과 시 자동 승인
- 자동 승인이 없는 경우: 인사담당자 또는 전사관리자가 수동으로 결재 흐름 우회 가능

### 7.2 결재자 퇴사·이직

- 결재 요청 발송 후 결재자가 퇴사한 경우: 시스템이 자동 재라우팅 안 함
- 인사담당자가 수동으로 처리 (해당 신청을 재라우팅하거나 직접 결재)
- **추후 개선** 항목: 퇴사/이직 시 결재 대기 건 자동 재라우팅 (§9 #6 참조)

### 7.3 결재자가 신청자 본인

- 본인이 자기 신청을 결재하는 상황은 시스템이 차단
- 예외 (드문 케이스): 주직책 매니저가 본인으로 잡혀 있는 hierarchy → 자동 승인 경로로 처리 (기존 동작 호환)
- 보조 직책으로 본인이 결재자가 되는 경우는 무조건 차단

### 7.4 위임 중첩

- A → B 위임 활성, B가 또 C에게 위임을 등록한 경우
- 시스템은 등록은 막지 않지만, 결재 라우팅은 첫 위임(A → B)만 따름
- 결과: B 부재 시 A의 결재 건은 B에게 가고, C는 받지 못함
- 이중 위임이 필요한 경우 인사담당자가 수동 처리 권고

### 7.5 다단계 결재 중간 단계 부재

- 예: 본부장 → 대표 2단계 중 본부장 부재 + 위임 미설정
- 본부장에게 알림만 누적, 자동 승인 일수가 없으면 무한정 대기
- 인사담당자/전사관리자가 본부장의 위임 설정을 대신 등록하거나, 직접 결재 우회 가능

### 7.6 법인 변경자의 진행 중 결재 건

- 직원이 법인을 이동(전입·전출)하는 경우 이미 결재 진행 중인 건은 원래 법인의 결재 흐름을 그대로 따름
- 새 법인의 결재 흐름은 이동 후 신청 건부터 적용

## 8. 자주 묻는 질문

**Q. 결재자를 누가 정하나요?**
A. 시스템이 자동으로 정합니다. 신청 시점에 신청자의 법인·소속·직책을 기반으로 결재 흐름에서 정의된 역할(직속 팀장·본부장·대표 등)을 실제 직원으로 변환합니다. 사용자가 결재자를 직접 선택하지 않습니다.

**Q. 출장 가서 결재가 밀릴 것 같은데 어떻게 하나요?**
A. 출국 전에 `결재 → 위임 설정`에서 대결자를 등록하세요. 범위는 휴가만/전 업무 중 선택, 기간 지정. 단, 현재 위임이 실제로 동작하는 모듈은 휴가 모듈만입니다. 다른 모듈은 인사담당자에게 사전 요청 필요.

**Q. 본부장이나 부서장이 등록 안 된 부서는 결재가 어떻게 되나요?**
A. 결재 흐름에서 "부서장" 역할을 변환할 직원이 없으면 결재가 중단됩니다. 부서장·인사담당자·대표 권한 보유자 등록을 마스터데이터 단계에서 우선 확인하세요.

**Q. 결재가 일주일 이상 안 처리되면 자동 승인되나요?**
A. 모듈별 결재 흐름에 "자동 승인 일수"가 설정되어 있는 경우에만 자동 승인됩니다. 기본값은 비활성이며, 인사담당자가 `설정 → 시스템 → 결재 플로우`에서 단계별로 일수를 입력해야 작동합니다.

**Q. 설정 화면에서 휴가 흐름을 본부장 2단계로 등록했는데 왜 직속 팀장이 결재자로 잡히나요?**
A. 현재 휴가 모듈은 결재 흐름 설정을 읽지 않고 직속 팀장 1단계로 하드코딩되어 있습니다. 등록은 가능하지만 실제 동작은 변경되지 않습니다. **추후 개선** 항목 (휴가 매뉴얼 §5.2 + 본 매뉴얼 §9 #3 참조).

**Q. 다른 모듈은 결재 흐름 설정이 실제로 반영되나요?**
A. 모듈별로 다릅니다. 급여·채용·성과는 반영, 휴가·근태는 미반영(코드 하드코딩), 나머지(징계·증명서·인사발령 등)는 모듈 코드 연동 여부가 상이합니다. §5.2 표 참조.

**Q. 다단계 결재에서 중간 단계 결재자가 반려하면 어떻게 되나요?**
A. 전체 신청이 반려 처리되고 신청자에게 알림이 갑니다. 이전 단계의 승인 이력은 보존되지만, 최종 결과는 반려입니다.

**Q. 위임받은 사람이 결재하면 누가 결재한 것으로 기록되나요?**
A. 이력에는 "대결자 X가 위임자 Y의 권한으로 결재"라고 양쪽 모두 기록됩니다. 감사 시 추적 가능합니다.

**Q. 결재가 끝난 건을 취소하거나 되돌릴 수 있나요?**
A. 결재 완료 후에는 신청자가 직접 취소할 수 없습니다. 인사담당자만 사유와 함께 수동으로 되돌릴 수 있으며, 모듈별로 처리 방식이 다릅니다.

**Q. 인사담당자나 전사관리자가 결재 흐름을 무시하고 직접 승인할 수 있나요?**
A. 가능합니다. 두 역할은 모든 결재 건에 직접 승인 권한이 있으며, 흔히 출산휴가 등 HR 대리 등록이나 부재로 막힌 결재 우회에 사용됩니다.

| 7 | "합의" 프로세스 미구현 | 위임전결규정에서 "HR 합의" 요건이 있는 모듈(채용 등)이 있으나 시스템은 단순 순차 결재만 지원. **추후 개선** — 합의(병행 결재) 워크플로 신규 기능 개발 (ADR A1 Gap #6) |
| 8 | 궐위·대행 자동화 미지원 | 직책이 비어 있을 때 상위 직책 보유자에게 자동 위임되는 로직 없음. 현재는 수동 위임 설정으로 대응. **추후 개선** — Position 기반 궐위 대행 로직 (ADR A1 Gap #7) |
| 9 | 결재 흐름 변경 시 진행 중 건은 기존 흐름 유지 | 신청 시점 흐름을 따르므로 변경된 흐름은 신규 신청부터 적용. 일관성은 보장되나 긴급 정책 변경 반영 지연 가능 |
| 10 | 결재 의견 첨부 파일 미지원 | 결재 시 의견 텍스트만 입력 가능, 첨부는 신청 단계에서만 가능. **추후 개선** 검토 항목 |
| 11 | 위임 자동 만료 cron 미확인 | `expireOverdueDelegations()` 함수는 코드에 존재하나 호출 cron이 `vercel.json` 또는 코드 흐름에 명시적이지 않음. 사용 시점 차단(만료된 위임 사용 막음)으로 운영상 안전하나 상태 자동 정리는 검증 필요. **추후 개선** — cron 등록 또는 호출 위치 명시 |
| 12 | auto-acknowledge cron 미등록 | `src/app/api/v1/cron/auto-acknowledge/route.ts` 코드는 존재하나 `vercel.json` crons에 미등록 — N일 후 자동 승인 옵션 **현재 비활성**. **추후 개선** — `vercel.json`에 cron 등록 후 활성화 |
| 13 | 채용 결재 시드와 위임전결규정 불일치 | **시드** (`prisma/seeds/42-approval-flow-defaults.ts` recruitment): 본부장 1단계 / **위임전결규정 Rev13** (`docs/regulation-audit/A1-delegation-authority-matrix.md` L44): CEO 전결 + 본부장·HR 합의. 시스템은 본부장 1단계로 운영되며, HR 합의 워크플로는 시스템 외부 처리. **추후 개선** — 시드를 규정에 맞추거나, 규정을 운영 현실에 맞춰 개정 (시스템 fix 트랙) |
| 14 | 휴직 결재가 ApprovalFlow 엔진을 사용하지 않음 | `prisma/seeds/42-approval-flow-defaults.ts`에 `leave-of-absence` 모듈 시드 없음. `PATCH /api/v1/leave-of-absence/[id]` approve가 RBAC 권한(`MODULE.LEAVE` UPDATE) 단일 게이트로 동작 — 다단계 결재·위임·자동 승인 등 ApprovalFlow 기능 모두 비활성. **추후 개선** — 휴직 결재 흐름 엔진 통합 (시스템 fix 트랙) |

## 10. 추후 개선

**기능 개발 트랙 (라우팅·위임 정합화)**:
1. 모듈별 결재 코드에 위임(`checkDelegation`) 조회 로직 추가 → 위임 "전 업무" 범위 실제 동작 (§9 #1, ADR A1 Gap #5)
2. 통합 결재함 API에 채용·근태·징계 모듈 통합 (§9 #2)
3. 휴가 신청·승인 코드를 결재 흐름 설정 기반으로 재작성 (§9 #3, 휴가 매뉴얼 §10)
4. 근태 결재 라우트에 결재자 RBAC 검증 강화 (§9 #4)
5. 결재자 상태 변경(퇴사·이직) 시 활성 결재 건 자동 재라우팅 (§9 #6)
6. "합의" 워크플로 신규 기능 (§9 #7, ADR A1 Gap #6)
7. Position 기반 궐위·대행 자동화 (§9 #8, ADR A1 Gap #7)
8. 휴직 결재 ApprovalFlow 엔진 통합 — `leave-of-absence` 모듈 시드 추가 + PATCH 로직을 ApprovalFlow 기반으로 변경 (§9 #14)

**데이터 모델 수준**:
8. `ApprovalFlowStep.approverRole`을 string → enum 전환 → 컴파일 타임 검증
9. 시스템 Role ↔ 규정 직급 매핑 테이블 도입 (본부장·부문장 대응) (§9 #5)

**UI 수준**:
10. 결재 흐름 설정 UI에 "이 설정이 실제로 동작하는가" 모듈별 표시 (현재 휴가는 미동작)
11. 결재 의견 첨부 파일 지원 (§9 #10)

## 11. 부록: 기술 메모

> 본문이 정책 설명, 이 섹션은 개발/QA 참고용. HR은 건너뛰어도 무방합니다.

### 11.1 데이터 모델

핵심 모델 (모두 `prisma/schema.prisma`):

- `ApprovalFlow` — 모듈별 결재 흐름 정의 (companyId null = 본사 표준, 값 있음 = 법인 오버라이드)
- `ApprovalFlowStep` — 결재 단계 (stepOrder, approverType, approverRole, approverUserId, autoApproveDays)
- `ApprovalDelegation` — 위임 레코드 (delegatorId, delegateeId, scope, status, startDate, endDate)
- `RequisitionApproval` — 채용 충원 요청 결재 (별도 모델, ApprovalFlow와 연동)
- `PayrollApproval` + `PayrollApprovalStep` — 급여 결재 (currentStep/totalSteps, 별도 모델)
- `OffCycleApprovalStep` — 비정기 급여 조정 결재 단계
- `AttendanceApprovalRequest` + `AttendanceApprovalStep` — 근태 통합 결재 (휴가·연장근무·근태정정·교대변경)

**enum**:
- `DelegationScope`: `LEAVE_ONLY`, `ALL`
- `DelegationStatus`: `ACTIVE`, `EXPIRED`, `REVOKED`
- `PayrollApprovalStatus`: `PENDING`, `IN_PROGRESS`, `APPROVED`, `REJECTED`
- `PayrollApprovalStepStatus`: `PENDING`, `APPROVED`, `REJECTED`

### 11.5 알려진 위험 코드 경로

- `src/lib/delegation/resolve-delegatee.ts` — PROTECTED 마커. 위임 결재 체인 핵심. 수정 전 architecture review 필요
- `resolveApproverByRole.direct_manager` — 비대칭 정책(primary/secondary). validate vs. resolve 함수 정합성 유지 필요 (Session 204 패턴)
- `direct_manager` 셀프 결재 정책 — primary는 self 허용, secondary는 self 차단. 변경 시 hierarchy oscillation 가능 (Codex Gate 2 R3 P1 기록)
- `ApprovalFlowStep.approverRole` 값(문자열) — schema의 인라인 주석에만 존재. 컴파일 타임 enum 미사용 → 새 역할 추가 시 resolver `switch` 보완 필수
- 채용 충원 요청은 `RequisitionApproval` 별도 모델 사용. 통합 흐름과 데이터 모델 분리 → drift 위험 (현재 의도된 분리)
