# Knowledge Transfer (KT) 일정

> **대상**: CEO + 개발팀 리드 + 인프라팀 리드
> **목적**: CEO 솔로 운영 → IT 팀 분리 인계 4~6주 단계별 일정
> **작성 상태**: 🟡 **CEO + 팀 리드 합의 필요** — 본 문서는 권장 골격. 팀 리드 시작일·인원·역할 합의 후 확정.

---

## 단계 모델 (권장)

| Phase | 주차 | 단계 | CEO 역할 | 팀 역할 |
|-------|------|------|---------|---------|
| **Phase 1** | Week 1-2 | Doc + Walkthrough | 주도 (100%) | 학습·질문 |
| **Phase 2** | Week 3-4 | Shadow 운영 | 리뷰·승인 (50%) | 작업·결정 (50%) |
| **Phase 3** | Week 5-6 | Independent 운영 | 응답만 (10%) | 독립 (90%) |
| **Phase 4** | Week 7+ | 완전 이양 | Product owner | 100% 책임 |

> **타임라인**: UAT 종료 후 Week 1 시작 가정. UAT가 2026-05-19 ~ 06-02이면 KT Week 1 = 2026-06-09.

## Phase 1: Doc + Walkthrough (Week 1-2)

### 목표

- 팀이 시스템 전체 구조·핵심 모듈·운영 절차를 이해
- 모든 핸드오버 문서 검토 + 질문 해소
- 첫 hands-on (페어 프로그래밍)

### 인프라팀 트랙 (운영 런북 hands-on)

12 런북 × 1시간 = 12시간 (2주 분산). 실제 배포·env 변경·cron 트리거를 함께 수행.

| 일자 | 런북 |
|------|------|
| W1 D1 PM | 시스템 오버뷰 + 인프라 매트릭스 |
| W1 D2 AM | 배포 파이프라인 hands-on (staging 배포 1회) |
| W1 D2 PM | 환경 변수 운영 (vercel env add/rm 실습) |
| W1 D3 AM | DB 마이그레이션 (로컬 → staging 마이그레이션 1회) |
| W1 D3 PM | Cron 카탈로그 + 미등록 5개 수동 트리거 |
| W1 D4 AM | 시드 관리 + Phase 6B 검증 실행 |
| W1 D4 PM | Build & npm scripts + GitHub Actions |
| W1 D5 | 모니터링 (Sentry / Vercel / Supabase 대시보드 hands-on) |
| W2 D1 AM | 알림 채널 운영 (Teams / SES / Web Push) |
| W2 D1 PM | 보안 + IAM owner 이양 시작 |
| W2 D2 AM | 장애 대응 시뮬레이션 1 (예: prod hotfix 가상 시나리오) |
| W2 D2 PM | 급여 월간 운영 흐름 (HR Admin과 함께) |

### 공통 세션

| 일자 | 세션 |
|------|------|
| W1 D1 AM | **킥오프** — 전체 인계 일정 + 역할 분담 + 질문 |
| W2 D3 | **위험·결함 등록부 리뷰** — P0~P3 우선순위 합의 |
| W2 D4 | **Q&A 종합** — 모든 문서 검토 후 누락·불명확 항목 |
| W2 D5 | **Phase 2 진입 결정** — 팀 준비도 평가 + 책임 분담 |

- [ ] 시크릿 회전 9종 계획 합의

## Phase 2: Shadow 운영 (Week 3-4)

### 목표

팀이 실제 운영 작업을 수행, CEO는 리뷰·승인만. 작업 실수가 prod에 도달하기 전에 catch.

### 활동

**개발팀**:
- 신규 PR 작성 (위험·결함 등록부 P0 → P1 순)
- 코드 리뷰 + 머지는 CEO + 팀 리드 양쪽 승인
- pre-commit hook + Codex Gate 1/2 정상 동작 확인
- Vercel preview 검증 + main 머지

**인프라팀**:
- env 변경, 시크릿 회전 일부 (CEO 가이드 받으며)
- 배포 모니터링 (Sentry / Vercel logs 분석)
- 알림 채널 운영 인계 시작 (Sentry alert routing → 인프라팀 Slack)
- DB 백업 검증 시뮬레이션 1회

**공통**:
- 매일 standup (15분): 작업 / 막힌 점 / 결정 필요한 항목
- 주간 retro (30분): 다음 주 조정사항

### Phase 2 출구 기준

- [ ] 위험·결함 P0 5건 중 최소 3건 PR 머지
- [ ] 인프라팀이 prod 배포 5회 무사고 수행
- [ ] 시크릿 회전 9종 모두 완료
- [ ] IAM owner 14항목 모두 이양 완료
- [ ] Sentry / Vercel / Supabase 알림이 CEO 개인 이메일 → 팀 채널로 전환 완료
- [ ] 결제 카드 13종 CTR 법인 카드 전환 완료
- [ ] 첫 incident response (P2 이상) 팀이 주도 + CEO 리뷰

## Phase 3: Independent 운영 (Week 5-6)

### 목표

팀이 독립적으로 운영. CEO는 질문에만 응답.

### 활동

**개발팀**:
- 모든 PR 자체 리뷰 + 머지 결정 (팀 리드 권한)
- 위험·결함 P0 잔여 + P1 진입
- 매뉴얼·핸드오버 문서 갱신 책임 이전 (코드 변경 시 동기화)

**인프라팀**:
- 모든 운영 작업 독립 수행
- 모니터링 알람 1차 응대 (CEO는 P0/P1만 알림)
- 장애 대응 시뮬레이션 2회 (P0 + P1 가상 시나리오)
- 비용 모니터링 책임 이전

**공통**:
- standup 빈도 줄임 (주 3회)
- CEO는 product owner 모드 — 기능 우선순위·정책 결정에만 참여

### Phase 3 출구 기준

- [ ] 팀이 prod 배포 10회 + incident 1건 무사고 처리
- [ ] CEO 응답 빈도 < 일 1회
- [ ] 인계 체크리스트 ([07_인계_체크리스트.md](07_인계_체크리스트.md), Tier 3) 90% 완료
- [ ] 운영 매뉴얼·핸드오버 문서 팀 책임으로 이전됨

---

## Phase 4: 완전 이양 (Week 7+)

### 목표

CEO는 product owner. 운영·코드는 팀 100% 책임.

### 활동

- CEO 역할: 분기 1회 리뷰 + 신규 기능 우선순위 결정 + 임원 보고
- 팀 역할: 운영·개발·유지보수 전체 100%

### CEO Phase 4 출구 기준 (인계 종결)

- [ ] 인계 체크리스트 100% 완료
- [ ] 30일 연속 prod 무사고
- [ ] CTR Group 임원 보고서에 "인계 완료" 명시
- [ ] CEO 개인 계정에 묶인 액세스 0건 (cross-check audit)
