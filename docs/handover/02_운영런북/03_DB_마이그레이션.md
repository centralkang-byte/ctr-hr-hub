# 03. DB 마이그레이션

> **대상**: 개발팀 + 인프라팀
> **빈도**: 신규 모델·필드 추가 시 (월 1~3회)
> **현재 마이그레이션 수**: 43개 (`prisma/migrations/` 디렉토리)
> **핵심 모델 수**: 209 models / 142 enums (`prisma/schema.prisma`)

## 마이그레이션 흐름

```
[개발자 로컬]                  [Staging]              [Production]
schema.prisma 수정 →       git push staging →       git push main
prisma migrate dev →       migrate deploy            migrate deploy
(자동 SQL 생성)               (PR 머지 시)             (PR 머지 시)
```

## 신규 마이그레이션 작성

```bash
# 1. schema.prisma 수정 (모델 추가/필드 추가/관계 수정)

# 2. 로컬 dev DB에 마이그레이션 적용 + 이름 부여
npx prisma migrate dev --name <descriptive-name>
# 예: npx prisma migrate dev --name add-department-head-id

# 3. 생성된 SQL 검토 (prisma/migrations/<timestamp>_<name>/migration.sql)
#    - 데이터 손실 위험 확인 (DROP COLUMN, ALTER COLUMN NOT NULL 등)
#    - 백필 필요한지 확인

# 4. Prisma client 재생성 (postinstall이 처리하지만 확인)
npx prisma generate

# 5. 시드 검증 (clean DB 기준)
npx tsx prisma/seed.ts

# 6. 시드 정합성 검증 (Phase 6B)
npx tsx scripts/seed-validate-phase6b.ts

# 7. PR 생성 → staging 머지 → CI E2E 통과 → main 머지
```

## 위험한 마이그레이션 (백필 SQL 필요)

다음 변경은 마이그레이션 SQL에 **직접 백필 코드 추가** 필요:
- `NOT NULL` 컬럼 추가 (기본값 없이) — 기존 행에 값 주입 필요
- `UNIQUE` 제약 추가 — 중복 행 정리 후 적용
- 컬럼 타입 변경 (예: `Int` → `String`) — 변환 로직
- 관계 변경 (FK 추가) — 참조 무결성 백필

예시 패턴 (Session 201 Department.head_employee_id):
```sql
-- 1. 컬럼 nullable로 추가
ALTER TABLE "Department" ADD COLUMN "head_employee_id" UUID;

-- 2. 휴리스틱 백필 (가능한 만큼만)
UPDATE "Department" d
SET head_employee_id = (
  SELECT e.id FROM "Employee" e
  WHERE e.department_id = d.id AND e.position LIKE '%부장%'
  LIMIT 1
);

-- 3. FK 추가 (NULL 허용 유지 — 운영팀이 점진 보강)
ALTER TABLE "Department"
  ADD CONSTRAINT "Department_head_employee_id_fkey"
  FOREIGN KEY ("head_employee_id") REFERENCES "Employee"("id");
```

## RLS (Row-Level Security)

CTR HR Hub는 **법인(company) 격리 RLS**를 적용:
- SSOT 파일: `src/lib/api/companyFilter.ts` (resolveCompanyId)
- 트랜잭션 래퍼: `src/lib/api/withRLS.ts`
- Prisma 래퍼: `src/lib/prisma-rls.ts`

⚠️ 위 3개 파일은 **DO NOT TOUCH** (CLAUDE.md). 보안 핵심 SSOT. 수정 필요 시 architecture review.

신규 모델 추가 시 RLS 적용 여부 결정:
- 회사 데이터인가? → `companyId` 필드 필수 + RLS 적용
- 시스템·플랫폼 데이터인가? (예: `Currency`, `Country`) → RLS 불필요

## 스테이징 → 프로덕션 배포

마이그레이션 SQL은 Vercel build 시 자동 실행:
```bash
# package.json: postinstall에 prisma generate 있음
# 별도로 vercel-build에 migrate deploy를 추가하거나, build command에 포함
```

⚠️ **현재 build pipeline에 `prisma migrate deploy`가 명시적으로 등록되어 있는지 확인 필수**. 없다면 `package.json` build script에 추가:
```json
"build": "prisma migrate deploy && next build"
```

또는 Vercel UI에서 build command 설정.

## 백업·복원

| 작업 | 명령 | 빈도 |
|------|------|------|
| Supabase 자동 백업 확인 | Supabase 대시보드 → Backups | 매일 자동 (Supabase Pro+) |
| 수동 DB 덤프 (긴급) | `pg_dump $DATABASE_URL > backup-$(date +%F).sql` | 마이그레이션 전 |
| 특정 시점 복원 (PITR) | Supabase 대시보드 → Database → Backups → Restore | 사고 시 |

⚠️ Production DB write가 가능한 시크릿 (`DATABASE_URL`)은 인프라팀 리드 2명 이상만 보유 권장.

## 마이그레이션 실패 시

```bash
# 1. 현재 마이그레이션 상태 확인
npx prisma migrate status

# 2. 실패한 마이그레이션 표시 확인 ("X migrations have not yet been applied")

# 3a. 롤백 가능한 경우 (SQL 작성 시점 백워드 호환):
git revert <commit-with-migration>
# 다시 마이그레이션 작성

# 3b. 롤백 불가능한 경우 (데이터 손실 수반):
# - Supabase 백업 복원 (PITR)
# - 또는 수동 SQL로 정정
# - 또는 prisma migrate resolve (전문가 모드 — DB 상태와 마이그레이션 표를 강제 동기화)
```
