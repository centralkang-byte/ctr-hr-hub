# CTR HR Hub — GP#4 성과 사이클 설계서 v1.1 (최종)

> **문서 목적:** GP#4 성과관리 파이프라인 최종 설계안. 인사팀/법인 HR/경영진 합의 및 개발 에이전트 구현 지시용.
> **작성일:** 2026-03-10
> **버전:** v1.1 Final (CTO/CEO 아키텍처 결정사항 3건 + 보완 2건 반영 + v1.1b 코드스캔 검증: CycleStatus 7단계 확정 — DRAFT/ACTIVE/CHECK_IN/EVAL_OPEN/CALIBRATION/FINALIZED/CLOSED)

---

## 1. 왜 성과 사이클을 다시 설계하는가

현재 CTR의 성과 모듈(MBO 목표, 역량 평가, 캘리브레이션 등)은 **개별 기능으로는 구현되어 있지만, 하나의 파이프라인으로 연결되어 있지 않습니다.**

GP#4의 목표는 이 모듈들을 **Workday 수준의 end-to-end 파이프라인**으로 연결하여:

- 목표 설정부터 보상 반영까지 **한 번의 사이클로 자동 흐르게** 만들고
- 평가의 **공정성**을 시스템적으로 보장하며 (동료평가, 캘리브레이션, AI 편향 감지)
- 최종 결과가 **급여 인상에 자동으로 연계**되는 구조를 완성하는 것입니다.

---

## 2. 7단계 파이프라인 요약

```
STEP 1        STEP 2        STEP 3         STEP 4          STEP 5           STEP 6         STEP 7
사이클 개시 → 목표 설정  → 중간 체크인 → 평가 실시   → 캘리브레이션 → 결과 통보  → 보상 기획
(HR 설정)    (직원↔매니저)  (필수/권장)   (자기+동료+상사) (HR+임원)      (매니저→직원)  (HR→급여)
                                                                                          ↓
                                                                                     PayrollRun 반영
```

| 단계 | 주체 | 소요 기간 (가이드) |
|------|------|:---:|
| STEP 1 사이클 개시 | HR | 1~2일 |
| STEP 2 목표 설정 | 직원 → 매니저 승인 | 2~4주 |
| STEP 3 중간 체크인 | 매니저 + 직원 | 평가 기간 중간 |
| STEP 4 평가 실시 | 직원 + 동료 + 매니저 | 2~3주 |
| STEP 5 캘리브레이션 | HR + 부서장 + 임원 | 1~2주 |
| STEP 6 결과 통보 | 매니저 → 직원 확인 | 1~2주 |
| STEP 7 보상 기획 | HR → 승인 → 급여 반영 | 1~2주 |

## 3. 🚨 핵심 아키텍처 원칙: 파이프라인 병목 방지

> **이 원칙은 STEP 2, 3, 4 전체에 공통 적용됩니다.**

### 문제

60명 법인에서 1명이라도 목표 미제출 / 체크인 미완료 / 자기평가 미제출이면, 전체 사이클이 다음 단계로 못 넘어가는 **병목(Bottleneck)** 현상.

### 해결: 개인별 독립 진행 + Overdue 페널티

```
마감일 도래
    ↓
사이클 전체(PerformanceCycle)는 → 다음 단계로 정상 전환
    ↓
미완료자 개인(PerformanceReview)은 → 현재 단계에 머무르며 "지연(Overdue)" 태그 영구 부여
    ↓
STEP 4 매니저 평가 화면에 → 🚨 경고 배지로 노출
```

### 단계별 적용

| 단계 | 마감 시 미완료자 처리 | Overdue 배지 내용 |
|------|---------------------|------------------|
| **STEP 2 목표 설정** | 목표 미제출/미승인자 개인만 지연 | 🚨 "이 직원은 목표 설정을 N일 지연 제출했습니다" |
| **STEP 3 중간 체크인** | 체크인 3조건 미충족자 개인만 지연 (필수 모드 시) | 🚨 "이 직원은 중간 체크인을 완료하지 않았습니다" |
| **STEP 4 자기평가** | 자기평가 미제출자 개인만 지연 | 🚨 "이 직원은 자기평가를 N일 지연 제출했습니다" |

**핵심:** 시스템이 직접 점수를 깎지 않습니다. Overdue 정보를 매니저에게 제공하여 **매니저의 판단에 반영되도록** 합니다. 이는 Workday의 "Manager Insight" 철학과 일치합니다.

## 4. STEP 1 — 사이클 개시

HR이 새로운 평가 사이클을 시스템에 등록하고 설정합니다.

### 설정 항목

| 설정 | 설명 | 예시 |
|------|------|------|
| 평가 기간 | 시작일 ~ 종료일 | 2026-H1: 2026.01.01 ~ 2026.06.30 |
| 평가 대상 | 법인/부서/레벨 필터 | CTR-KR 전체, 수습 제외 |
| 단계별 마감일 | 각 STEP의 마감 기한 | 목표설정 2/15, 체크인 4/30, 평가 6/30 |
| 등급 체계 | 등급 수 + 라벨 | 4등급 (아래 참조) |
| 배분 가이드라인 | 등급별 권장 비율 | 10/30/50/10 (아래 참조) |
| 동료평가 사용 여부 | ON/OFF + 설정 | ON, 3~5명, 반익명 |
| MBO:BEI 비중 | 기본값 + 레벨별 조정 | 기본 60:40 (아래 참조) |
| 중간 체크인 모드 | 필수(게이트) / 권장(Continuous) | 법인별 설정 |

### 사이클 관리: 하이브리드 방식

CTR은 6개국에 걸쳐 있고, 법인별 평가 주기가 다릅니다:

| 법인 | 평가 주기 |
|------|----------|
| CTR-KR, CTR-CN, CTR-VN | 반기 (연 2회) |
| CTR-US, CTR-RU, CTR-MX | 연간 (연 1회) |

**설계:** HQ에서 글로벌 사이클 1개를 생성하고, 법인별로 기간/마감일/등급 체계 등을 오버라이드합니다. 급여 파이프라인(GP#3)과 동일한 패턴.

## 5. 등급 체계

### 4등급 시스템

| 등급 | 한글 라벨 | 영문 라벨 | 정의 |
|:---:|----------|----------|------|
| **E** | 탁월 | Exceeds | 기대를 크게 상회하는 성과. 조직에 눈에 띄는 기여 |
| **M+** | 우수 | Meets+ | 기대를 충족하며 일부 영역에서 상회 |
| **M** | 보통 | Meets | 기대 수준을 충족하는 안정적 성과 |
| **B** | 미흡 | Below | 기대에 미달. 개선 계획(PIP) 대상 |

**왜 4등급인가?**

- 5등급(S/A/B+/B/C) 대비 **중간 등급이 없어서 평가자가 결정을 회피할 수 없음**
- Workday가 글로벌 고객에게 권장하는 방식
- Microsoft, Google 등 글로벌 기업 트렌드와 일치

### 가이드라인 배분 (강제 아님)

| 등급 | 권장 비율 |
|:---:|:---:|
| E (탁월) | 10% |
| M+ (우수) | 30% |
| M (보통) | 50% |
| B (미흡) | 10% |

- 이 비율은 **가이드라인**입니다. 캘리브레이션에서 초과하더라도 **경고만 표시**되며, 확정은 가능합니다.
- 강제 배분(Bell Curve)은 최근 글로벌 트렌드에서 후퇴 중이므로 채택하지 않습니다.
- 캘리브레이션 화면에서 "권장 분포 vs 실제 분포" 차트가 실시간으로 표시됩니다.

## 6. STEP 2 — 목표 설정

### 플로우

```
직원: MBO 목표 입력 (3~5개)
    ↓
매니저: 검토
    ├→ 승인 → 확정(Lock)
    └→ 반려 → 직원 재작성 → 재제출
```

### 목표 설정 규칙

| 규칙 | 내용 |
|------|------|
| 목표 수 | 3~5개 권장 (Settings에서 min/max 조정 가능) |
| 가중치 | 직원이 설정, 매니저가 조정 가능 |
| **가중치 합계** | **100% 강제** — 저장 시 합계 검증 |
| 마감 후 Lock | 마감일 이후 수정 불가 (HR만 예외 해제 가능) |
| Cascading | 상위 조직 목표 → 개인 목표 연결 (선택 사항) |
| **병목 방지** | 마감 시 미완료자는 개인만 Overdue, 사이클은 정상 진행 (섹션 3 참조) |

### 매니저 승인

- 매니저가 승인하면 목표가 확정(Lock)되어 직원이 수정할 수 없습니다.
- 반려 시 매니저 코멘트 필수 → 직원이 수정 후 재제출.
- 마감일까지 미제출 시 매니저에게 알림.

## 7. STEP 3 — 중간 체크인

### 두 가지 운영 모드 (Settings에서 선택)

| 모드 | 동작 | 권장 대상 |
|------|------|----------|
| **필수 (게이트)** | 체크인 완료 조건을 충족하지 않으면 개인이 Overdue 처리 | 성과 관리 문화 정착 초기 단계 |
| **권장 (Continuous)** | 체크인을 독려하되, 미완료해도 경고만. 사이클은 정상 진행 | 문화가 성숙한 법인 |

- 법인별로 다른 모드 설정 가능 (예: CTR-KR은 필수, CTR-US는 권장)
- 모드와 무관하게 체크인 기록은 STEP 4 평가 시 매니저에게 참고 자료로 표시됩니다.
- **병목 방지:** 필수 모드에서도 미완료자 개인만 Overdue 처리, 사이클 전체는 정상 전환 (섹션 3 참조)

> **설계 배경:** Workday는 체크인을 별도 모듈("Continuous Performance")로 운영하며 필수가 아닙니다. 그러나 CTR처럼 성과 관리를 처음 시스템화하는 조직에서는 강제하지 않으면 형식화될 가능성이 높으므로, 두 모드를 모두 지원하여 조직 성숙도에 맞게 선택할 수 있도록 합니다.

### 완료 조건 (필수 모드 시 3가지 모두 충족 필요)

| 조건 | 누가 | 내용 |
|------|------|------|
| ① 매니저 체크인 기록 | 매니저 | 1:1 미팅 결과 코멘트 최소 1회 등록 |
| ② 직원 체크인 기록 | 직원 | 본인 시각의 진행 상황 코멘트 최소 1회 등록 |
| ③ 목표 진행률 업데이트 | 직원 | MBO 목표별 중간 진행률(%) 입력 |

### 왜 중간 체크인이 중요한가?

- 매니저가 최근 한두 달만 기억하는 **최신 편향(Recency Bias)** 방지
- 목표가 비현실적이었을 때 **중간에 조정할 기회** 제공
- 직원이 "내 성과를 매니저가 알고 있다"는 **심리적 안전감** 확보
- 기존 CFR(Continuous Feedback) 및 1:1 미팅 모듈과 자연스럽게 연동

## 8. MBO:BEI 평가 비중

### 기본 구조

종합 점수 = **MBO 달성도 × MBO 비중** + **BEI 역량 점수 × BEI 비중**

### 레벨 체계

CTR은 내부적으로 L1~L5 + Executive 레벨을 사용하며, **법인별로 레벨에 매핑되는 직급명이 다를 수 있습니다.**

| 레벨 | 구분 | KR 예시 | US 예시 | CN 예시 |
|:---:|:---:|---------|---------|---------|
| L1 | 일반직 | 사원 | Associate | 职员 |
| L2 | 일반직 | 대리 | Senior Associate | 主管 |
| L3 | 일반직 | 과장 | Manager | 经理 |
| L4 | 일반직 | 차장 | Senior Manager | 高级经理 |
| L5 | 일반직 | 부장 | Director | 总监 |
| Exec | 직책자(임원) | 이사/상무/대표 | VP/SVP/President | 副总/总经理 |

### 비중 설정

> **글로벌 기본값은 60:40 (MBO:BEI)이며, 레벨별/법인별로 자유롭게 오버라이드할 수 있습니다.**
>
> 예시:
> - CTR-KR L1~L2를 50:50으로 조정 (역량 성장 강조)
> - CTR-KR Executive를 70:30으로 조정 (결과 책임 강조)
>
> 설정하지 않으면 글로벌 기본값(60:40)이 자동 적용됩니다.

### 우선순위

```
레벨별 설정 > 법인별 설정 > 글로벌 기본값 (60:40)
```

설정이 없으면 상위 값을 상속합니다.

## 9. STEP 4 — 평가 실시

가장 복잡한 단계입니다. **세 가지 평가가 병렬로 진행**된 후 매니저가 종합합니다.

### 플로우

```
┌─ 동료평가 지명 (매니저) → 동료평가 수집 (동료들) ──┐
│                                                     │
│  사이클 상태: EVALUATION                            ├→ 매니저 종합 평가 → 제출
│                                                     │
└─ 자기평가 작성 (직원) → 제출 ──────────────────────┘
```

- **병목 방지:** 자기평가 미제출자는 개인만 Overdue 처리, 사이클은 정상 진행 (섹션 3 참조)

### 핵심 원칙: Informed Evaluation

매니저는 **자기평가 + 동료평가 결과를 모두 확인한 후에** 최종 평가를 작성합니다.

- 매니저가 먼저 평가하고 나중에 자기평가를 보는 방식(일부 기업)은 편향을 강화하므로 **채택하지 않습니다.**
- 매니저 평가 화면에 표시되는 참고 정보:
  - 자기평가 내용
  - 동료평가 익명 요약
  - 중간 체크인 기록
  - 🚨 Overdue 배지 (목표 지연, 체크인 미완료, 자기평가 지연 등)

### 9-1. 자기평가

| 항목 | 내용 |
|------|------|
| MBO 달성도 | 목표별 달성률(%) + 근거 서술 |
| BEI 역량 | CTR 핵심가치 4개 항목별 자기 점수(1~5) + 근거 |
| 종합 코멘트 | 성과 요약, 성장 포인트, 지원 필요 사항 |

### 9-2. 상사평가

| 항목 | 내용 |
|------|------|
| MBO 달성도 | 목표별 달성률(%) + 매니저 판단 근거 |
| BEI 역량 | 핵심가치 4개 항목별 점수(1~5) + 근거 |
| 종합 등급 | E / M+ / M / B 중 선택 |
| 종합 코멘트 | 강점, 개선 영역, 육성 방향 |

### 동료평가 점수의 역할

> **동료평가 결과는 최종 등급 산출 공식에 합산되지 않습니다.**
>
> 동료평가의 점수와 서술은 다음 단계에서 **참고 자료**로 활용됩니다:
> - **STEP 4** 매니저 종합 평가 시 — 매니저 평가 화면에 동료 피드백 요약 패널 표시
> - **STEP 5** 캘리브레이션 시 — "동료평가 평균 vs 상사평가" 괴리가 큰 경우 알림
>
> **설계 배경:** 동료평가 점수를 최종 등급에 수학적으로 합산하면 "점수 거래" (서로 좋은 점수 주고받기) 리스크가 발생합니다. 매니저가 동료 피드백을 참고하되, 본인 판단으로 종합하는 것이 더 신뢰도 높은 평가를 만듭니다. 이는 Workday의 권장 방식과도 일치합니다.

## 10. 동료평가 (360도) 상세

### 워크플로우

```
STEP 4-a  매니저가 동료 지명 (3~5명)
             ├ 시스템이 추천 후보 풀 자동 생성
             └ 매니저가 후보에서 선택 + 풀 밖 직원 추가 가능
    ↓
STEP 4-b  지명된 동료에게 알림 → 평가 양식 작성
    ↓
STEP 4-c  마감 → 매니저에게 익명 요약 전달
    ↓
STEP 4-d  매니저가 종합 평가에 동료 피드백 참고
```

### 동료 추천 후보 풀 (시스템 자동 생성)

시스템이 다음 데이터를 기반으로 추천 후보를 자동 생성합니다:

| 시그널 | 소스 | 설명 |
|--------|------|------|
| 같은 부서/프로젝트 | 조직도 + EmployeeAssignment | 직접 협업 관계 |
| 승인 체인 관계 | ApprovalFlow | 업무상 상호작용 이력 |
| CFR 피드백 교환 | CFR 모듈 | 피드백을 주고받은 이력 |
| 1:1 미팅 기록 | OneOnOne 모듈 | 정기적 소통 이력 |

- 매니저는 추천 목록에서 선택하거나, **전체 직원 검색으로 추천 외 인원도 추가** 가능합니다.
- 최종 3~5명을 확정합니다 (Settings에서 min/max 조정 가능).

### 왜 직원이 아닌 매니저가 지명하는가?

> **설계 배경:** Workday 기본 방식은 "직원이 제안 → 매니저가 승인"입니다. 그러나 CTR에서의 **7년간 동료평가 운영 경험**에 따르면:
>
> - **직원이 지명할 경우:** 매니저가 사회적 관계를 고려해 직원의 선택을 거의 수정하지 않음 → 친밀한 동료끼리 지명 → **점수 상향 평준화**
> - **HR/매니저가 지명할 경우:** 실제 협업 관계 기반으로 선정 → **점수 정상 분포 회복**
>
> 한국 조직 문화에서는 매니저 주도 지명이 더 공정한 결과를 만듭니다. 시스템이 협업 데이터 기반 후보를 추천하여 매니저의 판단을 보조합니다.

### 익명성: 반익명

| 누가 보는가 | 평가자 이름 노출 |
|------------|:---:|
| 평가 대상 직원 | ✗ 익명 |
| 직속 매니저 | ✓ 공개 |
| HR/임원 | ✓ 공개 |

- **직원에게는 익명**이므로 솔직한 피드백이 가능합니다.
- **매니저에게는 공개**이므로 피드백의 맥락을 파악할 수 있습니다 (예: "협업이 어려웠다"는 피드백이 어느 팀에서 왔는지).

### 평가 양식

CTR 핵심가치 4개 항목을 기준으로 평가합니다:

| 항목 | 점수 | 서술 |
|------|:---:|:---:|
| Challenge (도전) | 1~5점 | 코멘트 (선택) |
| Trust (신뢰) | 1~5점 | 코멘트 (선택) |
| Responsibility (책임) | 1~5점 | 코멘트 (선택) |
| Respect (존중) | 1~5점 | 코멘트 (선택) |
| **종합 코멘트** | — | 필수 |

- 점수는 캘리브레이션에서 동료평가-상사평가 괴리 감지에 활용됩니다.
- 서술은 매니저의 종합 평가 시 참고 자료로 활용됩니다.

### 미응답 처리

| 상황 | 처리 |
|------|------|
| 마감일까지 미응답 | 해당 동료 평가 제외 |
| 최소 인원(3명) 미달 | 매니저에게 알림 → 추가 지명 또는 미달 상태로 진행 |

## 11. STEP 5 — 캘리브레이션

### 목적

**"A팀 매니저는 평가가 후하고, B팀은 박하다"** 같은 평가자 편향을 조정하여 **조직 전체의 공정성**을 확보합니다.

### 플로우

```
전원 평가 제출 완료 (미제출자는 Overdue 처리 후 사이클 전환)
    ↓
부서별/법인별 캘리브레이션 세션 자동 생성
    ↓
HR + 부서장이 등급 분포 검토
  - 가이드라인(10/30/50/10) 대비 실제 분포 차트
  - 동료평가 평균 vs 상사평가 괴리 감지
  - AI 편향 감지 알림
  - 등급 조정 (근거 필수 기록)
    ↓
법인별 최종 확정 → 임원 승인
```

### 🚨 핵심 아키텍처: 등급 데이터 투트랙 보존 (Two-Track Traceability)

**문제:** 캘리브레이션에서 매니저가 부여한 원본 등급을 덮어쓰면, 향후 매니저 평가 성향 분석이 불가능해집니다.

**해결:**

| 필드 | 내용 | 수정 가능 여부 |
|------|------|:---:|
| **originalGrade** | 매니저가 STEP 4에서 부여한 원래 등급 | ✗ 영구 보존 (절대 수정 불가) |
| **finalGrade** (calibratedGrade) | 캘리브레이션 후 최종 확정 등급 | ✓ 캘리브레이션에서만 수정 |

**Data Masking (직원 보호):**
- 직원이 본인 결과를 조회할 때: **finalGrade만 표시** (originalGrade는 API 응답에서 제외)
- HR/임원 권한일 때: **두 등급 모두 조회 가능** (매니저 성향 분석, 감사 추적용)

> **활용 가치:**
> - "김팀장은 3년 연속 원본 등급이 실제보다 1단계 높았다" → 다음 사이클에서 사전 코칭 가능
> - 원본과 최종의 괴리 통계 → 캘리브레이션 효과 측정
> - 직원은 finalGrade만 보므로 "원래 B였는데 A로 올려줬다" 같은 불필요한 정보 노출 차단

### AI 편향 감지 (기 구현)

시스템이 자동으로 감지하는 편향 유형:

| 편향 유형 | 설명 | 예시 알림 |
|----------|------|----------|
| 관대화 편향 | 특정 매니저가 전원 E/M+ | "김팀장의 평가 평균이 조직 평균 대비 +1.2 높습니다" |
| 엄격화 편향 | 특정 매니저가 전원 M/B | "박팀장의 평가 평균이 조직 평균 대비 -0.8 낮습니다" |
| 중심화 편향 | 전원 M+/M에 몰림 | "이팀장의 등급이 2개 등급에 90% 집중되어 있습니다" |
| 성별 편향 | 성별에 따른 등급 차이 | "해당 부서에서 성별 간 평균 등급 차이가 유의미합니다" |
| 연차 편향 | 근속연수에 따른 등급 차이 | "5년 이상 근속자의 평균 등급이 유의미하게 높습니다" |
| 동료-상사 괴리 | 동료평가와 상사평가 점수 차이 큼 | "해당 직원의 동료평가 평균(4.2)과 상사평가(2.5) 간 괴리가 큽니다" |

- 편향 감지는 **알림만 제공**하며, 최종 판단은 HR/임원이 합니다.
- 등급 조정 시 **조정 사유 기록이 필수**입니다 (감사 추적).

## 12. STEP 6 — 결과 통보

### 결과 통보 플로우

```
캘리브레이션 확정
    ↓
매니저에게 "결과 통보 요청" 알림
    ↓
매니저가 직원별 면담 실시
    ↓
시스템에서 "통보 완료" 클릭
    ↓
직원에게 알림 → 최종 등급(finalGrade) + 피드백 확인 화면
    ↓
직원이 "확인(Acknowledge)" 클릭
    ↓
7일 내 미확인 시 → 자동 확인 처리 (로그 기록)
    ↓
전원 확인 완료 → 사이클 CLOSED → STEP 7 보상 기획 시작
```

### 직원 확인 (Acknowledge)

| 규칙 | 내용 |
|------|------|
| 확인 기한 | 통보 후 **7일** |
| 미확인 처리 | 7일 경과 시 **자동 확인** (시스템 로그에 "미확인 자동처리" 기록) |
| 확인의 의미 | "결과를 열람했음"이지, "결과에 동의함"이 아닙니다 |
| 표시 내용 | **finalGrade만 표시** (originalGrade는 직원에게 노출 안 됨) |

- 자동 확인 처리로 인해 1명의 미확인이 전체 보상 연계를 막지 않습니다.
- 직원이 결과에 이의가 있을 경우, v1에서는 HR에게 직접 연락합니다 (시스템 이의 절차는 v2 예정).

## 13. STEP 7 — 보상 기획 (Compensation Review)

### 왜 별도 단계가 필요한가?

성과 등급이 확정되었다고 바로 급여에 반영할 수 없습니다. 그 사이에 **보상 기획** 단계가 필요합니다:

- 전사/법인별 **인상 예산 총액** 확인 (올해 인건비 예산 내에서 배분)
- 법인별/부서별 **인상 총액 조율** (특정 부서에 예산이 편중되지 않도록)
- **핵심 인재 리텐션**, 시장 임금 조정 등 특수 케이스 반영
- 경영진의 최종 승인

> **설계 배경:** Workday에서는 성과 프로세스와 보상 프로세스가 별도 Business Process로 분리되어 있습니다. 성과 확정 후 HR이 "Compensation Review"를 명시적으로 시작하며, 이 과정에서 예산 배분과 개인별 조정이 이루어집니다.

### Merit Matrix (등급 × 밴드 위치)

같은 등급이라도 **현재 연봉이 밴드 내 어디에 위치하느냐**에 따라 인상률이 달라집니다.

| | 밴드 하위 25% | 밴드 중간 | 밴드 상위 25% |
|---|:---:|:---:|:---:|
| **E (탁월)** | 8~10% | 6~8% | 4~6% |
| **M+ (우수)** | 5~7% | 4~6% | 3~5% |
| **M (보통)** | 3~5% | 2~4% | 1~3% |
| **B (미흡)** | 0~2% | 0~1% | 0% |

> 위 표는 글로벌 기본값 예시입니다. 법인별로 물가/환율/시장 임금 차이를 반영하여 오버라이드 가능합니다.

### 🚨 핵심 아키텍처: Merit Matrix 예외 처리 (Soft Warning & Escalation)

**문제:** 핵심 인재 리텐션 등 실무적 예외 상황에서 시스템이 입력을 원천 차단하면 HR의 유연성이 떨어집니다.

**해결 (유연한 통제):**

```
HR이 추천 범위 벗어난 인상률 입력 (예: 가이드라인 4%인데 10%)
    ↓
① 즉시 붉은색 경고(Soft Warning) 표시
    ↓
② 예외 사유(Exception Reason) 텍스트 입력 — 필수 (미입력 시 저장 불가)
    ↓
③ 저장 가능 — 입력 자체는 차단하지 않음
    ↓
④ 최종 승인권자(대표/CFO) 결재 화면에서
   🚨 [매트릭스 이탈 예외 건]으로 최상단 하이라이트
```

### 보상 기획 대시보드

| KPI | 내용 |
|-----|------|
| 법인별 인상 예산 vs 실제 | 예산 초과/잔여 현황 |
| 부서별 평균 인상률 | 부서 간 형평성 확인 |
| 등급별 인상률 분포 | Merit Matrix 가이드라인 대비 실제 |
| **🚨 예외 건 통계** | **매트릭스 이탈 건수 / 전체 대비 비율 / 예외 사유 분류** |

> 예외 건 통계를 통해 경영진이 "예외가 너무 많으면 매트릭스 자체를 조정해야 하는 건 아닌가?" 를 판단할 수 있습니다.

### 보상 기획 플로우

```
사이클 CLOSED
    ↓
① 시스템이 직원별 자동 산출:
   - 최종 등급(finalGrade) 확인
   - 현재 연봉의 밴드 내 위치(Compa-Ratio) 계산
   - Merit Matrix에서 인상률 범위 + 추천값 생성
    ↓
② HR이 법인별/부서별 총액 검토:
   - 인상 총액이 예산 내인지 확인
   - 부서별 배분 균형 조율
    ↓
③ HR이 개인별 미세 조정 (범위 내):
   - 핵심 인재 리텐션 가산
   - 시장 임금 조정 등 특수 사유
   - 범위 초과 시 → Soft Warning + 예외 사유 필수
    ↓
④ 경영진 최종 승인
   - 예외 건 🚨 최상단 하이라이트
   - 대시보드에서 전체 현황 확인
    ↓
⑤ 확정 → CompensationHistory에 기록 → 다음 PayrollRun에 반영
```

## 14. 상태 머신

### 사이클 전체 상태 (PerformanceCycle) — 구현 확정 7단계

> **참고:** 설계 초기 9단계 → 구현 시 7단계로 확정. 코드베이스 스캔(2026-03-11)으로 최종 검증 완료.

```
DRAFT → ACTIVE → CHECK_IN → EVAL_OPEN → CALIBRATION → FINALIZED → CLOSED
```

| 상태 | 설계상 대응 | 진입 조건 | 주체 |
|------|-----------|----------|------|
| DRAFT | SETUP | HR이 사이클 생성 | HR |
| ACTIVE | GOAL_SETTING | HR이 개시 클릭. 직원이 MBO 목표를 설정하고 매니저가 승인. 미완료자는 Overdue | 직원+매니저 |
| CHECK_IN | CHECK_IN | 목표 마감일 도래 또는 HR 수동 전환. 매니저-직원 중간 체크인 진행. 미완료자는 Overdue | 매니저+직원 |
| EVAL_OPEN | EVALUATION | 체크인 마감 후. 자기평가 + 동료평가 + 매니저평가 병렬 진행. 미완료자는 Overdue | 직원+동료+매니저 |
| CALIBRATION | CALIBRATION | 평가 마감 (미제출자는 Overdue) | HR+임원 |
| FINALIZED | FINALIZED | 임원 캘리브레이션 승인 완료. 매니저가 결과 통보 + 직원 확인(7일 미확인 시 자동 Acknowledge) | 매니저→직원 |
| CLOSED | CLOSED + COMP_REVIEW + COMP_COMPLETED | 전원 확인 완료 → 보상 기획 → 경영진 승인 → PayrollRun 반영 | HR+시스템 |

### 개별 평가 상태 (PerformanceReview)

사이클 안에서 직원 개인별 평가 진행 상태:

```
NOT_STARTED → GOAL_SETTING → SELF_EVAL → PEER_EVAL → MANAGER_EVAL → CALIBRATED → NOTIFIED → ACKNOWLEDGED
```

- 각 상태에서 마감일 초과 시 **OVERDUE 플래그** 추가 (상태 자체는 변경하지 않음)
- Overdue 플래그는 이후 단계에서 누적 표시 (예: "목표 5일 지연 + 체크인 미완료")

### 캘리브레이션 데이터 모델

```
PerformanceReview {
  ...
  originalGrade    // 매니저 원본 등급 (STEP 4) — 영구 보존
  finalGrade       // 캘리브레이션 후 최종 등급 (STEP 5)
  calibrationNote  // 조정 사유 (변경 시 필수)
  overdueFlags     // ["GOAL_LATE_5D", "CHECKIN_MISSING"] 등
}
```

## 15. 이벤트 체인

| 이벤트 | 트리거 | 자동 처리 |
|--------|--------|----------|
| CYCLE_STARTED | HR이 사이클 개시 | 대상 직원에게 목표 설정 알림 |
| GOAL_SUBMITTED | 직원이 목표 제출 | 매니저에게 승인 요청 알림 |
| GOAL_APPROVED | 매니저 승인 | 개인 상태 업데이트 |
| GOAL_OVERDUE | 목표 마감일 도래 + 미완료자 존재 | 미완료자에 Overdue 태그, 사이클은 CHECK_IN 전환 |
| CHECKIN_COMPLETED | 체크인 3조건 충족 | 개인 상태 업데이트 |
| CHECKIN_OVERDUE | 체크인 마감일 도래 + 미완료자 존재 | 미완료자에 Overdue 태그, 사이클은 EVALUATION 전환 |
| SELF_EVAL_SUBMITTED | 직원 자기평가 제출 | 매니저에게 알림 |
| SELF_EVAL_OVERDUE | 자기평가 마감 + 미제출자 존재 | 미제출자에 Overdue 태그 |
| PEER_NOMINATION_COMPLETED | 매니저 동료 지명 완료 | 지명된 동료에게 평가 요청 알림 |
| PEER_EVAL_SUBMITTED | 동료 평가 제출 | 전원 완료 시 매니저에게 종합 가능 알림 |
| MANAGER_EVAL_SUBMITTED | 매니저 평가 제출 (originalGrade 저장) | 전원 완료 시 → CALIBRATION 전환 체크 |
| CALIBRATION_ADJUSTED | HR/임원이 등급 조정 | finalGrade 저장 (originalGrade 보존) + 조정 사유 필수 |
| CALIBRATION_APPROVED | 임원 승인 | → FINALIZED, 매니저에게 통보 요청 알림 |
| RESULT_NOTIFIED | 매니저 통보 완료 | 직원에게 확인 요청 알림 (finalGrade만 포함) |
| RESULT_ACKNOWLEDGED | 직원 확인 (또는 7일 자동) | 전원 완료 시 → CLOSED |
| COMP_REVIEW_STARTED | HR이 보상 기획 시작 | Merit Matrix 기반 추천값 자동 생성 |
| COMP_EXCEPTION_FLAGGED | HR이 매트릭스 범위 초과 입력 | Soft Warning + 예외 사유 필수 + 승인 화면 하이라이트 |
| COMP_APPROVED | 경영진 승인 | CompensationHistory 기록 → PayrollRun 반영 |

## 19. Workday와의 설계 비교 (CTR 특화)

CTR이 의도적으로 Workday와 다르게 설계한 항목과 그 근거입니다.

| 항목 | Workday 방식 | CTR 설계 | 근거 |
|------|-------------|---------|------|
| 중간 체크인 | 별도 모듈, 필수 아님 | Settings에서 "필수/권장" 선택 | 도입기에는 강제가 필요, 성숙기엔 완화 |
| 동료 지명 주체 | 직원이 제안 → 매니저 승인 | 시스템 추천 → 매니저 선택 | 7년 경험 — 직원 주도 시 점수 상향 평준화 (한국 문화) |
| 동료평가 반영 | 정성적 참고 (주류) | 정량+정성 수집, 최종 등급에 합산 안 함 | 점수 거래 방지. Workday 방향과 동일 |
| 보상 연계 | 별도 Compensation Review | 별도 STEP 7 보상 기획 | 예산 총액 확인 없이 자동 반영은 위험 |
| 파이프라인 병목 | 일반적으로 마감일에 일괄 전환 | 사이클은 진행 + 개인만 Overdue | 1명 때문에 전체가 멈추는 것 방지 |
| 등급 데이터 | 최종 등급만 저장 (일반적) | originalGrade + finalGrade 투트랙 | 매니저 성향 분석 + 직원 보호 (Data Masking) |
| Merit Matrix 예외 | 범위 초과 차단 또는 단순 경고 | Soft Warning + 예외 사유 필수 + 승인 화면 에스컬레이션 | 유연성과 거버넌스 동시 확보 |

## 20. 피드백 요청 사항

이 설계서를 검토하신 후, 다음 질문에 대해 피드백 부탁드립니다:

### 인사팀 (전사)

1. 4등급 체계(탁월/우수/보통/미흡)가 현재 평가 운영에 적합한가요?
2. 가이드라인 배분(10/30/50/10)의 비율이 적절한가요?
3. 중간 체크인 — 우리 법인에는 "필수" 모드와 "권장" 모드 중 어떤 것이 적합한가요?
4. 동료평가를 매니저가 지명하는 방식에 대한 현장 반응은 어떨까요?
5. Overdue(지연) 태그가 평가 화면에 표시되는 것에 대한 직원/매니저 수용성은?

### 법인 HR

6. 법인별로 MBO:BEI 비중, 등급 체계, 캘리브레이션 방식 중 다르게 설정하고 싶은 항목이 있나요?
7. 동료평가 운영이 가능한 법인 규모인가요? (소규모 법인에서는 익명성 보장이 어려울 수 있음)
8. 엣지 케이스 7번(법인 간 전출 시 사이클 처리)에 대해 현실적으로 어떤 방식이 적합한가요?
9. 레벨(L1~L5)과 법인 내 직급명 매핑이 현재 기준과 맞나요?

### 경영진

10. Merit Matrix 인상률 범위(예시 표)가 경영 예산과 부합하나요?
11. 보상 기획 최종 승인 권한은 SUPER_ADMIN(대표/CFO)으로 적절한가요?
12. 임원(Executive)의 MBO:BEI 비중을 어떻게 설정하고 싶으시나요?
13. Merit Matrix 예외 건에 대한 에스컬레이션 프로세스가 적절한가요?

## 부록 A: 용어 정리

| 용어 | 설명 |
|------|------|
| MBO | Management by Objectives — 목표 기반 성과 관리 |
| BEI | Behavioral Event Interview — 역량(행동) 기반 평가 |
| CFR | Continuous Feedback & Recognition — 상시 피드백 |
| 캘리브레이션 | 평가자 간 편향을 조정하여 조직 전체 공정성 확보 |
| Merit Matrix | 등급 × 연봉밴드 위치에 따른 인상률 결정 도구 |
| Compa-Ratio | 현재 연봉 ÷ 밴드 중간값 — 밴드 내 위치를 나타내는 비율 |
| Acknowledge | 결과 "열람 확인" (동의가 아닌 확인) |
| PIP | Performance Improvement Plan — 성과 개선 계획 (미흡 등급 대상) |
| Compensation Review | 성과 확정 후 예산 내에서 보상을 기획하는 별도 프로세스 |
| Overdue | 마감일 초과 미완료 상태 — 개인에게 영구 태그 부여 |
| Data Masking | 권한에 따라 API 응답 필드를 선택적으로 노출/차단 |
| originalGrade | 매니저가 부여한 캘리브레이션 전 원본 등급 |
| finalGrade | 캘리브레이션 후 확정된 최종 등급 |
| Soft Warning | 입력을 차단하지 않되, 경고와 사유 기록을 요구하는 방식 |
| Golden Path | 모듈들을 end-to-end 파이프라인으로 연결하는 프로젝트 |
