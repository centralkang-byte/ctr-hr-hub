## 0. GP#2 개요 — 왜 온보딩과 오프보딩을 하나의 Golden Path로 묶는가

온보딩과 오프보딩은 **동일 엔진의 정방향/역방향**입니다:

| 공통 기반 | 온보딩 (정방향) | 오프보딩 (역방향) |
|----------|:---:|:---:|
| 체크리스트 + 태스크 자동 생성 | ✓ | ✓ |
| BLOCKED 깃발 시스템 | ✓ | ✓ |
| 동적 Nudge | ✓ | ✓ |
| UnifiedTask Hub 연동 | ✓ | ✓ |
| 법인별 템플릿 오버라이드 | ✓ | ✓ |
| 이벤트 기반 자동화 | ✓ | ✓ |

**핵심 차이점:**

| 항목 | Onboarding | Offboarding |
|------|-----------|-------------|
| 시간 방향 | 입사일 **이후** (Day 1→90) | 퇴직일 **이전** (D-30→D-Day) |
| 듀 날짜 계산 | `hireDate + dueDays` | `lastWorkingDate - dueDaysBefore` |
| 긴급도 | 점진적 (90일 여유) | 하드 데드라인 (퇴직일 불변) |
| 보안 민감도 | 낮음 | 높음 (계정 비활성화, 데이터 접근 차단) |
| 감정 요소 | 소속감 형성 (감정 체크인) | 지식 이전 + 관계 정리 (퇴직 면담) |
| 완료 게이트 | 매니저 Sign-off | 자동 완료 + 급여 정산 |
| 추가 절차 | 수습 평가 연계 | M365 자동 차단 + 자산 공제 |

# PART 0: 공유 아키텍처 원칙

> **이 원칙들은 온보딩과 오프보딩 모두에 공통 적용됩니다.**

## P0-1. 이벤트 기반 자동화

```
트리거 이벤트 발행 (fire-and-forget, TX 외부)
    ↓
핸들러가 수신
    ↓
회사 전용 템플릿 조회 (없으면 글로벌 fallback)
    ↓
인스턴스 + 태스크 일괄 자동 생성
```

HR은 한 번만 액션(직원 등록 또는 퇴직 시작)하면 나머지는 시스템이 알아서 처리합니다.

## P0-2. BLOCKED 깃발 시스템 (애자일 방식)

**문제:** 태스크 간 선후행 관계가 존재하지만, 엄격한 순서 제어(Waterfall)는 불필요한 대기를 만듭니다.

**해결:** 모든 태스크는 해당 일자에 **동시 오픈**되지만, 선행 업무 지연으로 진행이 막히면 담당자가 **상태를 BLOCKED로 변경 + 사유 입력**합니다.

```
태스크 상태: BLOCKED
사유: "선행 태스크 미완료 — 담당자 처리 대기 중"
    ↓
즉시 효과:
  - HR 대시보드에 🔴 BLOCKED 경고 노출
  - 관련 담당자에게 알림 발송
  - UnifiedTask Hub에서 BLOCKED 태스크 별도 필터 가능
    ↓
선행 태스크 완료 → 담당자(또는 HR)가 BLOCKED 해제
  - 해제 사유 기록 (선택)
  - 담당자에게 "진행 가능합니다" 알림
```

**Unblock 권한:**

| 역할 | BLOCKED 설정 | BLOCKED 해제 |
|------|:---:|:---:|
| 태스크 담당자 | ✓ | ✓ (본인 태스크) |
| HR_ADMIN | ✓ | ✓ (전체) |
| 매니저 | ✗ | ✓ (팀원 태스크) |

**BLOCKED 운영 규칙:**

| 규칙 | 내용 |
|------|------|
| BLOCKED 설정 시 | 사유 입력 필수 |
| BLOCKED 알림 | 즉시 HR 대시보드에 🔴 경고 + 관련 담당자에게 알림 |
| BLOCKED 해제 | 담당자 본인, 매니저, HR이 해제 가능 |
| BLOCKED 이력 | 모든 BLOCKED/Unblock 이력 기록 (감사 추적) |
| BLOCKED 태스크 Nudge | BLOCKED 상태 2일 초과 시 → 관련 담당자에게 추가 Nudge |
| BLOCKED 집계 | HR 대시보드에 "현재 BLOCKED 태스크 N건" 실시간 표시 |

## P0-3. 태스크 상태 머신

```
PENDING → IN_PROGRESS → DONE
    │         │
    │         └→ BLOCKED (선행 태스크 지연 — 사유 필수)
    │              │
    │              └→ PENDING 또는 IN_PROGRESS (해제 시)
    │
    └→ SKIPPED (비필수 태스크에만 허용)
```

| 상태 | 설명 | 전환 |
|------|------|------|
| PENDING | 아직 시작 안 함 | → IN_PROGRESS, BLOCKED, SKIPPED |
| IN_PROGRESS | 진행 중 | → DONE, BLOCKED |
| DONE | 완료 | 최종 상태 |
| BLOCKED | 선행 업무 지연으로 진행 불가 | → PENDING, IN_PROGRESS (해제 시) |
| SKIPPED | 건너뜀 (비필수만) | 최종 상태 |

## P0-4. UnifiedTask Hub 연동

온보딩/오프보딩 태스크는 **통합 업무함(UnifiedTask Hub)**에 자동으로 표시됩니다:

| 매핑 | 온보딩 | 오프보딩 |
|------|--------|----------|
| 상태 | PENDING→PENDING, DONE→COMPLETED, SKIPPED→CANCELLED, BLOCKED→PENDING(blocked 표시) | 동일 |
| Priority | overdue 기반 동적 계산 (>3일 지연=URGENT, >0일=HIGH) | 퇴직일 역산 (기한 초과=URGENT, 당일=HIGH, 3일 이내=MEDIUM) |
| Title | `[온보딩] Day {N} ❘ {태스크명}` | `[퇴직] {태스크명}` |
| Assignee | EMPLOYEE=본인, BUDDY=buddyId, HR/IT=SystemActor | EMPLOYEE=본인, HR/IT/FINANCE=SystemActor |

## P0-5. 템플릿 시스템 (글로벌 + 법인별 오버라이드)

두 모듈 모두 동일한 패턴을 따릅니다:

```
companyId = NULL  → 글로벌 기본 템플릿
companyId = 'xxx' → 해당 법인 전용 템플릿 (오버라이드)
```

템플릿에 법인 전용이 있으면 그것을, 없으면 글로벌을 fallback으로 사용합니다.

# PART A: 온보딩 파이프라인

## A1. 왜 온보딩 파이프라인을 설계하는가

현재 CTR의 온보딩 모듈은 **체크리스트와 태스크가 구현되어 있지만, 신입 등록부터 완료까지 자동으로 흐르는 파이프라인이 아닙니다.**

온보딩 파이프라인의 목표는:

- HR이 신입을 등록하면 **체크리스트가 자동 생성되고, 관련자에게 태스크가 자동 배정**되며
- 태스크 진행 상황이 **통합 업무함(UnifiedTask Hub)**에서 한눈에 보이고
- 선행 태스크 지연 시 **BLOCKED 깃발**로 병목을 가시화하고, **자동 Nudge**로 독촉하며
- 전체 태스크 완료 후 **매니저 최종 서명(Sign-off)**을 거쳐야만 온보딩이 공식 완료되고, 수습 평가 파이프라인으로 연결되는 구조를 완성하는 것입니다.

### 왜 Onboarding이 중요한가?

| 이유 | 설명 |
|------|------|
| 첫인상 결정 | 입사 첫 90일 경험이 이직 의사에 가장 큰 영향 |
| 다관여자 협업 | HR + IT + 매니저 + 버디 + 본인 — 5개 역할이 동시 참여 |
| 법인별 차이 | 6개국별 필수 서류, IT 설정, 법적 요건이 모두 다름 |
| 이탈 방지 | 체계적 온보딩은 입사 1년 내 이직률을 50% 이상 낮춤 |
| 크로스보딩 | 법인 간 전출 시에도 동일 프레임워크 재활용 |

## A2. 4단계 파이프라인 요약

```
STEP 1          STEP 2          STEP 3              STEP 4
신입 등록    → 체크리스트 자동  → 태스크 수행       → 매니저 최종 서명
(HR 등록)      생성 + 배정      (Day 1/7/30/90)      (Sign-off → 완료)
    │               │                │                    │
    └─ EMPLOYEE_HIRED  └─ 자동 태스크 생성  └─ BLOCKED 깃발       └─ ONBOARDING_COMPLETED
       이벤트 발행        핸들러 실행         + Nudge 독촉           수습 평가 트리거
```

| 단계 | 주체 | 시점 |
|------|------|:---:|
| STEP 1 신입 등록 | HR | 입사일 전후 |
| STEP 2 체크리스트 자동 생성 | 시스템 (이벤트 핸들러) | STEP 1 직후 자동 |
| STEP 3 태스크 수행 | HR + IT + 매니저 + 버디 + 본인 | Day 1 ~ Day 90 |
| STEP 4 매니저 최종 서명 | 매니저 (Sign-off) | 모든 필수 태스크 완료 후 |

## A3. 온보딩 고유 원칙

### 원칙 A-1: 마일스톤 기반 태스크 구조 + 동적 Nudge

태스크를 시간 순서대로 4개 마일스톤으로 그룹화합니다:

| 마일스톤 | 시점 | 목적 | 예시 태스크 |
|---------|:---:|------|-----------|
| **Day 1** | 입사 당일 | 기본 셋업 | IT 계정 생성, 사원증 발급, 좌석 배정, 입사 서류 수령 |
| **Day 7** | 입사 1주 | 적응 확인 | 매니저 1:1, 팀 소개, 시스템 교육, 1주차 감정 체크인 |
| **Day 30** | 입사 1개월 | 업무 안착 | 업무 목표 설정, 버디 면담, 30일 감정 체크인 |
| **Day 90** | 입사 3개월 | 온보딩 종료 | 수습 평가 연계, 90일 감정 체크인, 온보딩 만족도 설문 |

**동적 Nudge:** 초기 마일스톤(Day 1)일수록 더 짧은 간격으로 독촉합니다. IT 계정이 안 만들어지면 나머지가 전부 막히니까요.

| 마일스톤 | 첫 알림 | 간격 | 최대 횟수 |
|---------|---------|------|:---:|
| Day 0-1 | 지연 1일 후 | 1일 | 2회 |
| Day 2-7 | 지연 1일 후 | 2일 | 3회 |
| Day 8+ | 지연 3일 후 | 3일 | 3회 |

### 원칙 A-2: 매니저 최종 서명 (Sign-off as Gatekeeper)

**문제:** 체크박스만 다 채웠다고 기계적으로 온보딩을 끝내면, 직원의 실제 적응도와 무관하게 수습 평가가 시작될 위험이 있습니다.

**해결:** 모든 필수 태스크가 DONE이 되어도 온보딩이 즉시 종료되지 않습니다. 직속 매니저에게 **[최종 완료 승인 (Sign-off)] 태스크**가 부여됩니다.

```
모든 isRequired=true 태스크가 DONE
    ↓
자동: 매니저에게 "온보딩 최종 서명 요청" 태스크 생성
    ↓
매니저가 확인하는 것:
  ├ 필수 태스크 전체 완료 여부 (시스템 자동 확인)
  ├ 감정 체크인 결과 요약 (Mood/Energy/Belonging 추이)
  ├ BLOCKED 이력 (병목이 있었는지)
  └ 매니저 본인의 판단: "이 직원은 온보딩을 제대로 마쳤는가?"
    ↓
매니저가 Sign-off 버튼 클릭
    ↓
ONBOARDING_COMPLETED 이벤트 발행
    ↓
후속 프로세스 트리거:
  - 수습 평가(Probation Evaluation) 시작
  - Employee.status 갱신
  - 온보딩 만족도 설문 발송
```

**왜 매니저 Sign-off인가:**

- **온보딩 → 수습 평가 연결 게이트** 역할: 매니저가 "준비됐다"고 판단해야 다음 단계로 넘어감
- **매니저에게 온보딩 품질 책임 부여**: 체크리스트를 형식적으로 채우는 것 방지
- **CTR의 3개월 수습 기간**에 특히 적합: "온보딩 마쳤으니 이제 본채용 심사 시작" 선언
- Workday의 "Manager Sign-off" 패턴과 일치

| 상황 | 처리 |
|------|------|
| 매니저 Sign-off 지연 (3일+) | Nudge 발송 (2일 간격, 최대 3회) |
| 매니저가 "아직 부족하다" 판단 | Sign-off 보류 + 추가 태스크 요청 가능 (HR 통해) |
| 매니저 부재 | Delegation 수임자가 Sign-off 가능 (GP#1 대결 시스템 활용) |

## A4. STEP 1 — 신입 등록 + 이벤트 발행

### 플로우

```
HR이 직원 등록 (POST /employees)
  ├ 이름, 이메일, 법인, 부서, 직급, 입사일
  ├ 고용 유형 (정규직/계약직/인턴)

DB에 Employee 레코드 생성

채용 파이프라인에서 합격 처리 시 원클릭으로 직원 등록이 가능합니다:

ATS에서 합격 확정 → "직원으로 등록" 버튼 → 직원 등록 + EMPLOYEE_HIRED 자동 발행
```

## A5. STEP 2 — 체크리스트 자동 생성

### 온보딩 템플릿

```
OnboardingTemplate (체크리스트 템플릿)
  ├ companyId (NULL = 글로벌 기본 템플릿)
  ├ targetType: NEW_HIRE / TRANSFER / REHIRE
  ├ isActive: true/false
  └ OnboardingTask[] (템플릿 태스크 목록)
      ├ title, description
      ├ assigneeType: HR / IT / MANAGER / BUDDY / EMPLOYEE
      ├ dueDays: 입사 후 N일 (마일스톤 결정)
      ├ isRequired: 필수 여부
      └ sortOrder
```

### 핸들러 로직 (employee-hired.handler)

```
EMPLOYEE_HIRED 이벤트 수신
    ↓
1. 회사 전용 템플릿 조회 (companyId + targetType 매칭)
2. 없으면 글로벌 fallback (companyId = NULL)
3. 글로벌도 없으면 warn 로그 + skip
    ↓
4. EmployeeOnboarding 인스턴스 생성
5. 템플릿 태스크 기반으로 EmployeeOnboardingTask 일괄 생성
6. 마지막에 "매니저 최종 서명(Sign-off)" 태스크 자동 추가
   - assigneeType: MANAGER
   - dueDays: 90 (Day 90 마일스톤)
   - isRequired: true
    ↓
중복 방지: EmployeeOnboarding이 이미 존재하면 skip
```

### 공유 함수

`create-onboarding-plan.ts` — Onboarding과 크로스보딩(crossboarding.ts)이 공유하는 함수입니다. 법인 간 전출(Transfer) 시에도 동일한 로직으로 온보딩 플랜을 생성합니다.

### 담당자(Assignee) 역할 분리

| 역할 | 담당 태스크 | 예시 |
|------|-----------|------|
| **HR** | 행정/법적 절차 | 입사 서류 수령, 4대보험 등록, 근로계약서 |
| **IT** | 시스템 셋업 | 이메일 계정, VPN, 장비 지급 |
| **MANAGER** | 업무 안착 + **최종 서명** | 1:1 면담, 업무 목표 설정, 수습 평가, **Sign-off** |
| **BUDDY** | 사회적 적응 | 회사 문화 안내, 점심 함께하기, 비공식 멘토링 |
| **EMPLOYEE** | 본인 참여 | 필수 교육 이수, 감정 체크인 응답, 서류 제출 |

## A6. STEP 3 — 태스크 수행

### 태스크 완료 시 이벤트

```
담당자가 태스크를 DONE으로 변경
    ↓
TX 내부:
  1. EmployeeOnboardingTask.status = DONE
  2. allRequiredDone 계산 (필수 태스크 중 미완료 건수 — Sign-off 태스크 제외)
  3. ONBOARDING_TASK_COMPLETED 이벤트 발행
  4. allRequiredDone = true이면:
     - 매니저 Sign-off 태스크를 PENDING → 활성화
     - 매니저에게 "온보딩 최종 서명 요청" 알림
```

**주의:** Sign-off 태스크는 다른 필수 태스크가 전부 완료되기 전에는 매니저에게 표시되지 않습니다 (또는 비활성 상태로 표시).

### 감정 체크인 (Emotion Check-in)

마일스톤 시점(Day 7 / Day 30 / Day 90)에 신입에게 감정 체크인을 요청합니다:

| 항목 | 내용 |
|------|------|
| 기분 (Mood) | 1~5점 |
| 에너지 (Energy) | 1~5점 |
| 소속감 (Belonging) | 1~5점 |
| 자유 코멘트 | 선택 |
| AI 요약 | 시스템 자동 생성 (ai_summary) |

- 체크인 결과는 매니저와 HR에게 공유됩니다.
- **소속감 3회 연속 2점 이하** → HR에게 🚨 Flight Risk 경고 알림
- 매니저 Sign-off 화면에 체크인 추이 차트 표시 (판단 근거)
- Workday의 "Onboarding Pulse" 기능과 유사

## A7. STEP 4 — 매니저 최종 서명 (Sign-off)

### 플로우

```
모든 isRequired=true 태스크가 DONE (Sign-off 태스크 자체 제외)
    ↓
Sign-off 태스크 활성화 → 매니저에게 알림
    ↓
매니저가 Sign-off 화면에서 확인:
  ├ ✅ 필수 태스크 전체 완료 (자동 확인)
  ├ 📊 감정 체크인 추이 (Mood/Energy/Belonging 그래프)
  ├ 🔴 BLOCKED 이력 요약 (병목이 있었는지)
  ├ 📝 버디/HR 코멘트 (있으면)
  └ 매니저 종합 코멘트 입력 (선택)
    ↓
매니저가 "온보딩 완료 승인" 클릭
    ↓
ONBOARDING_COMPLETED 이벤트 발행
    ↓
후속 프로세스 트리거:
  ├ 수습 평가(Probation Evaluation) 시작 → GP#4 연계
  ├ Employee 상태 갱신
  └ 온보딩 만족도 설문 발송
```

### Sign-off 엣지 케이스

| 상황 | 처리 |
|------|------|
| 매니저가 "아직 부족하다" 판단 | Sign-off 보류 + HR 통해 추가 태스크 생성 가능 |
| 매니저 Sign-off 3일+ 지연 | Nudge 발송 (2일 간격, 최대 3회) |
| 매니저 부재 | Delegation 수임자가 Sign-off 가능 (GP#1 대결 시스템 활용) |
| 수습 기간 없는 고용 유형 | Sign-off 없이 자동 완료 (계약직/인턴은 설정 가능) |

# PART B: 오프보딩 파이프라인

## B1. 왜 오프보딩 파이프라인을 설계하는가

퇴직은 온보딩의 역방향이지만, **놓치면 법적 리스크와 보안 사고로 직결**됩니다.

오프보딩 파이프라인의 목표는:

- HR이 퇴직 프로세스를 시작하면 **체크리스트가 자동 생성되고, 마감일 역산으로 태스크가 배정**되며
- 퇴직 당일 **M365 Graph API를 통해 IT 계정이 자동 차단**되고
- 퇴직 면담(Exit Interview) 데이터가 **직속 매니저로부터 철저히 격리**되어 진실된 이직 원인 분석이 가능하며
- 미반납 자산은 **급여 정산에서 자동 공제 상신**되어 회사 손실을 방어하는 **무결점(Zero-loophole) 파이프라인**을 구축하는 것입니다.

---

## B2. 4단계 파이프라인 요약

```
STEP 1           STEP 2            STEP 3              STEP 4
퇴직 프로세스  → 체크리스트 자동  → 태스크 수행       → 완료 + 정산
시작 (HR)        생성 + 배정       (D-30→D-Day)         (급여/자산 정산)
    │                │                 │                    │
    └─ OFFBOARDING    └─ 자동 태스크     └─ Nudge 독촉         └─ OFFBOARDING_COMPLETED
       _STARTED          생성              + M365 자동 차단        퇴직 정산 + 자산 공제
```

| 단계 | 주체 | 시점 |
|------|------|:---:|
| STEP 1 퇴직 프로세스 시작 | HR | 퇴직 통보 시 |
| STEP 2 체크리스트 자동 생성 | 시스템 (이벤트 핸들러) | STEP 1 직후 자동 |
| STEP 3 태스크 수행 | HR + IT + 매니저 + 본인 | D-30 ~ D-Day |
| STEP 4 완료 + 정산 | 시스템 + HR | D-Day 이후 |

---

## B3. 오프보딩 고유 원칙

### 원칙 B-1: 퇴직일 역산 스케줄링

온보딩은 "입사 후 N일"이지만, 오프보딩은 **"퇴직일 N일 전"**입니다.

```
퇴직일: 2026-03-31

D-30: 인수인계 대상자 지정 (매니저)
D-14: IT 장비 반납 안내 (IT)
D-7:  M365 계정 비활성화 예약 (IT/시스템)
D-3:  퇴직 면담 실시 (HR)
D-1:  사물함 정리, 사원증 반납 (본인)
D-Day: M365 자동 차단 실행 → 최종 퇴근
```

### 원칙 B-2: 하드 데드라인 — 공격적 Nudge

퇴직일은 **변경 불가능한 하드 데드라인**입니다. 온보딩보다 Nudge 간격이 훨씬 짧습니다:

| 퇴직일까지 남은 일수 | 첫 알림 | 간격 | 최대 횟수 |
|---|---|---|---|
| D-14 이상 | 기한 초과 1일 후 | 2일 | 3회 |
| D-7 ~ D-13 | 기한 초과 즉시 | 1일 | 3회 |
| D-6 이내 | 기한 초과 즉시 | 12시간 | 5회 |

### 원칙 B-3: M365 Graph API 기반 자동 보안 통제

퇴직 당일(D-Day) 시스템이 **M365 Graph API를 직접 호출**하여 3단계를 자동 실행합니다:

```
D-Day 00:00 (퇴직일 자정)
    ↓
1단계: OOF(Out of Office) 자동 설정
  - 수신 차단 후 자동 응답: "이 직원은 퇴사했습니다. 업무는 {인수인계자}에게 문의 바랍니다."
  - ⚠️ 이메일 포워딩은 절대 금지 (GDPR/개인정보 침해 리스크)
    ↓
2단계: 강제 로그아웃 (DISABLE)
  - 모든 디바이스(모바일/PC)의 M365 세션 강제 종료
  - 비밀번호 즉시 초기화
  - Azure AD 계정 비활성화
    ↓
3단계: 라이선스 회수 (LICENSE_REVOKE)
  - M365 라이선스 즉시 회수
  - 월 라이선스 비용 절감 (직원 1인당 약 $20/월)
```

**구현 시점 주의:** 이 자동화는 Stage 4(Heavy Integrations)에서 M365 API 연동이 구현된 후 활성화됩니다. 그 전까지는 IT 담당자의 수동 체크 태스크로 운영합니다.

```
// 분기 로직
if (m365IntegrationEnabled) {
  await m365Api.disableAccount(email);  // 자동 실행
} else {
  // IT 태스크로 수동 처리 (기존 방식)
  await createNotification(itAdmin, "계정 비활성화 필요: " + employeeName);
}
```

### 원칙 B-4: 퇴직 면담 데이터 철저 격리

**문제:** 퇴직 면담 내용을 직속 매니저가 볼 수 있으면, 퇴직자가 보복이나 평판 하락을 두려워하여 솔직한 사유를 숨깁니다. 이직 원인 데이터가 오염됩니다.

**해결 (Strict Isolation):**

| 역할 | ExitInterview 원본 열람 | 익명화 통계 열람 |
|------|:---:|:---:|
| 직속 매니저 | **✗ 원천 차단** | ✓ (부서 단위 집계만) |
| HR_ADMIN | ✓ | ✓ |
| SUPER_ADMIN | ✓ | ✓ |
| 퇴직자 본인 | ✗ (제출 후 수정/열람 불가) | ✗ |

```
매니저에게 제공되는 피드백 형태 (익명화 통계):

📊 본부 이직 사유 분기 리포트 (2026 Q1)
  • 보상 불만족: 35%
  • 커리어 성장 한계: 28%
  • 워라밸: 20%
  • 조직문화: 12%
  • 기타: 5%

※ 개인 식별 불가. 최소 5건 이상 집계 시에만 생성.
```

**최소 집계 기준:** 5건 미만이면 통계를 생성하지 않음 (소규모 팀에서 개인 특정 방지).

### 원칙 B-5: 미반납 자산 급여 공제 (법적 안전장치 포함)

**문제:** 퇴직자가 고가 IT 장비(노트북 등)를 반납하지 않고 잠적했을 때, 급여 지급 자체를 막으면 **근로기준법 위반(14일 이내 금품청산 의무)**.

**해결 (분리 정산 + 자동 공제 상신):**

```
IT팀의 '장비 반납' 태스크가 DONE이 아닌 상태로 오프보딩 완료
    ↓
① 퇴직금 + 최종 급여 계산은 무조건 진행 (노동법 준수)
    ↓
② 시스템이 GP#3 급여 파이프라인에 PayrollAdjustment 자동 생성:
   type: DEDUCTION
   category: UNRETURNED_ASSET
   amount: 장비 감가상각 후 잔존가액
   description: "미반납 장비: MacBook Pro 14" (자산번호 IT-2024-0312)"
    ↓
③ HR/경영진의 최종 승인 후 급여에서 공제 집행
```

**🚨 법적 전제 조건 (필수):**

| 조건 | 내용 |
|------|------|
| **사전 동의** | 입사 시 서약서 또는 장비 수령 시 "미반납 시 급여 공제 동의" 서명 필수 |
| **국가별 공제 한도** | KR: 근로자 동의 시 전액 공제 가능 / RU: 월급의 20% 한도 / US: 주(State)별 상이 |
| **감가상각** | 구매가가 아닌 감가상각 후 잔존가액으로 공제 |
| **공제 불가 시** | 법적으로 급여 공제가 불가능한 국가/케이스 → 별도 민사 청구로 전환 (HR 수동) |

> 공제 가능 여부와 한도는 `lib/labor/{country}.ts`에서 국가별 분기 처리합니다.

## C1. 온보딩 vs 오프보딩 설계 비교

| 항목 | 온보딩 | 오프보딩 |
|------|--------|----------|
| 파이프라인 | 4단계 (등록→생성→수행→**Sign-off**) | 4단계 (시작→생성→수행→**정산**) |
| 시간 방향 | 입사일 이후 (Day 1→90) | 퇴직일 이전 (D-30→D-Day) |
| 완료 게이트 | **매니저 Sign-off** (사람이 판단) | **자동 완료** (태스크 체크 기반) |
| Nudge 긴급도 | 마일스톤별 동적 (1~3일 간격) | 하드 데드라인 (12시간까지) |
| 보안 통제 | 없음 | M365 3단계 자동 차단 |
| 감정 데이터 | 감정 체크인 (Mood/Energy/Belonging) | 퇴직 면담 (Exit Interview) |
| 데이터 격리 | 매니저에게 공유 | **매니저 원천 차단** |
| 후속 프로세스 | 수습 평가 → GP#4 | 급여 정산 → GP#3 |
