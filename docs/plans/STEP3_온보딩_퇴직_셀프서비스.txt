기능

### 1. 온보딩 템플릿 관리 (/settings/onboarding) — HR_ADMIN

**템플릿 CRUD**
- DataTable: 템플릿명 / 대상 유형 / 태스크 수 / 상태
- 생성 폼: 이름, 설명, 대상 유형(NEW_HIRE/TRANSFER/REHIRE), 법인 선택
- 수정/삭제(Soft Delete)

**태스크 관리**
- 태스크 목록: 드래그 정렬 (sort_order)
- 태스크 추가: 제목, 설명, 담당자 유형(EMPLOYEE/MANAGER/HR/BUDDY), 기한(입사일+N일), 필수 여부, 카테고리(DOCUMENT/TRAINING/SETUP/INTRODUCTION/OTHER)
- 태스크 수정/삭제

---

### 2. 온보딩 현황 (/onboarding) — HR_ADMIN

**대시보드**
- 진행 중 온보딩 목록 DataTable:
  직원명 / 입사일 / 버디 / 템플릿 / 진행률(프로그레스 바) / 상태
- 필터: 완료/진행 중/지연
- 지연 기준: 기한 초과 태스크 1개 이상 → 행 bg-ctr-warning/5 하이라이트
- 담당자별 필터: "내 담당 태스크만" 토글

---

### 3. 내 온보딩 (/onboarding/me) — EMPLOYEE (신규 입사자)

**본인 온보딩 뷰**
- 상단: 환영 메시지 + 버디 정보 + 전체 진행률 프로그레스 바
- 체크리스트: 카테고리별 그룹핑 (📄 서류 / 🎓 교육 / 💻 장비 / 👋 소개 / 기타)
- 각 태스크: 체크박스 + 제목 + 담당자 유형 뱃지 + 기한 + 상태
- 체크 → PUT /api/v1/onboarding/tasks/:id/complete
  → employee_onboarding_tasks.status = DONE, completed_at 기록

**전체 완료 처리**
- 모든 is_required 태스크가 DONE →
  employee_onboarding.status = COMPLETED, completed_at 기록
  employees.onboarded_at = NOW()

---

### 4. 온보딩 완료 처리 — HR_ADMIN

- 전체 태스크 완료 시 자동 상태 변경 (위 로직)
- HR Admin 수동 완료: 미완료 태스크 SKIPPED 처리 후 강제 완료 가능
  → 사유 입력 필수

### 5. 감성 체크인 (/onboarding/checkin) — EMPLOYEE

**자동 알림**
- 입사 후 1주/2주/4주/8주/12주 시점에 자동 알림 발송
  (알림 트리거: ONBOARDING_CHECKIN)
- 알림 클릭 → 체크인 폼 이동

**체크인 폼**
- mood: 5단계 이모지 선택 (😃 GREAT / 🙂 GOOD / 😐 NEUTRAL / 😟 STRUGGLING / 😢 BAD)
- energy: 슬라이더 1~5 (체력/활력)
- belonging: 슬라이더 1~5 (소속감)
- comment: 자유 코멘트 (선택)
- 제출 → onboarding_checkins INSERT

**HR_ADMIN 뷰 (/onboarding/checkins)**
- 전체 체크인 현황 DataTable: 직원명 / 주차 / Mood / Energy / Belonging / 제출일
- 추세 차트: 직원별 mood/energy/belonging 추이 (Line Chart, Recharts)
- 위험 신호: mood=STRUGGLING/BAD 또는 belonging≤2 → 빨간 하이라이트

**AI 요약**
- HR_ADMIN → "AI 요약" 버튼 (직원별 전체 체크인 요약)
- lib/claude.ts의 onboardingCheckinSummary 함수 호출
```ts
onboardingCheckinSummary(
  employeeName: string,
  checkins: Array<{
    week: number,
    mood: string,
    energy: number,
    belonging: number,
    comment: string | null
  }>
): Promise<{
  overall_sentiment: 'POSITIVE' | 'MIXED' | 'CONCERNING',
  trend: 'IMPROVING' | 'STABLE' | 'DECLINING',
  key_observations: string[],
  recommended_actions: string[]
}>
prompt_version: "onboarding-checkin-summary-v1.0"
```
- 결과 → onboarding_checkins.ai_summary 저장 + AiGeneratedBadge 표시
- ai_logs (ONBOARDING_CHECKIN_SUMMARY) 기록

### 6. ★ 퇴직 관리 프로세스 (/offboarding)

#### 6-1. 퇴직 체크리스트 관리 (/settings/offboarding) — HR_ADMIN

**체크리스트 CRUD**
- DataTable: 이름 / 대상 유형 / 태스크 수 / 상태
- 생성 폼: 이름, 대상 유형(VOLUNTARY/INVOLUNTARY/RETIREMENT/CONTRACT_END), 법인 선택
- 태스크 관리: 드래그 정렬
  - 제목, 설명, 담당자 유형(EMPLOYEE/MANAGER/HR/IT/FINANCE), 기한(퇴직일-N일), 필수 여부

#### 6-2. 퇴직 처리 시작

**진입 경로**: /employees/:id 프로필 → "퇴직 처리" 버튼 (HR_ADMIN, status=ACTIVE)

**퇴직 처리 폼**
- 퇴직 유형 선택: 자발적 퇴사 / 비자발적 퇴사 / 정년퇴직 / 계약만료 / 합의 퇴직
- 최종 근무일 입력 (DatePicker)
- 퇴직 사유 코드 + 상세 사유 (선택)
- 인수인계 대상자 지정 (직원 검색, 선택)

**처리 로직 ($transaction)**
1. employees.status = RESIGNED (자발적/정년/계약/합의) 또는 TERMINATED (비자발적)
2. employees.resign_date = 최종 근무일
3. employee_histories INSERT:
   - change_type = RESIGN 또는 TERMINATE
   - reason = 상세 사유
4. employee_offboarding INSERT:
   - 퇴직 유형에 맞는 체크리스트 자동 매칭 (target_type 기준)
   - status = IN_PROGRESS
   - handover_to_id = 지정된 인수인계 대상자
5. employee_offboarding_tasks: 매칭 체크리스트의 전체 태스크 자동 생성 (status=PENDING)
6. 알림 발송:
   - 매니저: "팀원 {{name}}님이 퇴직 처리되었습니다"
   - IT: "계정 비활성화 필요 (D-{{days}})"
   - 재무: "퇴직금 정산 필요"
   - 인수인계 대상자: "업무 인수인계 요청"

#### 6-3. 퇴직 진행 대시보드 (/offboarding) — HR_ADMIN

**진행 중 퇴직 목록**
- DataTable: 직원명 / 법인 / 퇴직유형 / 최종근무일 / 진행률 / 상태
- 진행률: 완료 태스크 / 전체 태스크 (프로그레스 바)
- 퇴직일 임박 경고: D-7=🟡, D-3=🔴 아이콘

**필터**
- 상태: IN_PROGRESS / COMPLETED / CANCELLED
- 담당자 유형: "IT 미완료만" / "재무 미완료만" 토글
- 법인별 (CompanySelector)

**태스크 관리**
- 퇴직 건 클릭 → 태스크 상세 뷰
- 각 태스크: 담당자 유형 뱃지 + 상태 + 기한
- 태스크 완료 처리: PUT /api/v1/offboarding/:id/tasks/:taskId/complete
  → status=DONE, completed_by, completed_at
- 전체 필수 태스크 완료 → employee_offboarding.status=COMPLETED, completed_at

**완료된 퇴직 이력**
- 별도 탭 또는 필터로 조회
- 퇴직 면담 완료 여부 표시

#### 6-4. 퇴직 면담 (Exit Interview)

**면담 등록 — HR_ADMIN**
- POST /api/v1/offboarding/:id/exit-interview
- 폼:
  - 면담 일자 (DatePicker)
  - 면담자 (HR 담당자, 기본값=현재 로그인 사용자)
  - 주된 퇴사 사유 (선택형):
    COMPENSATION / CAREER_GROWTH / WORK_LIFE_BALANCE / MANAGEMENT / CULTURE / RELOCATION / PERSONAL / OTHER
  - 만족도 (1~5, 별점 UI)
  - 추천 의향 (Boolean, 토글)
  - 자유 의견 (Textarea)
- 저장 → exit_interviews INSERT
- employee_offboarding.exit_interview_completed = true

**AI 요약 버튼**
- "AI 분석" 버튼 → POST /api/v1/offboarding/:id/exit-interview/ai-summary
- lib/claude.ts:
```ts
exitInterviewSummary(
  employeeName: string,
  tenure_months: number,
  resign_type: string,
  primary_reason: string,
  satisfaction_score: number,
  feedback_text: string
): Promise<{
  sentiment: 'POSITIVE' | 'NEUTRAL' | 'NEGATIVE',
  key_issues: string[],
  retention_insight: string,
  action_needed: string | null
}>
prompt_version: "exit-interview-summary-v1.0"
```
- 결과 → exit_interviews.ai_summary 저장
- AiGeneratedBadge 표시
- ai_logs (EXIT_INTERVIEW_SUMMARY) 기록

**퇴직 사유 통계**: Analytics 연동 (STEP 7)

#### 6-5. 업무 인수인계 체크

**인수인계 프로세스**
- employee_offboarding.handover_to_id 지정 (퇴직 처리 시 또는 이후 수정)
- 인수인계 문서 업로드:
  - S3 Presigned URL 업로드
  - employee_documents INSERT (doc_type='HANDOVER', employee_id=퇴직자)
- 인수자 확인:
  - 인수자가 "인수인계 확인" 버튼 클릭
  - 해당 offboarding_task의 status=DONE으로 변경
- 인수인계 미완료 경고: D-3까지 미완료 시 매니저 + HR 알림

#### 6-6. IT 계정 비활성화

**퇴직 완료 시 처리**
- employee_offboarding의 모든 필수 태스크 완료 시:
  - it_account_deactivated = true
  - sso_sessions: 해당 직원의 모든 세션 revoked_at = NOW()
  - employee_auth.locked_until = 무기한 (far future date)
- 즉시 로그아웃 효과: revoked_at 설정된 세션 → 다음 요청 시 인증 실패

**Phase 2**: M365 계정 자동 비활성화 API 연동 (기술 부채)

#### 6-7. Attrition Risk 피드백 루프

- 퇴직 확정 (employee_offboarding.status=COMPLETED) 시:
  - 해당 직원의 attrition_risk_history 마지막 레코드의 score 조회
  - 이 데이터는 STEP 7 Analytics에서 "예측 정확도" 지표로 활용
  - 퇴직 직원 중 High Risk(70+)였던 비율 = 예측 적중률

### 7. Self-Service 정보수정 승인

#### EMPLOYEE 뷰
- /employees/me → 본인 프로필
- 수정 가능 필드: 전화번호, 비상연락처 이름, 비상연락처 전화
- "수정 요청" 버튼 → 폼:
  - 변경 필드 선택
  - 현재 값 (자동 표시) → 새 값 입력
  - 제출 → profile_change_requests INSERT (status=PENDING)
- 내 요청 이력 조회 (상태별 뱃지: PENDING=노랑, APPROVED=초록, REJECTED=빨강)

#### HR_ADMIN 뷰
- /settings/profile-requests
- 미처리 요청 목록 DataTable: 직원명 / 필드 / 현재값 → 새값 / 요청일 / 상태
- 승인/반려:
  - 승인: employees 테이블 해당 필드 반영 + audit_logs 기록
    → profile_change_requests.status=APPROVED, reviewed_by, reviewed_at
    → 직원에게 "정보 수정이 승인되었습니다" 알림
  - 반려: rejection_reason 입력 필수
    → profile_change_requests.status=REJECTED
    → 직원에게 "정보 수정이 반려되었습니다: {{사유}}" 알림
